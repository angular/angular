# Файл конфигурации Service Worker

В этой теме описываются свойства файла конфигурации Service Worker.

## Изменение конфигурации {#modifying-the-configuration}

JSON-файл конфигурации `ngsw-config.json` задаёт, какие файлы и URL данных Angular Service Worker должен кэшировать и как он должен обновлять кэшированные файлы и данные.
[Angular CLI](tools/cli) обрабатывает этот файл конфигурации во время `ng build`.

Все пути файлов должны начинаться с `/`, что соответствует директории развёртывания — обычно `dist/<project-name>` в CLI-проектах.

Если не указано иное, паттерны используют **ограниченный\*** glob-формат, который внутренне будет преобразован в регулярное выражение:

| Glob-форматы | Подробности                                                                                                      |
| :----------- | :--------------------------------------------------------------------------------------------------------------- |
| `**`         | Совпадает с 0 или более сегментами пути                                                                          |
| `*`          | Совпадает с 0 или более символами, кроме `/`                                                                     |
| `?`          | Совпадает ровно с одним символом, кроме `/`                                                                      |
| Префикс `!`  | Помечает паттерн как отрицательный, то есть включаются только файлы, не соответствующие паттерну                |

<docs-callout important title="Специальные символы нужно экранировать">
Обратите внимание, что некоторые символы со специальным значением в регулярных выражениях не экранируются, а паттерн не заключается в `^`/`$` при внутреннем преобразовании glob в regex.

`$` — это специальный символ в регулярных выражениях, совпадающий с концом строки, и он не будет автоматически экранирован при преобразовании glob-паттерна в регулярное выражение.

Если вы хотите буквально совпадать с символом `$`, вам нужно самостоятельно его экранировать (с помощью `\\$`). Например, glob-паттерн `/foo/bar/$value` приводит к несовпадаемому выражению, поскольку невозможно иметь строку с символами после её завершения.

Паттерн не будет автоматически заключён в `^` и `$` при преобразовании в регулярное выражение. Поэтому паттерны будут частично совпадать с URL запросов.

Если вы хотите, чтобы паттерны совпадали с началом и/или концом URL, можно добавить `^`/`$` самостоятельно. Например, glob-паттерн `/foo/bar/*.js` будет совпадать как с файлами `.js`, так и с `.json`. Если вы хотите совпадать только с файлами `.js`, используйте `/foo/bar/*.js$`.
</docs-callout>

Примеры паттернов:

| Паттерны     | Подробности                                    |
| :----------- | :--------------------------------------------- |
| `/**/*.html` | Указывает все HTML-файлы                       |
| `/*.html`    | Указывает только HTML-файлы в корневой папке   |
| `!/**/*.map` | Исключает все sourcemap-файлы                  |

## Свойства конфигурации Service Worker {#service-worker-configuration-properties}

Следующие разделы описывают каждое свойство файла конфигурации.

### `appData`

Этот раздел позволяет передавать любые данные, описывающие конкретную версию приложения.
Сервис `SwUpdate` включает эти данные в уведомления об обновлении.
Многие приложения используют этот раздел для предоставления дополнительной информации при отображении всплывающих окон UI, уведомляющих пользователей о доступном обновлении.

### `index`

Указывает файл, служащий индексной страницей для удовлетворения навигационных запросов.
Обычно это `/index.html`.

### `assetGroups`

_Ресурсы (Assets)_ — это ресурсы, являющиеся частью версии приложения и обновляющиеся вместе с приложением.
Они могут включать ресурсы, загружаемые с источника страницы, а также сторонние ресурсы, загружаемые из CDN и других внешних URL.
Поскольку не все такие внешние URL могут быть известны во время сборки, можно использовать URL-паттерны.

HELPFUL: Чтобы Service Worker обрабатывал ресурсы, загружаемые из разных источников, убедитесь, что [CORS](https://developer.mozilla.org/docs/Web/HTTP/CORS) правильно настроен на сервере каждого источника.

Это поле содержит массив групп ресурсов, каждая из которых определяет набор ресурсов и политику их кэширования.

```ts
{
  "assetGroups": [
    {
      …
    },
    {
      …
    }
  ]
}
```

HELPFUL: Когда ServiceWorker обрабатывает запрос, он проверяет группы ресурсов в порядке их появления в `ngsw-config.json`.
Первая группа ресурсов, совпадающая с запрошенным ресурсом, обрабатывает запрос.

Рекомендуется размещать более конкретные группы ресурсов выше в списке.
Например, группа ресурсов, совпадающая с `/foo.js`, должна появляться раньше той, которая совпадает с `*.js`.

Каждая группа ресурсов задаёт как набор ресурсов, так и политику, управляющую ими.
Эта политика определяет, когда ресурсы загружаются и что происходит при обнаружении изменений.

Группы ресурсов следуют интерфейсу TypeScript:

```ts
interface AssetGroup {
  name: string;
  installMode?: 'prefetch' | 'lazy';
  updateMode?: 'prefetch' | 'lazy';
  resources: {
    files?: string[];
    urls?: string[];
  };
  cacheQueryOptions?: {
    ignoreSearch?: boolean;
  };
}
```

Каждая `AssetGroup` определяется следующими свойствами группы ресурсов.

#### `name`

`name` обязателен.
Он идентифицирует конкретную группу ресурсов между версиями конфигурации.

#### `installMode`

`installMode` определяет, как эти ресурсы первоначально кэшируются.
`installMode` может принимать одно из двух значений:

| Значения   | Подробности                                                                                                                                                                                                                                                                                                                                                                                                     |
| :--------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `prefetch` | Указывает Angular Service Worker загружать каждый перечисленный ресурс при кэшировании текущей версии приложения. Это требует большой пропускной способности, но гарантирует доступность ресурсов при запросе, даже если браузер находится в офлайн-режиме.                                                                                                                                                     |
| `lazy`     | Не кэширует ресурсы заранее. Вместо этого Angular Service Worker кэширует ресурсы только при получении запросов на них. Это режим кэширования по требованию. Ресурсы, которые никогда не запрашиваются, не кэшируются. Это полезно, например, для изображений с разным разрешением — Service Worker кэширует только правильные ресурсы для конкретного экрана и ориентации. |

По умолчанию: `prefetch`.

#### `updateMode`

Для ресурсов, уже находящихся в кэше, `updateMode` определяет поведение кэширования при обнаружении новой версии приложения.
Любые ресурсы в группе, изменившиеся с предыдущей версии, обновляются в соответствии с `updateMode`.

| Значения   | Подробности                                                                                                                                                                                                                                                   |
| :--------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `prefetch` | Указывает Service Worker немедленно загружать и кэшировать изменённые ресурсы.                                                                                                                                                                                |
| `lazy`     | Указывает Service Worker не кэшировать эти ресурсы. Вместо этого они обрабатываются как незапрошенные и ожидают повторного запроса перед обновлением. `updateMode` со значением `lazy` допустим только если `installMode` также `lazy`. |

По умолчанию: значение, установленное в `installMode`.

#### `resources`

Этот раздел описывает ресурсы для кэширования, разбитые на следующие группы:

| Группы ресурсов | Подробности                                                                                                                                                                                                                                                                                                                                                                                                           |
| :-------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `files`         | Перечисляет паттерны, совпадающие с файлами в директории дистрибутива. Это могут быть отдельные файлы или glob-паттерны, совпадающие с несколькими файлами.                                                                                                                                                                                                                                                           |
| `urls`          | Включает URL и URL-паттерны, совпадающие во время выполнения. Эти ресурсы не загружаются напрямую и не имеют хешей содержимого, но кэшируются в соответствии с их HTTP-заголовками. Наиболее полезно для CDN, таких как Google Fonts. <br /> _(Отрицательные glob-паттерны не поддерживаются, а `?` совпадает буквально, то есть не совпадает ни с каким символом, кроме `?`.)_ |

#### `cacheQueryOptions`

Эти опции используются для изменения поведения сопоставления запросов.
Они передаются в функцию `Cache#match` браузера.
Подробности см. на [MDN](https://developer.mozilla.org/docs/Web/API/Cache/match).
В настоящее время поддерживаются только следующие опции:

| Опции          | Подробности                                            |
| :------------- | :----------------------------------------------------- |
| `ignoreSearch` | Игнорировать параметры запроса. По умолчанию `false`. |

### `dataGroups`

В отличие от ресурсов, запросы данных не версионируются вместе с приложением.
Они кэшируются в соответствии с политиками, настроенными вручную, которые более полезны для таких ситуаций, как API-запросы и другие зависимости от данных.

Это поле содержит массив групп данных, каждая из которых определяет набор ресурсов данных и политику их кэширования.

```json
{
  "dataGroups": [
    {
      …
    },
    {
      …
    }
  ]
}
```

HELPFUL: Когда ServiceWorker обрабатывает запрос, он проверяет группы данных в порядке их появления в `ngsw-config.json`.
Первая группа данных, совпадающая с запрошенным ресурсом, обрабатывает запрос.

Рекомендуется размещать более конкретные группы данных выше в списке.
Например, группа данных, совпадающая с `/api/foo.json`, должна появляться раньше той, которая совпадает с `/api/*.json`.

Группы данных следуют этому интерфейсу TypeScript:

```ts
export interface DataGroup {
  name: string;
  urls: string[];
  version?: number;
  cacheConfig: {
    maxSize: number;
    maxAge: string;
    timeout?: string;
    refreshAhead?: string;
    strategy?: 'freshness' | 'performance';
  };
  cacheQueryOptions?: {
    ignoreSearch?: boolean;
  };
}
```

Каждая `DataGroup` определяется следующими свойствами группы данных.

#### `name`

Аналогично `assetGroups`, каждая группа данных имеет `name`, уникально её идентифицирующий.

#### `urls`

Список URL-паттернов.
URL, совпадающие с этими паттернами, кэшируются в соответствии с политикой этой группы данных.
Кэшируются только немутирующие запросы (GET и HEAD).

- Отрицательные glob-паттерны не поддерживаются
- `?` совпадает буквально, то есть совпадает _только_ с символом `?`

#### `version`

Иногда API меняют форматы способами, несовместимыми с предыдущими версиями.
Новая версия приложения может быть несовместима со старым форматом API и, следовательно, может быть несовместима с существующими кэшированными ресурсами этого API.

`version` предоставляет механизм для указания того, что кэшированные ресурсы были обновлены обратно несовместимым способом, и что старые записи кэша — из предыдущих версий — должны быть отброшены.

`version` — это целочисленное поле со значением по умолчанию `1`.

#### `cacheConfig`

Следующие свойства определяют политику кэширования совпадающих запросов.

##### `maxSize`

Максимальное количество записей или ответов в кэше.

CRITICAL: Открытые кэши могут расти неограниченно и в конечном итоге превысить квоты хранилища, что приведёт к вытеснению данных.

##### `maxAge`

Параметр `maxAge` указывает, как долго ответам разрешено оставаться в кэше до признания их недействительными и вытеснения. `maxAge` — это строка длительности, использующая следующие суффиксы единиц:

| Суффиксы | Подробности  |
| :------- | :----------- |
| `d`      | Дни          |
| `h`      | Часы         |
| `m`      | Минуты       |
| `s`      | Секунды      |
| `u`      | Миллисекунды |

Например, строка `3d12h` кэширует содержимое на срок до трёх с половиной дней.

##### `timeout`

Эта строка длительности указывает тайм-аут сети.
Тайм-аут сети — это время, которое Angular Service Worker ожидает ответа сети перед использованием кэшированного ответа, если настроено соответствующим образом.
`timeout` — это строка длительности, использующая следующие суффиксы единиц:

| Суффиксы | Подробности  |
| :------- | :----------- |
| `d`      | Дни          |
| `h`      | Часы         |
| `m`      | Минуты       |
| `s`      | Секунды      |
| `u`      | Миллисекунды |

Например, строка `5s30u` соответствует пяти секундам и 30 миллисекундам тайм-аута сети.

##### `refreshAhead`

Эта строка длительности указывает время до истечения срока действия кэшированного ресурса, когда Angular Service Worker должен проактивно попытаться обновить ресурс из сети.
Длительность `refreshAhead` — необязательная конфигурация, определяющая, за сколько времени до истечения срока кэшированного ответа Service Worker должен инициировать запрос на обновление ресурса из сети.

| Суффиксы | Подробности  |
| :------- | :----------- |
| `d`      | Дни          |
| `h`      | Часы         |
| `m`      | Минуты       |
| `s`      | Секунды      |
| `u`      | Миллисекунды |

Например, строка `1h30m` соответствует одному часу и 30 минутам до истечения срока действия.

##### `strategy`

Angular Service Worker может использовать одну из двух стратегий кэширования для ресурсов данных.

| Стратегии кэширования | Подробности                                                                                                                                                                                                                                                                                                                                                          |
| :-------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `performance`         | По умолчанию, оптимизирует для максимально быстрых ответов. Если ресурс существует в кэше, используется кэшированная версия без сетевого запроса. Это допускает некоторую устарелость данных в зависимости от `maxAge` в обмен на лучшую производительность. Подходит для ресурсов, которые редко меняются, например, аватары пользователей.                        |
| `freshness`           | Оптимизирует для актуальности данных, предпочтительно загружая запрошенные данные из сети. Запрос переходит к кэшу только при тайм-ауте сети согласно `timeout`. Полезно для ресурсов, которые часто меняются, например, балансы счетов.                                                                                                                            |

HELPFUL: Также можно эмулировать третью стратегию — [staleWhileRevalidate](https://developers.google.com/web/fundamentals/instant-and-offline/offline-cookbook/#stale-while-revalidate), которая возвращает кэшированные данные, если они доступны, но также загружает свежие данные из сети в фоновом режиме для следующего раза.
Для использования этой стратегии установите `strategy` в `freshness` и `timeout` в `0u` в `cacheConfig`.

По существу, это делает следующее:

1. Попытаться получить данные из сети первым.
2. Если сетевой запрос не завершается немедленно, то есть после тайм-аута 0&nbsp;мс, игнорировать возраст кэша и использовать кэшированное значение.
3. После завершения сетевого запроса обновить кэш для будущих запросов.
4. Если ресурс не существует в кэше, всё равно ждать сетевого запроса.

##### `cacheOpaqueResponses`

Следует ли Angular Service Worker кэшировать непрозрачные ответы.

Если не указано, значение по умолчанию зависит от настроенной стратегии группы данных:

| Стратегии                               | Подробности                                                                                                                                                                                                                                                                                                                                      |
| :-------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Группы со стратегией `freshness`        | Значение по умолчанию `true`, и Service Worker кэширует непрозрачные ответы. Эти группы будут запрашивать данные каждый раз и переходить к кэшированному ответу только при офлайн-режиме или медленной сети. Поэтому не имеет значения, кэширует ли Service Worker ответ с ошибкой.                                                              |
| Группы со стратегией `performance`      | Значение по умолчанию `false`, и Service Worker не кэширует непрозрачные ответы. Эти группы продолжали бы возвращать кэшированный ответ до истечения `maxAge`, даже если ошибка была вызвана временной сетевой проблемой или проблемой сервера. Поэтому для Service Worker было бы проблематично кэшировать ответ с ошибкой.                     |

<docs-callout title="Примечание о непрозрачных ответах">

Если вы не знакомы с понятием, [непрозрачный ответ](https://fetch.spec.whatwg.org#concept-filtered-response-opaque) — это специальный тип ответа, возвращаемый при запросе ресурса другого источника, который не возвращает заголовки CORS.
Одна из характеристик непрозрачного ответа заключается в том, что Service Worker не может читать его статус, то есть не может проверить успешность запроса.
Подробнее см. в [Introduction to `fetch()`](https://developers.google.com/web/updates/2015/03/introduction-to-fetch#response_types).

Если вы не можете реализовать CORS — например, если вы не контролируете источник — предпочтительно использовать стратегию `freshness` для ресурсов, приводящих к непрозрачным ответам.

</docs-callout>

### `navigationUrls`

Этот необязательный раздел позволяет указать пользовательский список URL, которые будут перенаправлены на индексный файл.

#### Обработка навигационных запросов {#handling-navigation-requests}

ServiceWorker перенаправляет навигационные запросы, не совпадающие ни с одной группой `asset` или `data`, к указанному [индексному файлу](#index).
Запрос считается навигационным, если:

- Его [метод](https://developer.mozilla.org/docs/Web/API/Request/method) — `GET`
- Его [режим](https://developer.mozilla.org/docs/Web/API/Request/mode) — `navigation`
- Он принимает ответ `text/html` согласно значению заголовка `Accept`
- Его URL соответствует следующим критериям:
  - URL не должен содержать расширение файла (то есть `.`) в последнем сегменте пути
  - URL не должен содержать `__`

HELPFUL: Чтобы настроить, отправляются ли навигационные запросы через сеть, см. разделы [navigationRequestStrategy](#navigationrequeststrategy) и [applicationMaxAge](#applicationmaxage).

#### Совпадение URL навигационных запросов {#matching-navigation-request-urls}

Хотя эти критерии по умолчанию подходят для большинства случаев, иногда желательно настроить другие правила.
Например, может потребоваться игнорировать определённые маршруты, не являющиеся частью Angular-приложения, и передавать их на сервер.

Это поле содержит массив URL и [glob-подобных](#modifying-the-configuration) URL-паттернов, совпадающих во время выполнения.
Может содержать как отрицательные паттерны (начинающиеся с `!`), так и неотрицательные паттерны и URL.

Только запросы, URL которых совпадает с _любым_ из неотрицательных URL/паттернов и _ни одним_ из отрицательных, считаются навигационными запросами.
Параметры запроса игнорируются при сопоставлении.

Если поле опущено, по умолчанию используется:

```ts
[
  '/**', // Include all URLs.
  '!/**/*.*', // Exclude URLs to files (containing a file extension in the last segment).
  '!/**/*__*', // Exclude URLs containing `__` in the last segment.
  '!/**/*__*/**', // Exclude URLs containing `__` in any other segment.
];
```

### `navigationRequestStrategy`

Это необязательное свойство позволяет настроить обработку навигационных запросов Service Worker:

```json
{
  "navigationRequestStrategy": "freshness"
}
```

| Возможные значения | Подробности                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| :----------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `'performance'`    | Настройка по умолчанию. Обслуживает указанный [индексный файл](#index), который обычно кэшируется.                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| `'freshness'`      | Передаёт запросы в сеть и при офлайн-режиме переходит к поведению `performance`. Это значение полезно, когда сервер перенаправляет навигационные запросы с помощью кода перенаправления HTTP `3xx`. Причины для использования этого значения: <ul> <li> Перенаправление на сайт аутентификации, когда аутентификация не обрабатывается приложением </li> <li> Перенаправление конкретных URL для предотвращения нарушения существующих ссылок/закладок после редизайна сайта </li> <li> Перенаправление на другой сайт, например страницу статуса сервера, пока страница временно недоступна </li> </ul> |

IMPORTANT: Стратегия `freshness` обычно приводит к большему количеству запросов, отправляемых на сервер, что может увеличить задержку ответа. Рекомендуется использовать стратегию `performance` по умолчанию, когда это возможно.

### `applicationMaxAge`

Это необязательное свойство позволяет настроить, как долго Service Worker будет кэшировать любые запросы. В рамках `maxAge` файлы будут обслуживаться из кэша. По истечении этого времени все запросы будут обслуживаться только из сети, включая запросы ресурсов и данных.
