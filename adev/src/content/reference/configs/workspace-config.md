# Конфигурация рабочего пространства Angular {#angular-workspace-configuration}

Файл `angular.json` на корневом уровне рабочего пространства Angular содержит значения конфигурации по умолчанию на уровне рабочего пространства и отдельных проектов. Они используются инструментами сборки и разработки, предоставляемыми Angular CLI.
Значения путей в конфигурации указываются относительно корневого каталога рабочего пространства.

## Общая структура JSON {#general-json-structure}

На верхнем уровне `angular.json` несколько свойств настраивают рабочее пространство, а раздел `projects` содержит остальные параметры конфигурации для каждого проекта.
Можно переопределить значения Angular CLI по умолчанию, установленные на уровне рабочего пространства, с помощью значений на уровне проекта.
Также можно переопределить значения по умолчанию на уровне проекта с помощью командной строки.

Следующие свойства на верхнем уровне файла настраивают рабочее пространство.

| Свойства         | Подробности                                                                                                                                                                                                       |
| :--------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `version`        | Версия файла конфигурации.                                                                                                                                                                                        |
| `newProjectRoot` | Путь, по которому создаются новые проекты с помощью инструментов вроде `ng generate application` или `ng generate library`. Путь может быть абсолютным или относительным каталогу рабочего пространства. По умолчанию: `projects` |
| `cli`            | Набор параметров для настройки [Angular CLI](tools/cli). См. [параметры конфигурации Angular CLI](#angular-cli-configuration-options) ниже.                                                                        |
| `schematics`     | Набор [схем](tools/cli/schematics), настраивающих значения по умолчанию для подкоманд `ng generate` в этом рабочем пространстве. См. [schemes](#schematics) ниже.                                                  |
| `projects`       | Содержит подраздел для каждого приложения или библиотеки в рабочем пространстве со специфичными для проекта параметрами конфигурации.                                                                              |

Исходное приложение, созданное с помощью `ng new app-name`, указывается в разделе "projects":

При создании библиотечного проекта с помощью `ng generate library` библиотечный проект также добавляется в раздел `projects`.

ПОДСКАЗКА: Раздел `projects` файла конфигурации не соответствует точно структуре файлов рабочего пространства.

<!-- markdownlint-disable-next-line MD032 -->

- Исходное приложение, созданное с помощью `ng new`, находится на верхнем уровне структуры файлов рабочего пространства.
- Другие приложения и библиотеки по умолчанию находятся в каталоге `projects`.

Подробнее см. в разделе [Структура файлов рабочего пространства и проекта](reference/configs/file-structure).

## Схемы {#schematics}

[Схемы Angular](tools/cli/schematics) — это инструкции по изменению проекта путём добавления новых файлов или изменения существующих.
Их можно настроить, сопоставив имя схемы с набором параметров по умолчанию.

«Имя» схемы имеет формат: `<schematic-package>:<schematic-name>`.
Схемы для подкоманд `ng generate` Angular CLI по умолчанию собраны в пакете [`@schematics/angular`](https://github.com/angular/angular-cli/blob/main/packages/schematics/angular/application/schema.json).
Например, схема для генерации компонента с помощью `ng generate component` — это `@schematics/angular:component`.

Поля, указанные в схеме схемы, соответствуют допустимым значениям аргументов командной строки и значениям по умолчанию для параметров подкоманд Angular CLI.
Вы можете обновить файл схемы рабочего пространства, чтобы задать другое значение по умолчанию для параметра подкоманды. Например, чтобы отключить `standalone` в `ng generate component` по умолчанию:

```json
{
  "projects": {
    "my-app": {
      "schematics": {
        "@schematics/angular:component": {
          "standalone": false
        }
      }
    }
  }
}
```

## Настройка сборщиков CLI {#configuring-cli-builders}

Architect — это инструмент, используемый Angular CLI для выполнения сложных задач, таких как компиляция и запуск тестов.
Architect — это оболочка, запускающая указанный сборщик для выполнения заданной задачи согласно конфигурации цели.
Вы можете определять и настраивать новые сборщики и цели для расширения Angular CLI.
См. раздел [Сборщики Angular CLI](tools/cli/cli-builder).

### Сборщики и цели Architect по умолчанию {#default-architect-builders-and-targets}

Angular определяет сборщики по умолчанию для использования с конкретными командами или с общей командой `ng run`.
JSON-схемы, определяющие параметры и значения по умолчанию для каждого из этих сборщиков, собраны в пакете [`@angular-devkit/build-angular`](https://github.com/angular/angular-cli/blob/main/packages/angular_devkit/build_angular/builders.json).
Схемы настраивают параметры для следующих сборщиков.

### Настройка целей сборщика {#configuring-builder-targets}

Раздел `architect` в `angular.json` содержит набор целей Architect.
Многие из целей соответствуют командам Angular CLI, которые их запускают.
Другие цели можно выполнить с помощью команды `ng run`, и вы можете определять собственные цели.

Каждый объект цели указывает `builder` для этой цели — npm-пакет инструмента, который запускает Architect.
Каждая цель также имеет раздел `options` с параметрами по умолчанию для цели и раздел `configurations` с именованными альтернативными конфигурациями.
См. пример в разделе [Цель сборки](#build-target) ниже.

| Свойство       | Подробности                                                                                                                                                                                                 |
| :------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `build`        | Настраивает значения по умолчанию для параметров команды `ng build`. Подробнее см. в разделе [Цель сборки](#build-target).                                                                                   |
| `serve`        | Переопределяет значения по умолчанию сборки и добавляет значения по умолчанию обслуживания для команды `ng serve`. Помимо параметров команды `ng build`, добавляет параметры, связанные с обслуживанием приложения. |
| `e2e`          | Переопределяет значения по умолчанию сборки для построения приложений сквозного тестирования с использованием команды `ng e2e`.                                                                              |
| `test`         | Переопределяет значения по умолчанию сборки для тестовых сборок и добавляет значения по умолчанию запуска тестов для команды `ng test`.                                                                     |
| `lint`         | Настраивает значения по умолчанию для параметров команды `ng lint`, которая выполняет статический анализ кода исходных файлов проекта.                                                                       |
| `extract-i18n` | Настраивает значения по умолчанию для параметров команды `ng extract-i18n`, которая извлекает локализованные строки сообщений из исходного кода и создаёт файлы перевода для интернационализации.            |

ПОДСКАЗКА: Все параметры в файле конфигурации должны использовать `camelCase`, а не `dash-case`, используемый в командной строке.

## Цель сборки {#build-target}

Каждая цель в разделе `architect` имеет следующие свойства:

| Свойство         | Подробности                                                                                                                                                                                                                                                     |
| :--------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `builder`        | Сборщик CLI, используемый для создания этой цели, в формате `<package-name>:<builder-name>`.                                                                                                                                                                    |
| `options`        | Параметры по умолчанию цели сборки.                                                                                                                                                                                                                             |
| `configurations` | Альтернативные конфигурации для выполнения цели. Каждая конфигурация задаёт параметры по умолчанию для предполагаемой среды, переопределяя связанное значение в разделе `options`. См. [Альтернативные конфигурации сборки](#alternate-build-configurations) ниже. |

Например, для настройки сборки с отключенными оптимизациями:

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "optimization": false
          }
        }
      }
    }
  }
}
```

### Альтернативные конфигурации сборки {#alternate-build-configurations}

Angular CLI поставляется с двумя конфигурациями сборки: `production` и `development`.
По умолчанию команда `ng build` использует конфигурацию `production`, которая применяет несколько оптимизаций сборки, включая:

- Бандлинг файлов
- Минимизацию лишних пробелов
- Удаление комментариев и мёртвого кода
- Минификацию кода с использованием коротких искажённых имён

Можно определять и именовать дополнительные альтернативные конфигурации (например, `staging`) в соответствии с процессом разработки.
Альтернативную конфигурацию можно выбрать, передав её имя флагу командной строки `--configuration`.

Например, для настройки сборки, где оптимизация включена только для продакшен-сборок (`ng build --configuration production`):

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "optimization": false
          },
          "configurations": {
            "production": {
              "optimization": true
            }
          }
        }
      }
    }
  }
}
```

Также можно передать несколько имён конфигураций в виде списка, разделённого запятыми.
Например, для применения как конфигурации `staging`, так и `french` используйте команду `ng build --configuration staging,french`.
В этом случае команда разбирает именованные конфигурации слева направо.
Если несколько конфигураций изменяют одну и ту же настройку, используется последнее установленное значение.
В этом примере, если обе конфигурации `staging` и `french` задают путь вывода, будет использоваться значение из `french`.

## Сложные значения конфигурации {#complex-configuration-values}

Параметры `assets`, `index`, `outputPath`, `styles` и `scripts` могут принимать как простые строковые значения путей, так и объекты с конкретными полями.
Параметры `sourceMap` и `optimization` можно задать простым логическим значением. Им также можно задать сложное значение с помощью файла конфигурации.

Следующие разделы содержат подробности о том, как эти сложные значения используются в каждом случае.

### Конфигурация ресурсов {#assets-configuration}

Каждая конфигурация цели `build` может включать массив `assets` со списком файлов или папок, которые нужно скопировать без изменений при сборке проекта.
По умолчанию копируется содержимое каталога `public/`.

Для исключения ресурса можно удалить его из конфигурации ресурсов.

Можно настроить более детальное копирование ресурсов, указывая их как объекты, а не как простые пути относительно корня рабочего пространства.
Объект спецификации ресурса может содержать следующие поля.

| Поля             | Подробности                                                                                                                                     |
| :--------------- | :---------------------------------------------------------------------------------------------------------------------------------------------- |
| `glob`           | [node-glob](https://github.com/isaacs/node-glob/blob/master/README.md) с использованием `input` в качестве базового каталога.                   |
| `input`          | Путь относительно корня рабочего пространства.                                                                                                  |
| `output`         | Путь относительно `outDir`. Из соображений безопасности Angular CLI никогда не записывает файлы за пределами пути вывода проекта.               |
| `ignore`         | Список шаблонов glob для исключения.                                                                                                            |
| `followSymlinks` | Разрешить шаблонам glob следовать по символическим ссылкам на каталоги. Это позволяет выполнять поиск в подкаталогах символических ссылок. По умолчанию: `false`. |

Например, пути к ресурсам по умолчанию можно представить более подробно с помощью следующих объектов.

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "assets": [
              {
                "glob": "**/*",
                "input": "src/assets/",
                "output": "/assets/"
              },
              {
                "glob": "favicon.ico",
                "input": "src/",
                "output": "/"
              }
            ]
          }
        }
      }
    }
  }
}
```

В следующем примере поле `ignore` используется для исключения определённых файлов из каталога ресурсов при копировании в сборку:

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "assets": [
              {
                "glob": "**/*",
                "input": "src/assets/",
                "ignore": ["**/*.svg"],
                "output": "/assets/"
              }
            ]
          }
        }
      }
    }
  }
}
```

### Конфигурация стилей и скриптов {#styles-and-scripts-configuration}

Запись массива для параметров `styles` и `scripts` может быть как простой строкой пути, так и объектом, указывающим на дополнительный файл точки входа.
Связанный сборщик загружает этот файл и его зависимости как отдельный бандл во время сборки.
С объектом конфигурации есть возможность задать имя бандлу для точки входа с помощью поля `bundleName`.

Бандл внедряется по умолчанию, но можно установить `inject` в `false` для исключения бандла из внедрения.
Например, следующие значения объекта создают и именуют бандл, содержащий стили и скрипты, и исключают его из внедрения:

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "styles": [
              {
                "input": "src/external-module/styles.scss",
                "inject": false,
                "bundleName": "external-module"
              }
            ],
            "scripts": [
              {
                "input": "src/external-module/main.js",
                "inject": false,
                "bundleName": "external-module"
              }
            ]
          }
        }
      }
    }
  }
}
```

#### Параметры препроцессора стилей {#style-preprocessor-options}

В Sass можно использовать функцию `includePaths` как для стилей компонентов, так и для глобальных стилей. Это позволяет добавлять дополнительные базовые пути, проверяемые при импортах.

Для добавления путей используйте параметр `stylePreprocessorOptions`:

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "stylePreprocessorOptions": {
              "includePaths": ["src/style-paths"]
            }
          }
        }
      }
    }
  }
}
```

Файлы в этом каталоге, например `src/style-paths/_variables.scss`, можно импортировать из любого места в проекте без необходимости указывать относительный путь:

```scss
// src/app/app.scss
// Относительный путь работает
@import '../style-paths/variables';

// Но теперь это тоже работает
@import 'variables';
```

ПОДСКАЗКА: Также необходимо добавить стили или скрипты в сборщик `test`, если они нужны для модульных тестов.
См. также раздел [Использование глобальных библиотек времени выполнения в приложении](tools/libraries/using-libraries#using-runtime-global-libraries-inside-your-app).

### Конфигурация оптимизации {#optimization-configuration}

Параметр `optimization` может быть либо логическим значением, либо объектом для более тонкой настройки.
Этот параметр включает различные оптимизации вывода сборки, включая:

- Минификацию скриптов и стилей
- Tree-shaking
- Устранение мёртвого кода
- [Встраивание критических CSS](/tools/cli/build#critical-css-inlining)
- Встраивание шрифтов

Для тонкой настройки оптимизации приложения можно использовать несколько параметров.

| Параметры | Подробности                                                                     | Тип значения                                                              | Значение по умолчанию |
| :-------- | :------------------------------------------------------------------------------ | :------------------------------------------------------------------------ | :-------------------- |
| `scripts` | Включает оптимизацию вывода скриптов.                                           | `boolean`                                                                 | `true`                |
| `styles`  | Включает оптимизацию вывода стилей.                                             | `boolean` \| [Параметры оптимизации стилей](#styles-optimization-options) | `true`                |
| `fonts`   | Включает оптимизацию шрифтов. Требует доступа к интернету.                      | `boolean` \| [Параметры оптимизации шрифтов](#fonts-optimization-options) | `true`                |

#### Параметры оптимизации стилей {#styles-optimization-options}

| Параметры               | Подробности                                                                                                                         | Тип значения | Значение по умолчанию |
| :---------------------- | :---------------------------------------------------------------------------------------------------------------------------------- | :--------- | :-------------------- |
| `minify`                | Минификация CSS-определений путём удаления лишних пробелов и комментариев, объединения идентификаторов и минимизации значений.       | `boolean`  | `true`                |
| `inlineCritical`        | Извлечение и встраивание критических CSS-определений для улучшения [First Contentful Paint](https://web.dev/first-contentful-paint). | `boolean`  | `true`                |
| `removeSpecialComments` | Удаление комментариев в глобальном CSS, содержащих `@license` или `@preserve`, или начинающихся с `//!` или `/*!`.                  | `boolean`  | `true`                |

#### Параметры оптимизации шрифтов {#fonts-optimization-options}

| Параметры | Подробности                                                                                                                                                                                                               | Тип значения | Значение по умолчанию |
| :-------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :--------- | :-------------------- |
| `inline`  | Уменьшение [блокирующих рендеринг запросов](https://web.dev/render-blocking-resources) путём встраивания CSS-определений внешних шрифтов Google Fonts и Adobe Fonts в HTML-файл index приложения. Требует доступа к интернету. | `boolean`  | `true`                |

Можно указать такое значение для применения оптимизации к тому или иному элементу:

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "stylePreprocessorOptions": {
              "includePaths": ["src/style-paths"]
            }
          }
        }
      }
    }
  }
}
```

### Конфигурация карт источников {#source-map-configuration}

Параметр сборщика `sourceMap` может быть либо логическим значением, либо объектом для более тонкой настройки карт источников приложения.

| Параметры        | Подробности                                                          | Тип значения | Значение по умолчанию |
| :--------------- | :------------------------------------------------------------------- | :--------- | :-------------------- |
| `scripts`        | Вывод карт источников для всех скриптов.                             | `boolean`  | `true`                |
| `styles`         | Вывод карт источников для всех стилей.                               | `boolean`  | `true`                |
| `vendor`         | Разрешение карт источников сторонних пакетов.                        | `boolean`  | `false`               |
| `hidden`         | Опустить ссылку на карты источников из вывода JavaScript.            | `boolean`  | `false`               |
| `sourcesContent` | Вывод исходного содержимого файлов в картах источников.              | `boolean`  | `true`                |

В следующем примере показано, как переключить одно или несколько значений для настройки вывода карт источников:

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "sourceMap": {
              "scripts": true,
              "styles": false,
              "hidden": true,
              "vendor": true
            }
          }
        }
      }
    }
  }
}
```

ПОДСКАЗКА: При использовании скрытых карт источников они не упоминаются в бандле.
Это полезно, если вы хотите, чтобы карты источников использовались только для сопоставления трассировок стека в инструментах отчётов об ошибках без отображения в инструментах разработчика браузера.
Обратите внимание, что даже если `hidden` предотвращает ссылку на карту источников в выходном бандле, процесс развёртывания должен следить за тем, чтобы не предоставлять сгенерированные карты источников в продакшене, иначе информация всё равно будет раскрыта.

#### Карты источников без содержимого источников {#source-maps-without-sources-content}

Можно генерировать карты источников без поля `sourcesContent`, содержащего исходный код.
Это позволяет развёртывать карты источников в продакшене для улучшения отчётности об ошибках с исходными именами, защищая при этом исходный код от раскрытия.

Для исключения содержимого источников из карт источников установите параметр `sourcesContent` в `false`:

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "sourceMap": {
              "scripts": true,
              "styles": true,
              "sourcesContent": false
            }
          }
        }
      }
    }
  }
}
```

### Конфигурация index {#index-configuration}

Настраивает генерацию HTML-index приложения.

Параметр `index` может быть либо строкой, либо объектом для более тонкой настройки.

При указании значения в виде строки имя файла указанного пути будет использоваться для сгенерированного файла и будет создано в корне настроенного пути вывода приложения.

#### Параметры index {#index-options}

| Параметры | Подробности                                                                                                                                                                                      | Тип значения | Значение по умолчанию |
| :-------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :--------- | :-------------------- |
| `input`   | Путь к файлу, используемому для сгенерированного HTML-index приложения.                                                                                                                          | `string`   | Нет (обязательно)     |
| `output`  | Путь вывода сгенерированного HTML-index файла приложения. Будет использован полный указанный путь и он будет считаться относительным настроенному пути вывода приложения. | `string`   | `index.html`          |

### Конфигурация пути вывода {#output-path-configuration}

Параметр `outputPath` может быть либо строкой, которая будет использоваться как значение `base`, либо объектом для более тонкой настройки.

Для тонкой настройки структуры вывода приложения можно использовать несколько параметров.

| Параметры | Подробности                                                                                                                                                                      | Тип значения | Значение по умолчанию |
| :-------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :--------- | :-------------------- |
| `base`    | Указание пути вывода относительно корня рабочего пространства.                                                                                                                   | `string`   |                       |
| `browser` | Имя выходного каталога для браузерной сборки находится внутри базового пути вывода. Может безопасно предоставляться пользователям.                                                | `string`   | `browser`             |
| `server`  | Имя выходного каталога серверной сборки внутри базового пути вывода.                                                                                                             | `string`   | `server`              |
| `media`   | Имя выходного каталога для медиафайлов, расположенных в выходном браузерном каталоге. Эти медиафайлы обычно называются ресурсами в CSS-файлах.                                   | `string`   | `media`               |
