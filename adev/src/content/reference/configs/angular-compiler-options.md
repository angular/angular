# Параметры компилятора Angular

При использовании [компиляции ahead-of-time (AOT)](tools/cli/aot-compiler) вы можете управлять тем, как компилируется
ваше приложение, указав параметры компилятора Angular в
[файле конфигурации TypeScript](https://www.typescriptlang.org/docs/handbook/tsconfig-json.html).

Объект параметров Angular, `angularCompilerOptions`, является соседом объекта `compilerOptions`, который предоставляет
стандартные параметры компилятору TypeScript.

## Наследование конфигурации с помощью `extends` {#configuration-inheritance-with-extends}

Как и компилятор TypeScript, AOT-компилятор Angular также поддерживает `extends` в секции
`angularCompilerOptions` файла конфигурации TypeScript. Свойство `extends` находится на верхнем уровне, рядом с
`compilerOptions` и `angularCompilerOptions`.

Конфигурация TypeScript может наследовать настройки из другого файла с помощью свойства `extends`. Параметры
конфигурации из базового файла загружаются первыми, а затем переопределяются параметрами из наследующего файла
конфигурации.

Подробнее см. в [Руководстве по TypeScript](https://www.typescriptlang.org/docs/handbook/tsconfig-json.html).

## Параметры шаблона {#template-options}

Следующие параметры доступны для настройки AOT-компилятора Angular.

### `annotationsAs` {#annotationsas}

Изменяет способ генерации аннотаций, специфичных для Angular, для улучшения tree-shaking. Аннотации, не относящиеся к
Angular, не затрагиваются. Одно из значений: `static fields` или `decorators`. Значение по умолчанию: `static fields`.

- По умолчанию компилятор заменяет декораторы статическим полем в классе, что позволяет продвинутым инструментам
  tree-shaking, таким как [Closure compiler](https://github.com/google/closure-compiler), удалять неиспользуемые классы
- Значение `decorators` оставляет декораторы на месте, что ускоряет компиляцию. TypeScript генерирует вызовы хелпера
  `__decorate`. Используйте `--emitDecoratorMetadata` для рефлексии во время выполнения.

  HELPFUL: Результирующий код не может корректно проходить tree-shaking.

### `annotateForClosureCompiler` {#annotateforclosurecompiler}

<!-- vale Angular.Angular_Spelling = NO -->

При значении `true` использует [Tsickle](https://github.com/angular/tsickle) для аннотирования генерируемого JavaScript
комментариями [JSDoc](https://jsdoc.app), необходимыми для
[Closure Compiler](https://github.com/google/closure-compiler). Значение по умолчанию: `false`.

<!-- vale Angular.Angular_Spelling = YES -->

### `compilationMode` {#compilationmode}

Задает режим компиляции. Доступны следующие режимы:

| Режимы      | Описание                                                                                                 |
| :---------- | :------------------------------------------------------------------------------------------------------- |
| `'full'`    | Генерирует полностью AOT-скомпилированный код в соответствии с используемой версией Angular.             |
| `'partial'` | Генерирует код в стабильном, но промежуточном формате, подходящем для публикуемой библиотеки.            |

Значение по умолчанию: `'full'`.

Для большинства приложений `'full'` является правильным режимом компиляции.

Используйте `'partial'` для независимо публикуемых библиотек, например npm-пакетов. Компиляции в режиме `'partial'`
генерируют стабильный промежуточный формат, который лучше поддерживает использование приложениями, собранными с
отличными от библиотеки версиями Angular. Библиотеки, собираемые на "HEAD" вместе с приложениями и использующие ту же
версию Angular (например, в монорепозитории), могут использовать `'full'`, так как риска расхождения версий нет.

### `disableExpressionLowering` {#disableexpressionlowering}

При значении `true` (по умолчанию) преобразует код, который используется или может использоваться в аннотации, чтобы
его можно было импортировать из фабричных модулей шаблонов. Подробнее см. в разделе
[перезапись метаданных](tools/cli/aot-compiler#metadata-rewriting).

При значении `false` отключает эту перезапись, требуя выполнять её вручную.

### `disableTypeScriptVersionCheck` {#disabletypescriptversioncheck}

При значении `true` компилятор не проверяет версию TypeScript и не сообщает об ошибке при использовании
неподдерживаемой версии TypeScript. Не рекомендуется, поскольку неподдерживаемые версии TypeScript могут иметь
неопределенное поведение. Значение по умолчанию: `false`.

### `enableI18nLegacyMessageIdFormat` {#enablei18nlegacymessageidformat}

Указывает компилятору шаблонов Angular создавать устаревшие ID для сообщений, помеченных в шаблонах атрибутом `i18n`.
Подробнее о разметке сообщений для локализации см. в разделе
[Разметка текста для переводов][GuideI18nCommonPrepareMarkTextInComponentTemplate].

Установите этот параметр в `false`, если ваш проект не опирается на переводы, созданные ранее с использованием
устаревших ID. Значение по умолчанию: `true`.

Инструментарий для извлечения сообщений, созданный до Ivy, генерировал различные устаревшие форматы для извлечённых
ID сообщений. Эти форматы имеют ряд проблем, таких как обработка пробельных символов и зависимость от информации в
исходном HTML шаблона.

Новый формат сообщений более устойчив к изменениям пробельных символов, одинаков для всех форматов файлов переводов
и может быть создан непосредственно из вызовов `$localize`. Это позволяет сообщениям `$localize` в коде приложения
использовать тот же ID, что и идентичные сообщения `i18n` в шаблонах компонентов.

### `enableResourceInlining` {#enableresourceinlining}

При значении `true` заменяет свойства `templateUrl` и `styleUrls` во всех декораторах `@Component` встроенным
содержимым в свойствах `template` и `styles`.

При включении вывод `.js` команды `ngc` не включает никаких лениво загружаемых URL шаблонов или стилей.

Для библиотечных проектов, созданных с помощью Angular CLI, значение по умолчанию для конфигурации разработки: `true`.

### `enableLegacyTemplate` {#enablelegacytemplate}

При значении `true` включает устаревший элемент `<template>` вместо `<ng-template>`. Значение по умолчанию: `false`.
Может требоваться некоторыми сторонними библиотеками Angular.

### `flatModuleId` {#flatmoduleid}

ID модуля для импорта плоского модуля \(когда `flatModuleOutFile` равно `true`\). Ссылки, создаваемые компилятором
шаблонов, используют это имя модуля при импорте символов из плоского модуля. Игнорируется, если `flatModuleOutFile`
равно `false`.

### `flatModuleOutFile` {#flatmoduleoutfile}

При значении `true` генерирует индекс плоского модуля с указанным именем файла и соответствующие метаданные плоского
модуля. Используется для создания плоских модулей, упакованных аналогично `@angular/core` и `@angular/common`. При
использовании этого параметра `package.json` библиотеки должен ссылаться на созданный индекс плоского модуля вместо
файла индекса библиотеки.

Создает только один файл `.metadata.json`, содержащий все метаданные, необходимые для символов, экспортируемых из
индекса библиотеки. В создаваемых файлах `.ngfactory.js` для импорта символов используется индекс плоского модуля,
включающий как публичный API из индекса библиотеки, так и скрытые внутренние символы.

По умолчанию файл `.ts`, указанный в поле `files`, считается индексом библиотеки. Если указано более одного файла
`.ts`, для выбора используемого файла применяется `libraryIndex`. Если указано более одного файла `.ts` без
`libraryIndex`, возникает ошибка.

`.d.ts` и `.js` плоского модуля создаются с именем `flatModuleOutFile` в том же месте, что и файл `.d.ts` индекса
библиотеки.

Например, если библиотека использует файл `public_api.ts` в качестве индекса модуля, поле `files` в `tsconfig.json`
будет `["public_api.ts"]`. Параметр `flatModuleOutFile` можно установить, например, в `"index.js"`, что создаст файлы
`index.d.ts` и `index.metadata.json`. Поле `module` в `package.json` библиотеки будет `"index.js"`, а поле `typings`
— `"index.d.ts"`.

### `fullTemplateTypeCheck` {#fulltemplatetypecheck}

При значении `true` (рекомендуемое значение) включает фазу проверки выражений привязки компилятора шаблонов. Эта фаза
использует TypeScript для проверки выражений привязки. Подробнее см. в разделе
[Проверка типов шаблонов](tools/cli/template-typecheck).

Значение по умолчанию: `false`, но при использовании команды Angular CLI `ng new --strict` устанавливается в `true`
в конфигурации нового проекта.

IMPORTANT: Параметр `fullTemplateTypeCheck` устарел в Angular 13 в пользу семейства параметров компилятора
`strictTemplates`.

### `generateCodeForLibraries` {#generatecodeforlibraries}

При значении `true` создает файлы фабрик \(`.ngfactory.js` и `.ngstyle.js`\) для файлов `.d.ts` с соответствующим
файлом `.metadata.json`. Значение по умолчанию: `true`.

При значении `false` файлы фабрик создаются только для файлов `.ts`. Используйте это при работе с сводками фабрик.

### `preserveWhitespaces` {#preservewhitespaces}

При значении `false` (по умолчанию) удаляет пустые текстовые узлы из скомпилированных шаблонов, что дает меньший
размер генерируемых фабричных модулей шаблонов. Установите в `true`, чтобы сохранять пустые текстовые узлы.

HELPFUL: При использовании гидратации рекомендуется использовать `preserveWhitespaces: false`, что является значением
по умолчанию. Если вы включите сохранение пробельных символов, добавив `preserveWhitespaces: true` в свой tsconfig,
возможны проблемы с гидратацией. Эта конфигурация пока не поддерживается в полной мере. Убедитесь, что это значение
одинаково установлено в файлах tsconfig сервера и клиента. Подробнее см. в
[руководстве по гидратации](guide/hydration#preserve-whitespaces-configuration).

### `skipMetadataEmit` {#skipmetadataemit}

При значении `true` не создает файлы `.metadata.json`. Значение по умолчанию: `false`.

Файлы `.metadata.json` содержат информацию, необходимую компилятору шаблонов из файла `.ts`, которая не включена в
файл `.d.ts`, создаваемый компилятором TypeScript. Эта информация включает, например, содержимое аннотаций, таких как
шаблон компонента, который TypeScript генерирует в файл `.js`, но не в файл `.d.ts`.

Можно установить в `true` при использовании сводок фабрик, поскольку сводки фабрик включают копию информации из файла
`.metadata.json`.

Установите в `true`, если вы используете параметр TypeScript `--outFile`, поскольку файлы метаданных недопустимы для
этого стиля вывода TypeScript. Сообщество Angular не рекомендует использовать `--outFile` с Angular. Вместо этого
используйте бандлер, например [webpack](https://webpack.js.org).

### `skipTemplateCodegen` {#skiptemplatecodegen}

При значении `true` не генерирует файлы `.ngfactory.js` и `.ngstyle.js`. Это отключает большую часть компилятора
шаблонов и отключает отчеты о диагностике шаблонов.

Может использоваться для инструктирования компилятора шаблонов создавать файлы `.metadata.json` для распространения
с npm-пакетом. Это позволяет избежать создания файлов `.ngfactory.js` и `.ngstyle.js`, которые нельзя распространять
через npm.

Для библиотечных проектов, созданных с помощью Angular CLI, значение по умолчанию для конфигурации разработки: `true`.

### `strictMetadataEmit` {#strictmetadataemit}

При значении `true` сообщает об ошибке в файл `.metadata.json`, если `"skipMetadataEmit"` равно `false`. Значение по
умолчанию: `false`. Используйте только когда `"skipMetadataEmit"` равно `false` и `"skipTemplateCodegen"` равно
`true`.

Этот параметр предназначен для проверки файлов `.metadata.json`, генерируемых для бандлинга с npm-пакетом. Проверка
является строгой и может генерировать ошибки для метаданных, которые никогда не вызвали бы ошибки при использовании
компилятором шаблонов. Вы можете подавить ошибку, генерируемую этим параметром для экспортируемого символа, включив
`@dynamic` в комментарий, документирующий символ.

Файлы `.metadata.json` могут содержать ошибки. Компилятор шаблонов сообщает об этих ошибках, если метаданные
используются для определения содержимого аннотации. Сборщик метаданных не может предсказать символы, предназначенные
для использования в аннотации. Он заблаговременно включает узлы ошибок в метаданные для экспортируемых символов.
Компилятор шаблонов затем может использовать узлы ошибок для сообщения об ошибке, если эти символы используются.

Если клиент библиотеки намерен использовать символ в аннотации, компилятор шаблонов обычно не сообщает об этом.
Сообщение появляется после того, как клиент действительно использует символ. Этот параметр позволяет обнаруживать
такие ошибки на этапе сборки библиотеки и используется, например, при создании самих библиотек Angular.

Для библиотечных проектов, созданных с помощью Angular CLI, значение по умолчанию для конфигурации разработки: `true`.

### `strictInjectionParameters` {#strictinjectionparameters}

При значении `true` сообщает об ошибке для предоставленного параметра, чей тип внедрения не может быть определен. При
значении `false` параметры конструктора классов, помеченных `@Injectable`, чей тип не может быть разрешен, выдают
предупреждение. Рекомендуемое значение: `true`, но значение по умолчанию: `false`.

При использовании команды Angular CLI `ng new --strict` устанавливается в `true` в конфигурации создаваемого проекта.

### `strictTemplates` {#stricttemplates}

При значении `true` включает
[строгую проверку типов шаблонов](tools/cli/template-typecheck#strict-mode).

Флаги строгости, включаемые этим параметром, позволяют включать и отключать конкретные типы строгой проверки типов
шаблонов. См. раздел [устранение ошибок шаблонов](tools/cli/template-typecheck#troubleshooting-template-errors).

При использовании команды Angular CLI `ng new --strict` устанавливается в `true` в конфигурации нового проекта.

### `strictStandalone` {#strictstandalone}

При значении `true` сообщает об ошибке, если компонент, директива или pipe не является standalone.

### `trace` {#trace}

При значении `true` выводит дополнительную информацию при компиляции шаблонов. Значение по умолчанию: `false`.

## Параметры командной строки {#command-line-options}

В большинстве случаев вы взаимодействуете с компилятором Angular косвенно через
[Angular CLI](reference/configs/angular-compiler-options). При отладке определенных проблем может оказаться полезным
вызвать компилятор Angular напрямую. Вы можете использовать команду `ngc`, предоставляемую npm-пакетом
`@angular/compiler-cli`, для вызова компилятора из командной строки.

Команда `ngc` является оберткой вокруг команды компилятора TypeScript `tsc`. Компилятор Angular настраивается
преимущественно через `tsconfig.json`, а Angular CLI — преимущественно через `angular.json`.

Помимо файла конфигурации, вы также можете использовать
[параметры командной строки `tsc`](https://www.typescriptlang.org/docs/handbook/compiler-options.html) для настройки
`ngc`.
