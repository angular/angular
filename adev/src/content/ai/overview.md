HELPFUL: Хотите начать разработку в вашей любимой IDE с поддержкой ИИ? <br>Ознакомьтесь с нашими [правилами промптов и лучшими практиками](/ai/develop-with-ai).

Генеративный ИИ (GenAI) с большими языковыми моделями (LLM) позволяет создавать сложные и увлекательные приложения, включая персонализированный контент, интеллектуальные рекомендации, генерацию и понимание медиаконтента, суммаризацию информации и динамическую функциональность.

Ранее разработка подобных возможностей требовала глубокой экспертизы в предметной области и значительных инженерных усилий. Однако новые продукты и SDK снижают порог входа. Angular хорошо подходит для интеграции ИИ в веб-приложения благодаря:

- Мощным шаблонным API Angular, позволяющим создавать динамические, чисто скомпонованные пользовательские интерфейсы из генерируемого контента
- Сильной архитектуре на основе сигналов, предназначенной для динамического управления данными и состоянием
- Бесшовной интеграции Angular с SDK и API ИИ

Это руководство демонстрирует, как можно использовать [Genkit](/ai#build-ai-powered-applications-with-genkit-and-angular), [Firebase AI Logic](/ai#build-ai-powered-applications-with-firebase-ai-logic-and-angular) и [Gemini API](/ai#build-ai-powered-applications-with-gemini-api-and-angular) для внедрения ИИ в Angular-приложения уже сегодня. Руководство поможет быстро начать разработку веб-приложений на основе ИИ, объясняя, как приступить к интеграции ИИ в Angular-приложения. Также приводятся ресурсы — стартовые наборы, примеры кода и рецепты для типичных рабочих процессов — для быстрого старта.

Для начала работы необходимо базовое понимание Angular. Впервые знакомитесь с Angular? Попробуйте наше [руководство по основам](/essentials) или [вводные руководства](/tutorials).

NOTE: Хотя на этой странице рассматриваются интеграции и примеры с продуктами Google AI, инструменты, например Genkit, не привязаны к конкретной модели и позволяют выбирать собственную. Во многих случаях примеры и образцы кода применимы к другим сторонним решениям.

## Начало работы {#getting-started}

Создание приложений на основе ИИ — это новая и стремительно развивающаяся область. Бывает сложно решить, с чего начать и какие технологии выбрать. В следующем разделе предложены три варианта на выбор:

1. _Genkit_ даёт возможность выбирать из [поддерживаемых моделей и интерфейс с унифицированным API](https://genkit.dev) для создания полностековых приложений. Идеально для приложений, требующих сложной серверной логики ИИ, например, персонализированных рекомендаций.

1. _Firebase AI Logic_ предоставляет безопасный клиентский API для моделей Google для создания клиентских приложений или мобильных приложений. Лучший вариант для интерактивных функций ИИ прямо в браузере, таких как анализ текста в реальном времени или базовые чат-боты.

1. _Gemini API_ позволяет создавать приложения, использующие методы и функциональность, предоставляемые через API напрямую — лучший вариант для полностековых приложений. Подходит для приложений, требующих прямого управления моделями ИИ, например, пользовательской генерации изображений или глубокой обработки данных.

### Создание приложений на основе ИИ с Genkit и Angular {#build-ai-powered-applications-with-genkit-and-angular}

[Genkit](https://genkit.dev) — набор инструментов с открытым исходным кодом для создания функций на основе ИИ в веб- и мобильных приложениях. Предлагает унифицированный интерфейс для интеграции AI-моделей от Google, OpenAI, Anthropic, Ollama и других, позволяя изучать и выбирать наилучшие модели для своих нужд. Как серверное решение, для интеграции с Genkit веб-приложениям требуется поддерживаемая серверная среда, например, сервер на основе Node. Создание полностекового приложения с Angular SSR, например, даёт начальный код на стороне сервера.

## Лучшие практики {#best-practices}

### Подключение к провайдерам моделей и защита учётных данных API {#connecting-to-model-providers-and-keeping-your-api-credentials-secure}

При подключении к провайдерам моделей важно хранить секреты API в безопасности. _Никогда не помещайте ключ API в файл, отправляемый клиенту, например `environments.ts`_.

Архитектура приложения определяет, какие AI API и инструменты следует выбирать. В частности, выбор зависит от того, является ли приложение клиентским или серверным. Инструменты, такие как Firebase AI Logic, обеспечивают безопасное подключение к API моделей для клиентского кода. Если вы хотите использовать другой API или предпочитаете другого провайдера моделей, рассмотрите создание прокси-сервера или даже [Cloud Functions for Firebase](https://firebase.google.com/docs/functions) в качестве прокси, чтобы не раскрывать ключи API.

Пример подключения с использованием клиентского приложения см. в коде: [репозиторий примера Firebase AI Logic Angular](https://github.com/angular/examples/tree/main/vertex-ai-firebase-angular-example).

Для серверных подключений к API моделей, требующим ключей API, предпочтительнее использовать менеджер секретов или переменные среды, а не `environments.ts`. Следует соблюдать стандартные лучшие практики защиты ключей API и учётных данных. Firebase теперь предоставляет новый менеджер секретов с последними обновлениями Firebase App Hosting. Подробнее см. в [официальной документации](https://firebase.google.com/docs/app-hosting/configure).

Пример серверного подключения в полностековом приложении см. в коде: [репозиторий Angular AI Example (Genkit and Angular Story Generator)](https://github.com/angular/examples/tree/main/genkit-angular-story-generator).

### Использование вызова инструментов для расширения приложений {#use-tool-calling-to-enhance-apps}

Если вы хотите создавать агентные рабочие процессы, где агенты способны действовать и использовать инструменты для решения задач на основе промптов, используйте «вызов инструментов». Вызов инструментов, также известный как вызов функций — это способ дать LLM возможность выполнять обратные вызовы к приложению, которое его вызвало. Как разработчик, вы определяете, какие инструменты доступны, и контролируете, как и когда они вызываются.

Вызов инструментов дополнительно расширяет веб-приложения, выводя интеграцию ИИ за рамки чат-бота в формате «вопрос-ответ». Фактически можно расширить возможности модели для выполнения вызовов функций через API вызова функций провайдера модели. Доступные инструменты могут использоваться для выполнения более сложных действий в контексте приложения.

В [примере e-commerce](https://github.com/angular/examples/blob/main/vertex-ai-firebase-angular-example/src/app/ai.service.ts#L88) в [репозитории примеров Angular](https://github.com/angular/examples) LLM запрашивает вызовы функций для инвентаря, чтобы получить необходимый контекст для выполнения более сложных задач, например, расчёта стоимости группы товаров в магазине. Область доступного API определяется вами как разработчиком, как и решение о том, вызывать ли запрошенную LLM функцию. Вы сохраняете контроль над потоком выполнения. Например, можно открыть доступ к определённым функциям сервиса, но не ко всем его функциям.

### Обработка недетерминированных ответов {#handling-non-deterministic-responses}

Поскольку модели могут возвращать недетерминированные результаты, приложения следует разрабатывать с учётом этого. Вот несколько стратегий для реализации в приложении:

- Настраивайте промпты и параметры модели (например, [температуру](https://ai.google.dev/gemini-api/docs/prompting-strategies)) для получения более или менее детерминированных ответов. Подробнее см. в [разделе о стратегиях промптинга](https://ai.google.dev/gemini-api/docs/prompting-strategies) на [ai.google.dev](https://ai.google.dev/).
- Используйте стратегию «человек в контуре», когда человек проверяет результаты перед продолжением рабочего процесса. Создавайте рабочие процессы приложения так, чтобы операторы (люди или другие модели) могли проверять результаты и подтверждать ключевые решения.
- Применяйте вызов инструментов (или функций) и ограничения схемы для направления и ограничения ответов модели предопределёнными форматами, повышая предсказуемость ответов.

Даже с учётом этих стратегий и техник в дизайне приложения должны быть предусмотрены разумные запасные варианты. Соблюдайте существующие стандарты устойчивости приложений. Например, неприемлемо, чтобы приложение падало при недоступности ресурса или API. В таком сценарии пользователю отображается сообщение об ошибке и, при необходимости, варианты дальнейших действий. Создание приложений на основе ИИ требует тех же соображений. Убедитесь, что ответ соответствует ожидаемому выводу, и обеспечьте «мягкое приземление» в случае несоответствия с помощью [постепенной деградации](https://developer.mozilla.org/en-US/docs/Glossary/Graceful_degradation). Это также применимо к сбоям API у провайдеров LLM.

Рассмотрим пример: провайдер LLM не отвечает. Возможная стратегия обработки сбоя:

- Сохранить ответ пользователя для использования в сценарии повторной попытки (сейчас или позднее)
- Уведомить пользователя о сбое соответствующим сообщением, не раскрывающим конфиденциальную информацию
- Возобновить разговор позднее, когда сервисы снова станут доступны.
