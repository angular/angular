HELPFUL: Хотите начать разработку в любимой IDE с поддержкой ИИ? <br>Ознакомьтесь с нашими [правилами промптов и лучшими практиками](/ai/develop-with-ai).

Генеративный ИИ (GenAI) с большими языковыми моделями (LLM) обеспечивает создание сложных и привлекательных возможностей приложений, включая персонализированный контент, интеллектуальные рекомендации, генерацию и понимание медиа, суммаризацию информации и динамическую функциональность.

Ранее разработка таких функций требовала глубокой экспертизы в предметной области и значительных инженерных усилий. Однако новые продукты и SDK снижают барьер входа. Angular хорошо подходит для интеграции ИИ в веб-приложение благодаря:

- Мощным API шаблонизации Angular, обеспечивающим создание динамических, чисто скомпонованных UI из сгенерированного контента
- Надёжной, основанной на сигналах архитектуре, разработанной для динамического управления данными и состоянием
- Бесшовной интеграции Angular с AI SDK и API

Это руководство демонстрирует, как можно использовать [Genkit](/ai#build-ai-powered-applications-with-genkit-and-angular), [Firebase AI Logic](/ai#build-ai-powered-applications-with-firebase-ai-logic-and-angular) и [Gemini API](/ai#build-ai-powered-applications-with-gemini-api-and-angular) для интеграции ИИ в Angular-приложения уже сегодня. Руководство даёт быстрый старт в разработке веб-приложений с ИИ, объясняя, как начать интеграцию ИИ в Angular-приложения. Также предоставляются ресурсы, такие как стартовые наборы, примеры кода и рецепты для распространённых рабочих процессов, позволяющие быстро войти в курс дела.

Для начала работы необходимо базовое понимание Angular. Новичок в Angular? Попробуйте наше [руководство по основам](/essentials) или [учебные пособия по началу работы](/tutorials).

NOTE: Хотя на этой странице представлены интеграции и примеры с продуктами Google AI, такие инструменты, как Genkit, не привязаны к конкретной модели и позволяют выбирать собственную модель. Во многих случаях примеры и образцы кода применимы и к другим сторонним решениям.

## Начало работы {#getting-started}

Создание приложений на основе ИИ — новая и быстро развивающаяся область. Бывает сложно решить, с чего начать и какие технологии выбрать. В следующем разделе представлены три варианта на выбор:

1. _Genkit_ даёт возможность выбора [поддерживаемой модели и интерфейса с единым API](https://genkit.dev) для создания полнофункциональных приложений. Идеален для приложений, требующих сложной серверной логики ИИ, например, персонализированных рекомендаций.

1. _Firebase AI Logic_ предоставляет безопасный клиентский API для моделей Google для создания только клиентских приложений или мобильных приложений. Лучший выбор для интерактивных функций ИИ непосредственно в браузере, например, анализа текста в реальном времени или простых чат-ботов.

1. _Gemini API_ позволяет создавать приложения, использующие методы и функциональность, доступные через API напрямую — лучший вариант для полнофункциональных приложений. Подходит для приложений, требующих прямого управления моделями ИИ, например, пользовательской генерации изображений или глубокой обработки данных.

### Создание приложений на основе ИИ с Genkit и Angular {#build-ai-powered-applications-with-genkit-and-angular}

[Genkit](https://genkit.dev) — это набор инструментов с открытым исходным кодом, разработанный для помощи в создании функций на основе ИИ в веб- и мобильных приложениях. Он предлагает единый интерфейс для интеграции моделей ИИ от Google, OpenAI, Anthropic, Ollama и других, чтобы вы могли изучать и выбирать лучшие модели для своих нужд. Как серверное решение, веб-приложениям необходима поддерживаемая серверная среда, например, сервер на основе Node, для интеграции с Genkit. Создание полнофункционального приложения с использованием Angular SSR, например, даёт начальный серверный код.

## Лучшие практики {#best-practices}

### Подключение к провайдерам моделей и обеспечение безопасности учётных данных API {#connecting-to-model-providers-and-keeping-your-api-credentials-secure}

При подключении к провайдерам моделей важно обеспечить безопасность секретов API. _Никогда не помещайте ваш API-ключ в файл, который отправляется клиенту, например `environments.ts`_.

Архитектура вашего приложения определяет, какие AI API и инструменты следует выбирать. Конкретно, выбирайте на основе того, является ли ваше приложение клиентским или серверным. Такие инструменты, как Firebase AI Logic, обеспечивают безопасное подключение к API моделей для клиентского кода. Если вы хотите использовать другой API, отличный от Firebase AI Logic, или предпочитаете другого провайдера модели, рассмотрите создание прокси-сервера или даже [Cloud Functions for Firebase](https://firebase.google.com/docs/functions) для работы в качестве прокси, чтобы не раскрывать ключи API.

Пример подключения с использованием клиентского приложения см. в коде: [репозиторий примеров Firebase AI Logic Angular](https://github.com/angular/examples/tree/main/vertex-ai-firebase-angular-example).

Для серверных подключений к API моделей, требующих ключей API, предпочтительно использовать менеджер секретов или переменную среды, а не `environments.ts`. Следует придерживаться стандартных лучших практик обеспечения безопасности ключей API и учётных данных. Firebase теперь предоставляет новый менеджер секретов с последними обновлениями Firebase App Hosting. Подробнее см. в [официальной документации](https://firebase.google.com/docs/app-hosting/configure).

Пример серверного подключения в полнофункциональном приложении см. в коде: [репозиторий Angular AI Example (Genkit and Angular Story Generator)](https://github.com/angular/examples/tree/main/genkit-angular-story-generator).

### Использование вызовов инструментов для улучшения приложений {#use-tool-calling-to-enhance-apps}

Если вы хотите создавать агентные рабочие процессы, где агенты могут действовать и использовать инструменты для решения задач на основе промптов, используйте «вызовы инструментов». Вызов инструментов, также известный как вызов функций, — это способ предоставить LLM возможность делать запросы обратно в приложение, вызвавшее его. Как разработчик, вы определяете, какие инструменты доступны, и контролируете, как и когда они вызываются.

Вызовы инструментов дополнительно расширяют возможности веб-приложений, выходя за рамки чат-бота в стиле вопрос-ответ. Фактически, вы можете наделить модель возможностью запрашивать вызовы функций с помощью API вызова функций вашего провайдера модели. Доступные инструменты могут использоваться для выполнения более сложных действий в контексте вашего приложения.

В [примере e-commerce](https://github.com/angular/examples/blob/main/vertex-ai-firebase-angular-example/src/app/ai.service.ts#L88) из [репозитория примеров Angular](https://github.com/angular/examples) LLM запрашивает вызовы функций для инвентаря, чтобы получить необходимый контекст для выполнения более сложных задач, таких как расчёт стоимости группы товаров в магазине. Область доступного API определяется вами как разработчиком, как и вопрос о том, вызывать ли функцию, запрошенную LLM. Вы сохраняете контроль над потоком выполнения. Вы можете предоставить доступ к конкретным функциям сервиса, но не ко всем его функциям.

### Обработка недетерминированных ответов {#handling-non-deterministic-responses}

Поскольку модели могут возвращать недетерминированные результаты, приложения должны проектироваться с учётом этого. Вот несколько стратегий для реализации в приложении:

- Настраивайте промпты и параметры модели (такие как [temperature](https://ai.google.dev/gemini-api/docs/prompting-strategies)) для получения более или менее детерминированных ответов. Подробнее см. в [разделе о стратегиях промптов](https://ai.google.dev/gemini-api/docs/prompting-strategies) на [ai.google.dev](https://ai.google.dev/).
- Используйте стратегию «человек в цикле», где человек проверяет результаты перед продолжением рабочего процесса. Создавайте рабочие процессы приложения таким образом, чтобы операторы (люди или другие модели) могли проверять результаты и подтверждать ключевые решения.
- Применяйте вызовы инструментов (или функций) и ограничения схемы для направления и ограничения ответов модели предопределёнными форматами, повышая предсказуемость ответов.

Даже с учётом этих стратегий и методов в дизайне приложения должны быть предусмотрены разумные запасные варианты. Следуйте существующим стандартам устойчивости приложений. Например, недопустимо, чтобы приложение падало, если ресурс или API недоступен. В таком сценарии пользователю отображается сообщение об ошибке и, если применимо, варианты следующих шагов. Создание приложений на основе ИИ требует тех же соображений. Убедитесь, что ответ соответствует ожидаемому выводу, и обеспечьте «безопасное приземление» на случай несоответствия с помощью [плавной деградации](https://developer.mozilla.org/en-US/docs/Glossary/Graceful_degradation). Это также применимо к перебоям в работе API провайдеров LLM.

Рассмотрим пример: провайдер LLM не отвечает. Возможная стратегия для обработки перебоя:

- Сохраните ответ пользователя для использования в сценарии повтора (сейчас или позже)
- Уведомите пользователя о перебое с соответствующим сообщением, не раскрывающим конфиденциальную информацию
- Возобновите разговор позже, когда сервисы снова будут доступны.
