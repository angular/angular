## Эффекты {#effects}

Сигналы полезны тем, что уведомляют заинтересованных потребителей при изменении значения. **Эффект** — это операция, которая запускается при изменении значения одного или нескольких сигналов. Эффект создаётся с помощью функции `effect`:

```ts
effect(() => {
  console.log(`The current count is: ${count()}`);
});
```

Эффекты всегда запускаются **как минимум один раз.** При запуске эффект отслеживает все чтения значений сигналов. При изменении любого из этих значений эффект запускается снова. Аналогично вычисляемым сигналам, эффекты динамически отслеживают свои зависимости и следят только за сигналами, которые были прочитаны при последнем выполнении.

Эффекты всегда выполняются **асинхронно**, в процессе обнаружения изменений.

### Варианты использования эффектов {#use-cases-for-effects}

Эффекты должны быть последним инструментом, к которому вы обращаетесь. Всегда отдавайте предпочтение `computed()` для вычисляемых значений и `linkedSignal()` для значений, которые могут быть как вычислены, так и заданы вручную. Если вы копируете данные из одного сигнала в другой с помощью эффекта, это признак того, что следует перенести источник истины выше и использовать `computed()` или `linkedSignal()`. Эффекты лучше всего подходят для синхронизации состояния сигналов с императивными, нереактивными API.

TIP: Нет ситуаций, где эффект был бы хорошим выбором — есть только ситуации, где он уместен.

- Логирование значений сигналов — для аналитики или отладки.
- Синхронизация данных с различными хранилищами: `window.localStorage`, session storage, cookies и т.д.
- Добавление пользовательского поведения DOM, которое нельзя выразить синтаксисом шаблона.
- Выполнение пользовательского рендеринга в элемент `<canvas>`, библиотеку чартов или другую стороннюю библиотеку UI.

<docs-callout critical title="Когда не использовать эффекты">
Избегайте использования эффектов для распространения изменений состояния. Это может привести к ошибкам `ExpressionChangedAfterItHasBeenChecked`, бесконечным циклическим обновлениям или лишним циклам обнаружения изменений.

Вместо этого используйте сигналы `computed` для моделирования состояния, зависящего от другого состояния.
</docs-callout>

### Контекст внедрения зависимостей {#injection-context}

По умолчанию создание `effect()` допустимо только в [контексте внедрения зависимостей](guide/di/dependency-injection-context) (где есть доступ к функции `inject`). Самый простой способ выполнить это требование — вызвать `effect` в `constructor` компонента, директивы или сервиса:

```ts
@Component({
  /*...*/
})
export class EffectiveCounter {
  readonly count = signal(0);

  constructor() {
    // Register a new effect.
    effect(() => {
      console.log(`The count is: ${this.count()}`);
    });
  }
}
```

Чтобы создать эффект вне конструктора, можно передать `Injector` в `effect` через параметры:

```ts
@Component({
  /*...*/
})
export class EffectiveCounter {
  readonly count = signal(0);
  private injector = inject(Injector);

  initializeLogging(): void {
    effect(
      () => {
        console.log(`The count is: ${this.count()}`);
      },
      {injector: this.injector},
    );
  }
}
```

### Выполнение эффектов {#execution-of-effects}

Angular неявно определяет два вида поведения для эффектов в зависимости от контекста, в котором они были созданы.

«View Effect» (эффект представления) — это `effect`, созданный в контексте инициализации компонента. Это включает эффекты, созданные сервисами, привязанными к инжекторам компонентов.<br>
«Root Effect» (корневой эффект) — создаётся в контексте инициализации сервиса, предоставленного на корневом уровне.

Выполнение обоих видов `effect` связано с процессом обнаружения изменений.

- «View effects» выполняются _до_ проверки соответствующего компонента процессом обнаружения изменений.
- «Root effects» выполняются до проверки всех компонентов процессом обнаружения изменений.

В обоих случаях, если хотя бы одна из зависимостей эффекта изменилась во время его выполнения, эффект запустится повторно перед продолжением процесса обнаружения изменений.

### Уничтожение эффектов {#destroying-effects}

При уничтожении компонента или директивы Angular автоматически очищает все связанные эффекты.

`effect` может быть создан в двух разных контекстах, что влияет на момент его уничтожения:

- «View effect» уничтожается при уничтожении компонента.
- «Root effect» уничтожается при уничтожении приложения.

Эффекты возвращают `EffectRef`. Можно использовать метод `destroy` этой ссылки для ручного уничтожения эффекта. Это можно сочетать с опцией `manualCleanup` при создании эффекта, чтобы отключить автоматическую очистку. Следите за тем, чтобы действительно уничтожать такие эффекты, когда они больше не нужны.

### Функции очистки эффектов {#effect-cleanup-functions}

При уничтожении компонента или директивы Angular автоматически очищает все связанные эффекты.
Эффекты могут запускать длительные операции, которые следует отменять, если эффект уничтожается или запускается повторно до завершения первой операции. При создании эффекта функция может опционально принимать функцию `onCleanup` в качестве первого параметра. Эта функция `onCleanup` позволяет зарегистрировать колбэк, который вызывается перед следующим запуском эффекта или при его уничтожении.

```ts
effect((onCleanup) => {
  const user = currentUser();

  const timer = setTimeout(() => {
    console.log(`1 second ago, the user became ${user}`);
  }, 1000);

  onCleanup(() => {
    clearTimeout(timer);
  });
});
```

## Побочные эффекты для DOM-элементов {#side-effects-on-dom-elements}

Функция `effect` — это универсальный инструмент для выполнения кода в ответ на изменения сигналов. Однако она запускается _до_ того, как Angular обновит DOM. В некоторых ситуациях может потребоваться вручную инспектировать или изменять DOM, либо интегрировать стороннюю библиотеку, требующую прямого доступа к DOM.

Для таких ситуаций можно использовать `afterRenderEffect`. Он работает как `effect`, но запускается после того, как Angular завершит рендеринг и зафиксирует изменения в DOM.

```ts
@Component({
  /*...*/
})
export class MyFancyChart {
  chartData = input.required<ChartData>();
  canvas = viewChild.required<ElementRef<HTMLCanvasElement>>('canvas');
  chart: ChartInstance;

  constructor() {
    // Run a single time to create the chart instance
    afterNextRender({
      write: () => {
        this.chart = initializeChart(this.canvas().nativeElement(), this.chartData());
      },
    });

    // Re-run after DOM has been updated whenever `chartData` changes
    afterRenderEffect(() => {
      this.chart.updateData(this.chartData());
    });
  }
}
```

В этом примере `afterRenderEffect` используется для обновления графика, созданного сторонней библиотекой.

TIP: Зачастую `afterRenderEffect` не нужен для отслеживания изменений DOM. При возможности вместо `effect` или `afterRenderEffect` предпочтительнее использовать API наподобие `ResizeObserver`, `MutationObserver` и `IntersectionObserver`.

### Фазы рендеринга {#render-phases}

Доступ к DOM и его изменение могут влиять на производительность приложения, например вызывая лишние [рефлоу](https://developer.mozilla.org/en-US/docs/Glossary/Reflow).

Для оптимизации этих операций `afterRenderEffect` предлагает четыре фазы для группировки колбэков и их выполнения в оптимальном порядке.

Фазы:

| Фаза             | Описание                                                                                                                                                                                                      |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `earlyRead`      | Используется для чтения DOM перед последующим колбэком записи, например для реализации пользовательского макета, который браузер нативно не поддерживает. Предпочтительно использовать фазу `read`, если чтение может подождать. |
| `write`          | Используется для записи в DOM. **Никогда** не читайте DOM в этой фазе.                                                                                                                                        |
| `mixedReadWrite` | Используется для одновременного чтения и записи в DOM. Никогда не используйте эту фазу, если работу можно разделить между другими фазами.                                                                     |
| `read`           | Используется для чтения DOM. **Никогда** не пишите в DOM в этой фазе.                                                                                                                                         |

Использование этих фаз помогает предотвратить layout thrashing и гарантирует безопасное и эффективное выполнение операций с DOM.

Фазу можно указать, передав объект со свойством `phase` в `afterRender` или `afterNextRender`:

```ts
afterRenderEffect({
  earlyRead: (cleanupFn) => {
    /* ... */
  },
  write: (previousPhaseValue, cleanupFn) => {
    /* ... */
  },
  mixedReadWrite: (previousPhaseValue, cleanupFn) => {
    /* ... */
  },
  read: (previousPhaseValue, cleanupFn) => {
    /* ... */
  },
});
```

CRITICAL: Если фаза не указана, `afterRenderEffect` запускает колбэки в фазе `mixedReadWrite`. Это может ухудшить производительность приложения, вызывая дополнительные рефлоу DOM.

#### Выполнение фаз {#phase-executions}

Колбэк фазы `earlyRead` не принимает параметров. Каждая последующая фаза получает возвращаемое значение колбэка предыдущей фазы в виде Signal. Это позволяет координировать работу между фазами.

Эффекты выполняются в следующем порядке фаз:

1. `earlyRead`
2. `write`
3. `mixedReadWrite`
4. `read`

Если одна из фаз изменяет значение сигнала, отслеживаемого `afterRenderEffect`, затронутые фазы выполняются повторно.

#### Очистка {#cleanup}

Каждая фаза предоставляет функцию колбэка очистки в качестве аргумента. Колбэки очистки выполняются при уничтожении `afterRenderEffect` или перед повторным запуском фазовых эффектов.

### Особенности серверного рендеринга {#server-side-rendering-caveats}

`afterRenderEffect`, аналогично `afterNextRender`/`afterEveryRender`, выполняется только на клиенте.

NOTE: Не гарантируется, что компоненты будут [гидратированы](guide/hydration) до запуска колбэка. Следует проявлять осторожность при прямом чтении или записи DOM и макета.
