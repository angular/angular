# Гидратация {#hydration}

## Что такое гидратация {#what-is-hydration}

Гидратация — это процесс восстановления отрендеренного на сервере приложения на стороне клиента. Он включает повторное использование DOM-структур, отрендеренных на сервере, сохранение состояния приложения, передачу данных, уже полученных сервером, и другие процессы.

## Зачем нужна гидратация? {#why-is-hydration-important}

Гидратация улучшает производительность приложения, избегая лишней работы по повторному созданию DOM-узлов. Вместо этого Angular пытается сопоставить существующие DOM-элементы со структурой приложения во время выполнения и повторно использует DOM-узлы там, где это возможно. Это приводит к повышению производительности, которое можно измерить с помощью статистики [Core Web Vitals (CWV)](https://web.dev/learn-core-web-vitals/), например, сокращением First Input Delay ([FID](https://web.dev/fid/)) и Largest Contentful Paint ([LCP](https://web.dev/lcp/)), а также Cumulative Layout Shift ([CLS](https://web.dev/cls/)). Улучшение этих показателей также влияет на такие вещи, как производительность SEO.

Без включённой гидратации серверно-рендеренные Angular-приложения уничтожают и повторно отрисовывают DOM приложения, что может привести к видимому мерцанию пользовательского интерфейса. Эта повторная отрисовка может негативно влиять на [Core Web Vitals](https://web.dev/learn-core-web-vitals/), такие как [LCP](https://web.dev/lcp/), и вызывать сдвиги макета. Включение гидратации позволяет повторно использовать существующий DOM и предотвращает мерцание.

## Как включить гидратацию в Angular {#how-do-you-enable-hydration-in-angular}

Гидратацию можно включить только для приложений с рендерингом на стороне сервера (SSR). Сначала следуйте [руководству Angular SSR](guide/ssr), чтобы включить рендеринг на стороне сервера.

### Использование Angular CLI {#using-angular-cli}

Если для включения SSR использовался Angular CLI (при создании приложения или позднее через `ng add @angular/ssr`), код, включающий гидратацию, уже должен быть включён в ваше приложение.

### Ручная настройка {#manual-setup}

Если у вас пользовательская настройка и вы не использовали Angular CLI для включения SSR, вы можете включить гидратацию вручную, посетив главный компонент или модуль приложения и импортировав `provideClientHydration` из `@angular/platform-browser`. Затем добавьте этот провайдер в список провайдеров начальной загрузки приложения.

```typescript
import {
  bootstrapApplication,
  provideClientHydration,
} from '@angular/platform-browser';
...

bootstrapApplication(App, {
  providers: [provideClientHydration()]
});
```

Alternatively if you are using NgModules, you would add `provideClientHydration` to your root app module's provider list.

```typescript
import {provideClientHydration} from '@angular/platform-browser';
import {NgModule} from '@angular/core';

@NgModule({
  declarations: [App],
  exports: [App],
  bootstrap: [App],
  providers: [provideClientHydration()],
})
export class AppModule {}
```

IMPORTANT: Убедитесь, что вызов `provideClientHydration()` также включён в набор провайдеров, используемых для начальной загрузки приложения на **сервере**. В приложениях со стандартной структурой проекта (созданной командой `ng new`) добавление вызова в корневой `AppModule` должно быть достаточным, поскольку этот модуль импортируется серверным модулем. Если вы используете пользовательскую настройку, добавьте вызов `provideClientHydration()` в список провайдеров серверной конфигурации начальной загрузки.

### Проверка того, что гидратация включена {#verify-that-hydration-is-enabled}

После настройки гидратации и запуска сервера загрузите приложение в браузере.

HELPFUL: Вероятно, потребуется исправить случаи прямого манипулирования DOM до того, как гидратация заработает полностью — либо переключившись на конструкции Angular, либо использовав `ngSkipHydration`. Подробнее см. в разделах [Ограничения](#constraints), [Прямое манипулирование DOM](#direct-dom-manipulation) и [Как пропустить гидратацию для отдельных компонентов](#how-to-skip-hydration-for-particular-components).

При запуске приложения в режиме разработки можно подтвердить, что гидратация включена, открыв инструменты разработчика в браузере и просмотрев консоль. Должно появиться сообщение со статистикой гидратации, например, количеством гидратированных компонентов и узлов. Angular вычисляет статистику на основе всех компонентов, отрендеренных на странице, включая компоненты из сторонних библиотек.

Также можно использовать [расширение Angular DevTools для браузера](tools/devtools), чтобы увидеть статус гидратации компонентов на странице. Angular DevTools позволяет включить оверлей для отображения того, какие части страницы были гидратированы. При наличии ошибки несоответствия гидратации DevTools также выделит компонент, вызвавший ошибку.

## Захват и воспроизведение событий {#capturing-and-replaying-events}

Когда приложение рендерится на сервере, оно становится видимым в браузере, как только загружается произведённый HTML. Пользователи могут предположить, что могут взаимодействовать со страницей, но обработчики событий не подключаются до завершения гидратации. Начиная с версии v18, можно включить функцию воспроизведения событий (Event Replay), которая позволяет захватывать все события, произошедшие до гидратации, и воспроизводить их после её завершения. Включить её можно с помощью функции `withEventReplay()`, например:

```typescript
import {provideClientHydration, withEventReplay} from '@angular/platform-browser';

bootstrapApplication(App, {
  providers: [provideClientHydration(withEventReplay())],
});
```

### Как работает воспроизведение событий {#how-event-replay-works}

Воспроизведение событий (Event Replay) — это функция, улучшающая пользовательский опыт путём захвата пользовательских событий, произошедших до завершения процесса гидратации. Затем эти события воспроизводятся, гарантируя, что ни одно взаимодействие не было потеряно.

Event Replay делится на три основных фазы:

- **Захват пользовательских взаимодействий**<br>
  До **гидратации** Event Replay захватывает и сохраняет все взаимодействия пользователя, такие как клики и другие нативные события браузера.

- **Хранение событий**<br>
  **Event Contract** хранит в памяти все взаимодействия, записанные на предыдущем шаге, гарантируя, что они не будут потеряны для последующего воспроизведения.

- **Повторный запуск событий**<br>
  После завершения **гидратации** Angular повторно вызывает захваченные события.

Воспроизведение событий поддерживает _нативные события браузера_, например, `click`, `mouseover` и `focusin`. Если вы хотите узнать больше о JSAction — библиотеке, лежащей в основе воспроизведения событий, — можно прочитать подробнее [в readme](https://github.com/angular/angular/tree/main/packages/core/primitives/event-dispatch#readme).

---

Эта функция обеспечивает единообразный пользовательский опыт, предотвращая игнорирование пользовательских действий, выполненных до гидратации.

NOTE: Если включена [инкрементальная гидратация](guide/incremental-hydration), воспроизведение событий автоматически включается под капотом.

## Ограничения {#constraints}

Гидратация накладывает ряд ограничений на ваше приложение, которых нет при отключённой гидратации. Приложение должно иметь одинаковую сгенерированную DOM-структуру как на сервере, так и на клиенте. В процессе гидратации ожидается, что дерево DOM имеет одинаковую структуру в обоих местах. Это также включает пробелы и узлы комментариев, которые Angular создаёт при рендеринге на сервере. Эти пробелы и узлы должны присутствовать в HTML, сгенерированном процессом серверного рендеринга.

IMPORTANT: HTML, произведённый операцией серверного рендеринга, **не должен** изменяться между сервером и клиентом.

Если между структурами DOM сервера и клиента есть несоответствие, процесс гидратации столкнётся с проблемами при попытке сопоставить ожидаемое с тем, что фактически присутствует в DOM. Компоненты, которые напрямую манипулируют DOM с помощью нативных DOM API, являются наиболее частой причиной таких проблем.

### Прямое манипулирование DOM {#direct-dom-manipulation}

Если у вас есть компоненты, которые манипулируют DOM с помощью нативных DOM API или используют `innerHTML` или `outerHTML`, процесс гидратации столкнётся с ошибками. Конкретные случаи, когда манипулирование DOM является проблемой, — это ситуации, такие как обращение к `document`, запросы к конкретным элементам и внедрение дополнительных узлов с помощью `appendChild`. Отсоединение DOM-узлов и их перемещение в другие места также приведёт к ошибкам.

Это происходит потому, что Angular не знает об этих изменениях DOM и не может разрешить их в процессе гидратации. Angular ожидает определённую структуру, но при попытке гидратировать обнаружит другую структуру. Это несоответствие приведёт к сбою гидратации и вызову ошибки несоответствия DOM ([см. ниже](#errors)).

Лучше всего рефакторить компонент, чтобы избежать подобного манипулирования DOM. Постарайтесь использовать Angular API для этой работы, если это возможно. Если рефакторинг невозможен, используйте атрибут `ngSkipHydration` ([описан ниже](#how-to-skip-hydration-for-particular-components)) до тех пор, пока не сможете рефакторировать до решения, совместимого с гидратацией.

### Корректная HTML-структура {#valid-html-structure}

Существует ряд случаев, когда некорректная HTML-структура шаблона компонента может привести к ошибке несоответствия DOM при гидратации.

Вот некоторые из наиболее распространённых случаев этой проблемы.

- `<table>` без `<tbody>`
- `<div>` внутри `<p>`
- `<a>` внутри другого `<a>`

Если вы не уверены в корректности HTML, можно использовать [валидатор синтаксиса](https://validator.w3.org/) для проверки.

NOTE: Хотя стандарт HTML не требует элемент `<tbody>` внутри таблиц, современные браузеры автоматически создают элемент `<tbody>` в таблицах, в которых он не объявлен. Из-за этой несогласованности всегда явно объявляйте элемент `<tbody>` в таблицах, чтобы избежать ошибок гидратации.

### Конфигурация сохранения пробелов {#preserve-whitespaces-configuration}

При использовании функции гидратации рекомендуется использовать значение по умолчанию `false` для параметра `preserveWhitespaces`. Если этот параметр отсутствует в вашем tsconfig, его значение будет `false`, и никаких изменений не требуется. Если вы решите включить сохранение пробелов, добавив `preserveWhitespaces: true` в tsconfig, возможны проблемы с гидратацией. Это ещё не полностью поддерживаемая конфигурация.

HELPFUL: Убедитесь, что этот параметр **согласованно** установлен в `tsconfig.server.json` для сервера и `tsconfig.app.json` для браузерных сборок. Несоответствие значений приведёт к сбою гидратации.

Если вы решите задать этот параметр в tsconfig, рекомендуется устанавливать его только в `tsconfig.app.json`, из которого `tsconfig.server.json` унаследует его по умолчанию.

### Пользовательский или Noop Zone.js пока не поддерживается {#custom-or-noop-zonejs-are-not-yet-supported}

Гидратация опирается на сигнал от Zone.js о том, когда он становится стабильным внутри приложения, чтобы Angular мог начать процесс сериализации на сервере или постгидратационную очистку на клиенте для удаления невостребованных DOM-узлов.

Предоставление пользовательской или «noop» реализации Zone.js может привести к другому времени возникновения события «stable», что вызовет сериализацию или очистку слишком рано или слишком поздно. Это ещё не полностью поддерживаемая конфигурация, и может потребоваться настройка времени события `onStable` в пользовательской реализации Zone.js.

## Ошибки {#errors}

Существует несколько связанных с гидратацией ошибок, начиная от несоответствия узлов и заканчивая случаями использования `ngSkipHydration` на недопустимом хост-узле. Наиболее распространённый случай ошибки возникает из-за прямого манипулирования DOM с использованием нативных API, что приводит к неспособности гидратации найти или сопоставить ожидаемую DOM-структуру на клиенте с той, что была отрендерена сервером. Другой случай, упомянутый ранее в разделе [Корректная HTML-структура](#valid-html-structure). Поэтому убедитесь, что HTML в ваших шаблонах использует корректную структуру, и вы избежите этого случая ошибки.

Полный справочник ошибок гидратации см. в [Руководстве по ошибкам](/errors).

## Как пропустить гидратацию для отдельных компонентов {#how-to-skip-hydration-for-particular-components}

Некоторые компоненты могут работать неправильно при включённой гидратации из-за некоторых упомянутых выше проблем, таких как [Прямое манипулирование DOM](#direct-dom-manipulation). В качестве обходного решения можно добавить атрибут `ngSkipHydration` к тегу компонента, чтобы пропустить гидратацию всего компонента.

```angular-html
<app-example ngSkipHydration />
```

Alternatively you can set `ngSkipHydration` as a host binding.

```typescript
@Component({
  ...
  host: {ngSkipHydration: 'true'},
})
class ExampleComponent {}
```

Атрибут `ngSkipHydration` заставит Angular пропустить гидратацию всего компонента и его дочерних элементов. Использование этого атрибута означает, что компонент будет вести себя так, как будто гидратация не включена, то есть будет уничтожен и повторно отрендерен.

HELPFUL: Это исправит проблемы с рендерингом, но означает, что для этого компонента (и его дочерних элементов) вы не получите преимуществ гидратации. Потребуется скорректировать реализацию компонента, чтобы избежать паттернов, нарушающих гидратацию (т.е. прямого манипулирования DOM), чтобы иметь возможность удалить аннотацию пропуска гидратации.

Атрибут `ngSkipHydration` можно использовать только на хост-узлах компонентов. Angular выдаёт ошибку, если этот атрибут добавлен к другим узлам.

Имейте в виду, что добавление атрибута `ngSkipHydration` к корневому компоненту приложения фактически отключит гидратацию для всего приложения. Будьте осторожны и вдумчивы при использовании этого атрибута. Он предназначен в качестве последнего резерва. Компоненты, нарушающие гидратацию, следует считать ошибками, требующими исправления.

## Время гидратации и стабильность приложения {#hydration-timing-and-application-stability}

Стабильность приложения является важной частью процесса гидратации. Гидратация и любые постгидратационные процессы происходят только после того, как приложение сообщает о стабильности. Существует ряд способов задержать стабильность: таймауты и интервалы, неразрешённые промисы и незавершённые микрозадачи. В таких случаях может возникнуть ошибка [Приложение остаётся нестабильным](errors/NG0506), указывающая на то, что приложение не достигло стабильного состояния через 10 секунд. Если приложение не гидратируется сразу, обратите внимание на то, что влияет на стабильность приложения, и рефакторите код, чтобы избежать этих задержек.

### Отладка стабильности приложения {#debugging-application-stability}

Утилита `provideStabilityDebugging` помогает определить, почему приложение не может стабилизироваться. В режиме разработки при использовании `provideClientHydration` эта утилита предоставляется по умолчанию. Её также можно добавить вручную в провайдеры приложения для использования в production-бандлах или при использовании SSR без гидратации. Функция записывает информацию в консоль, если приложению требуется больше времени, чем ожидалось, для стабилизации.

```typescript
import {provideStabilityDebugging} from '@angular/core';
import {bootstrapApplication} from '@angular/platform-browser';
import 'zone.js/plugins/task-tracking'; // Use if you have Zone.js with `provideZoneChangeDetection`

bootstrapApplication(App, {
  providers: [provideStabilityDebugging()],
});
```

При включении утилита записывает ожидающие задачи (`PendingTasks`) в консоль. Если приложение использует Zone.js, можно также импортировать `zone.js/plugins/task-tracking`, чтобы увидеть, какие макрозадачи не дают Angular Zone стабилизироваться. Этот плагин предоставляет трассировку стека создания макрозадачи, что эффективно помогает определить источник задержки.

IMPORTANT: Angular не удаляет плагин отслеживания задач zone.js или эту утилиту из production-бандлов. Используйте их только для временной отладки проблем стабильности во время разработки, включая оптимизированные production-сборки.

## I18N {#i18n}

HELPFUL: По умолчанию Angular пропустит гидратацию для компонентов, использующих блоки i18n, фактически повторно отрендерив эти компоненты с нуля.

Чтобы включить гидратацию для блоков i18n, можно добавить [`withI18nSupport`](/api/platform-browser/withI18nSupport) в вызов `provideClientHydration`.

```typescript
import {
  bootstrapApplication,
  provideClientHydration,
  withI18nSupport,
} from '@angular/platform-browser';
...

bootstrapApplication(App, {
  providers: [provideClientHydration(withI18nSupport())]
});
```

## Согласованный рендеринг на стороне сервера и клиента {#consistent-rendering-across-server-side-and-client-side}

Избегайте введения блоков `@if` и других условных операторов, отображающих разный контент при рендеринге на стороне сервера и на стороне клиента, например, использования блока `@if` с функцией Angular `isPlatformBrowser`. Эти различия в рендеринге вызывают сдвиги макета, что негативно сказывается на пользовательском опыте и основных показателях веб-производительности.

## Сторонние библиотеки с манипулированием DOM {#third-party-libraries-with-dom-manipulation}

Существует ряд сторонних библиотек, которые зависят от манипулирования DOM для рендеринга. Графики D3 — яркий пример. Эти библиотеки работали без гидратации, но могут вызывать ошибки несоответствия DOM при включении гидратации. Если вы столкнулись с ошибками несоответствия DOM при использовании одной из таких библиотек, можно добавить атрибут `ngSkipHydration` к компоненту, который использует эту библиотеку для рендеринга.

## Сторонние скрипты с манипулированием DOM {#third-party-scripts-with-dom-manipulation}

Многие сторонние скрипты, например трекеры рекламы и аналитики, изменяют DOM до того, как может произойти гидратация. Эти скрипты могут вызывать ошибки гидратации, поскольку страница больше не соответствует структуре, ожидаемой Angular. По возможности старайтесь откладывать загрузку таких скриптов до завершения гидратации. Рассмотрите использование [`AfterNextRender`](api/core/afterNextRender), чтобы задержать выполнение скрипта до завершения постгидратационных процессов.

## Инкрементальная гидратация {#incremental-hydration}

Инкрементальная гидратация — это продвинутая форма гидратации, обеспечивающая более детальный контроль над тем, когда происходит гидратация. Подробнее см. в [руководстве по инкрементальной гидратации](guide/incremental-hydration).
