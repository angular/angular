# Безопасность {#security}

В этом разделе описаны встроенные средства защиты Angular от распространённых уязвимостей и атак на веб-приложения, таких как межсайтовый скриптинг.
Здесь не рассматривается безопасность на уровне приложения — аутентификация и авторизация.

Подробнее об атаках и методах защиты: [руководство Open Web Application Security Project (OWASP)](https://www.owasp.org/index.php/Category:OWASP_Guide_Project).

## Рекомендации {#best-practices}

Ниже приведены рекомендации для обеспечения безопасности Angular-приложения.

1. **Используйте последние версии библиотек Angular** — библиотеки Angular регулярно обновляются, и эти обновления могут исправлять уязвимости, обнаруженные в предыдущих версиях. Следите за [журналом изменений](https://github.com/angular/angular/blob/main/CHANGELOG.md) Angular на предмет обновлений, связанных с безопасностью.
2. **Не изменяйте собственную копию Angular** — приватные, кастомизированные версии Angular отстают от текущей версии и могут не содержать важных исправлений безопасности и улучшений. Вместо этого делитесь улучшениями Angular с сообществом и создавайте pull request.
3. **Избегайте Angular API, помеченных в документации как «_Security Risk_»** — подробнее: раздел [Доверие безопасным значениям](#trusting-safe-values) на этой странице.

## Предотвращение межсайтового скриптинга (XSS) {#preventing-cross-site-scripting-xss}

[Межсайтовый скриптинг (XSS)](https://en.wikipedia.org/wiki/Cross-site_scripting) позволяет злоумышленникам внедрять вредоносный код в веб-страницы.
Такой код может, например, похищать данные пользователей и учётные данные, или выполнять действия от имени пользователя.
Это одна из наиболее распространённых атак в интернете.

Для блокировки XSS-атак необходимо предотвратить попадание вредоносного кода в Document Object Model (DOM).
Например, если злоумышленники смогут заставить вас вставить тег `<script>` в DOM, они смогут выполнить произвольный код на вашем сайте.
Атака не ограничивается тегами `<script>` — многие элементы и свойства DOM допускают выполнение кода: например, `<img alt="" onerror="...">` и `<a href="javascript:...">`.
Если данные, контролируемые злоумышленником, попадают в DOM, следует ожидать уязвимостей безопасности.

### Модель безопасности Angular для защиты от XSS {#angulars-cross-site-scripting-security-model}

Для систематической блокировки XSS-ошибок Angular по умолчанию считает все значения ненадёжными.
При вставке значения в DOM из привязки Шаблона или интерполяции Angular санирует и экранирует ненадёжные значения.
Если значение уже было санировано вне Angular и считается безопасным, сообщите об этом Angular, [пометив значение как доверенное](#trusting-safe-values).

В отличие от значений для рендеринга, Шаблоны Angular по умолчанию считаются доверенными и должны рассматриваться как исполняемый код.
Никогда не создавайте Шаблоны путём конкатенации пользовательского ввода и синтаксиса Шаблона.
Это позволило бы злоумышленникам [внедрять произвольный код](https://en.wikipedia.org/wiki/Code_injection) в приложение.
Для предотвращения таких уязвимостей всегда используйте стандартный компилятор [Ahead-Of-Time (AOT)](#use-the-aot-template-compiler) в production-окружениях.

Дополнительный уровень защиты обеспечивают Content Security Policy и Trusted Types.
Эти возможности веб-платформы работают на уровне DOM — наиболее эффективном месте для предотвращения XSS. Здесь их нельзя обойти через другие, более низкоуровневые API.
По этой причине настоятельно рекомендуется использовать эти возможности. Для этого настройте [политику безопасности содержимого](#content-security-policy) для приложения и включите [принудительное применение Trusted Types](#enforcing-trusted-types).

### Санирование и контексты безопасности {#sanitization-and-security-contexts}

_Санирование_ — это проверка ненадёжного значения с целью превратить его в значение, безопасное для вставки в DOM.
Во многих случаях санирование не изменяет значение вовсе.
Санирование зависит от контекста.
Например, значение, безвредное в CSS, потенциально опасно в URL.

Angular определяет следующие контексты безопасности:

| Контексты безопасности | Описание                                                                               |
| :--------------------- | :------------------------------------------------------------------------------------- |
| HTML                   | Используется при интерпретации значения как HTML, например при привязке к `innerHtml`. |
| Style                  | Используется при привязке CSS к свойству `style`.                                      |
| URL                    | Используется для свойств URL, таких как `<a href>`.                                    |
| Resource URL           | URL, загружаемый и выполняемый как код, например в `<script src>`.                    |

Angular санирует ненадёжные значения для HTML и URL. Санирование URL ресурсов невозможно, поскольку они содержат произвольный код.
В режиме разработки Angular выводит предупреждение в консоль, когда вынужден изменить значение при санировании.

### Прямое использование DOM API и явные вызовы санирования {#direct-use-of-the-dom-apis-and-explicit-sanitization-calls}

Если вы не используете Trusted Types, встроенные браузерные DOM API не защищают автоматически от уязвимостей безопасности.
Например, `document`, узел, доступный через `ElementRef`, и многие сторонние API содержат небезопасные методы.
Аналогично, при взаимодействии с другими библиотеками, манипулирующими DOM, автоматическое санирование, как у Angular, скорее всего, отсутствует.
По возможности избегайте прямого взаимодействия с DOM и используйте Шаблоны Angular.

В случаях, когда это неизбежно, используйте встроенные функции санирования Angular.
Санируйте ненадёжные значения методом [DomSanitizer.sanitize](api/platform-browser/DomSanitizer#sanitize) с указанием соответствующего `SecurityContext`.
Эта функция также принимает значения, помеченные как доверенные с помощью функций `bypassSecurityTrust`, и не санирует их, как [описано ниже](#trusting-safe-values).

### Доверие безопасным значениям {#trusting-safe-values}

Иногда приложению действительно требуется включать исполняемый код, отображать `<iframe>` по какому-либо URL или создавать потенциально опасные URL.
Чтобы предотвратить автоматическое санирование в таких ситуациях, сообщите Angular, что вы проверили значение, убедились в способе его создания и уверены в его безопасности.
Будьте _осторожны_.
Если вы доверяете значению, которое может быть вредоносным, вы вносите уязвимость безопасности в приложение.
При сомнениях обратитесь к профессиональному аудитору безопасности.

Чтобы пометить значение как доверенное, внедрите `DomSanitizer` и вызовите один из следующих методов:

- `bypassSecurityTrustHtml`
- `bypassSecurityTrustScript`
- `bypassSecurityTrustStyle`
- `bypassSecurityTrustUrl`
- `bypassSecurityTrustResourceUrl`

### Политика безопасности содержимого {#content-security-policy}

Content Security Policy \(CSP\) — это техника глубокоэшелонированной защиты для предотвращения XSS.
Чтобы включить CSP, настройте веб-сервер на возврат соответствующего HTTP-заголовка `Content-Security-Policy`.
Подробнее о политике безопасности содержимого: [руководство Web Fundamentals](https://developers.google.com/web/fundamentals/security/csp) на сайте Google Developers.

Минимальная политика, необходимая для нового Angular-приложения:

```txt
default-src 'self'; style-src 'self' 'nonce-randomNonceGoesHere'; script-src 'self' 'nonce-randomNonceGoesHere';
```

При обслуживании Angular-приложения сервер должен включать случайно сгенерированный nonce в HTTP-заголовок для каждого запроса.
Этот nonce необходимо предоставить Angular, чтобы фреймворк мог рендерить элементы `<style>`.
Задать nonce для Angular можно одним из следующих способов:

1. Установить опцию `autoCsp` в `true` в [конфигурации рабочего пространства](reference/configs/workspace-config#extra-build-and-test-options).
1. Установить атрибут `ngCspNonce` на корневом элементе приложения: `<app ngCspNonce="randomNonceGoesHere"></app>`. Используйте этот подход, если есть доступ к серверному шаблонизатору, который может добавить nonce как в заголовок, так и в `index.html` при формировании ответа.
1. Предоставить nonce через инъекционный токен `CSP_NONCE`. Используйте этот подход, если nonce доступен во время выполнения и вы хотите иметь возможность кешировать `index.html`.

```ts
import {bootstrapApplication, CSP_NONCE} from '@angular/core';
import {AppComponent} from './app/app.component';

bootstrapApplication(AppComponent, {
  providers: [
    {
      provide: CSP_NONCE,
      useValue: globalThis.myRandomNonceValue,
    },
  ],
});
```

<docs-callout title="Уникальные nonce">

Всегда убеждайтесь, что предоставляемые nonce являются <strong>уникальными для каждого запроса</strong> и не предсказуемы.
Если злоумышленник сможет предсказать будущие nonce, он сможет обойти защиту CSP.

</docs-callout>

NOTE: Если вы хотите [встраивать критический CSS](/tools/cli/build#critical-css-inlining) в приложение, нельзя использовать токен `CSP_NONCE` — предпочтительнее использовать опцию `autoCsp` или установить атрибут `ngCspNonce` на корневом элементе приложения.

Если генерация nonce в проекте невозможна, можно разрешить встроенные стили, добавив `'unsafe-inline'` в раздел `style-src` заголовка CSP.

| Разделы                                          | Описание                                                                                                                                                                                                              |
| :----------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `default-src 'self';`                            | Разрешает странице загружать все необходимые ресурсы с того же источника.                                                                                                                                             |
| `style-src 'self' 'nonce-randomNonceGoesHere';`  | Разрешает странице загружать глобальные стили с того же источника \(`'self'`\) и стили, вставляемые Angular с `nonce-randomNonceGoesHere`.                                                                            |
| `script-src 'self' 'nonce-randomNonceGoesHere';` | Разрешает странице загружать JavaScript с того же источника \(`'self'`\) и скрипты, вставляемые Angular CLI с `nonce-randomNonceGoesHere`. Это требуется только при использовании встраивания критического CSS.       |

Angular сам по себе требует только этих настроек для корректной работы.
По мере роста проекта может потребоваться расширить настройки CSP для поддержки дополнительных функций вашего приложения.

### Принудительное применение Trusted Types {#enforcing-trusted-types}

Рекомендуется использовать [Trusted Types](https://w3c.github.io/trusted-types/dist/spec/) для защиты приложений от XSS-атак.
Trusted Types — это возможность [веб-платформы](https://en.wikipedia.org/wiki/Web_platform), помогающая предотвращать XSS-атаки за счёт применения более безопасных практик написания кода.
Trusted Types также упрощает аудит кода приложения.

<docs-callout title="Trusted types">

Trusted Types могут быть недоступны во всех браузерах, которые использует ваше приложение.
Если приложение с включёнными Trusted Types запущено в браузере без поддержки Trusted Types, функциональность приложения сохраняется. Приложение защищено от XSS через `DomSanitizer` Angular.
Текущую поддержку браузерами: [caniuse.com/trusted-types](https://caniuse.com/trusted-types).

</docs-callout>

Чтобы принудительно применять Trusted Types в приложении, необходимо настроить веб-сервер на отправку HTTP-заголовков с одной из следующих политик Angular:

| Политики                 | Описание                                                                                                                                                                                                                                                                                     |
| :----------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `angular`                | Эта политика используется в проверенном с точки зрения безопасности коде, внутреннем для Angular, и необходима для работы Angular при включённых Trusted Types. Любые встроенные значения Шаблона или контент, санированный Angular, считается безопасным согласно этой политике.           |
| `angular#bundler`        | Эта политика используется бандлером Angular CLI при создании lazy chunk-файлов.                                                                                                                                                                                                              |
| `angular#unsafe-bypass`  | Эта политика используется в приложениях, применяющих любые методы [DomSanitizer](api/platform-browser/DomSanitizer) Angular, обходящие безопасность, например `bypassSecurityTrustHtml`. Любое приложение, использующее эти методы, должно включить эту политику.                           |
| `angular#unsafe-jit`     | Эта политика используется [Just-In-Time (JIT) компилятором](api/core/Compiler). Необходимо включить, если приложение напрямую взаимодействует с JIT-компилятором или работает в JIT-режиме через [platform browser dynamic](api/platform-browser-dynamic/platformBrowserDynamic).           |
| `angular#unsafe-upgrade` | Эта политика используется пакетом [@angular/upgrade](api/upgrade/static/UpgradeModule). Необходимо включить, если приложение является гибридным AngularJS-приложением.                                                                                                                       |

Настройте HTTP-заголовки для Trusted Types в следующих местах:

- Производственная инфраструктура
- Angular CLI \(`ng serve`\) — через свойство `headers` в файле `angular.json` — для локальной разработки и end-to-end тестирования
- Karma \(`ng test`\) — через свойство `customHeaders` в файле `karma.config.js` — для юнит-тестирования

Пример заголовка, специально настроенного для Trusted Types и Angular:

```html
Content-Security-Policy: trusted-types angular; require-trusted-types-for 'script';
```

Пример заголовка для приложений Angular, использующих методы [DomSanitizer](api/platform-browser/DomSanitizer), обходящие безопасность:

```html
Content-Security-Policy: trusted-types angular angular#unsafe-bypass; require-trusted-types-for
'script';
```

Пример заголовка для приложений Angular с JIT:

```html
Content-Security-Policy: trusted-types angular angular#unsafe-jit; require-trusted-types-for
'script';
```

Пример заголовка для приложений Angular с Ленивой загрузкой модулей:

```html
Content-Security-Policy: trusted-types angular angular#bundler; require-trusted-types-for 'script';
```

### Использование AOT-компилятора Шаблонов {#use-the-aot-template-compiler}

AOT-компилятор Шаблонов предотвращает целый класс уязвимостей — инъекции Шаблонов — и значительно улучшает производительность приложения.
AOT-компилятор Шаблонов является компилятором по умолчанию в приложениях Angular CLI, и его следует использовать во всех production-окружениях.

Альтернативой AOT-компилятору является JIT-компилятор, который компилирует Шаблоны в исполняемый код Шаблона в браузере во время выполнения.
Angular доверяет коду Шаблона, поэтому динамическое создание Шаблонов и их компиляция — особенно Шаблонов, содержащих пользовательские данные — обходит встроенные средства защиты Angular. Это антипаттерн безопасности.
Информация о безопасном динамическом создании форм: [руководство по динамическим формам](guide/forms/dynamic-forms).

### Защита от XSS на стороне сервера {#server-side-xss-protection}

HTML, сформированный на сервере, уязвим к инъекционным атакам.
Внедрение кода Шаблона в Angular-приложение равнозначно внедрению исполняемого кода:
злоумышленник получает полный контроль над приложением.
Для предотвращения этого используйте язык шаблонизации, автоматически экранирующий значения для предотвращения XSS-уязвимостей на сервере.
Не создавайте Angular-шаблоны на стороне сервера с использованием языка шаблонизации. Это сопряжено с высоким риском возникновения уязвимостей инъекции Шаблонов.

## Уязвимости на уровне HTTP {#http-level-vulnerabilities}

Angular имеет встроенную поддержку защиты от двух распространённых HTTP-уязвимостей: межсайтовой подделки запросов \(CSRF или XSRF\) и включения межсайтовых скриптов \(XSSI\).
Обе угрозы должны устраняться прежде всего на стороне сервера, но Angular предоставляет вспомогательные средства для упрощения интеграции на стороне клиента.

### Межсайтовая подделка запросов {#cross-site-request-forgery}

При межсайтовой подделке запросов \(CSRF или XSRF\) злоумышленник обманом заставляет пользователя посетить другую веб-страницу \(например, `evil.com`\) с вредоносным кодом. Эта веб-страница тайно отправляет вредоносный запрос к веб-серверу приложения \(например, `example-bank.com`\).

Предположим, что пользователь вошёл в приложение на `example-bank.com`.
Пользователь открывает письмо и нажимает ссылку на `evil.com`, которая открывается в новой вкладке.

Страница `evil.com` немедленно отправляет вредоносный запрос на `example-bank.com`.
Это может быть, например, запрос на перевод денег со счёта пользователя на счёт злоумышленника.
Браузер автоматически отправляет куки `example-bank.com`, включая куки аутентификации, с этим запросом.

Если сервер `example-bank.com` не защищён от XSRF, он не сможет отличить легитимный запрос от приложения от поддельного запроса с `evil.com`.

Для предотвращения этого приложение должно гарантировать, что запрос пользователя исходит от реального приложения, а не с другого сайта.
Сервер и клиент должны взаимодействовать для предотвращения этой атаки.

В распространённой технике защиты от XSRF сервер приложения отправляет случайно созданный токен аутентификации в куки.
Клиентский код считывает куки и добавляет пользовательский заголовок запроса с токеном во все последующие запросы.
Сервер сравнивает полученное значение куки со значением заголовка запроса и отклоняет запрос, если значения отсутствуют или не совпадают.

Этот метод эффективен, поскольку все браузеры реализуют политику _same origin_ (единого источника).
Только код с сайта, для которого установлены куки, может читать эти куки и устанавливать пользовательские заголовки в запросах к этому сайту.
Это означает, что только ваше приложение может прочитать этот токен куки и установить пользовательский заголовок.
Вредоносный код на `evil.com` не может.

### Безопасность XSRF/CSRF в `HttpClient` {#httpclient-xsrfcsrf-security}

`HttpClient` поддерживает [распространённый механизм](https://en.wikipedia.org/wiki/Cross-site_request_forgery#Cookie-to-header_token) защиты от XSRF-атак. При выполнении HTTP-запросов перехватчик считывает токен из куки (по умолчанию `XSRF-TOKEN`) и устанавливает его в HTTP-заголовке `X-XSRF-TOKEN`. Поскольку только код, выполняющийся в вашем домене, может читать куки, бэкенд может быть уверен, что HTTP-запрос пришёл от клиентского приложения, а не от злоумышленника.

По умолчанию перехватчик отправляет этот заголовок во всех мутирующих запросах (например, `POST`) на относительные и одноимённые URL, но не в запросах `GET` или `HEAD`.

<docs-callout helpful title="Почему GET-запросы не защищены?">
Защита от CSRF нужна только для запросов, изменяющих состояние на бэкенде. По своей природе CSRF-атаки пересекают границы доменов, и политика [same-origin](https://developer.mozilla.org/docs/Web/Security/Same-origin_policy) браузера предотвращает получение атакующей страницей результатов аутентифицированных GET-запросов.
</docs-callout>

Чтобы воспользоваться этим, сервер должен установить токен в читаемом JavaScript cookie под названием `XSRF-TOKEN` при загрузке страницы или при первом GET-запросе. В последующих запросах сервер может проверить, что куки соответствует HTTP-заголовку `X-XSRF-TOKEN`, и тем самым убедиться, что запрос мог отправить только код, выполняющийся в вашем домене. Токен должен быть уникальным для каждого пользователя и проверяемым сервером — это предотвращает создание клиентом собственных токенов. Для повышения безопасности установите токен как дайджест куки аутентификации сайта с солью.

Чтобы избежать конфликтов в средах, где несколько Angular-приложений используют один домен или поддомен, дайте каждому приложению уникальное имя куки.

<docs-callout important title="HttpClient поддерживает только клиентскую половину схемы защиты XSRF">
  Ваш бэкенд-сервис должен быть настроен на установку куки для вашей страницы и проверку наличия заголовка во всех соответствующих запросах. Несоблюдение этого делает стандартную защиту Angular неэффективной.
</docs-callout>

### Настройка пользовательских имён куки/заголовка {#configure-custom-cookieheader-names}

Если ваш бэкенд-сервис использует другие имена для куки или заголовка XSRF-токена, используйте `withXsrfConfiguration` для переопределения значений по умолчанию.

Добавьте это в вызов `provideHttpClient` следующим образом:

```ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withXsrfConfiguration({
        cookieName: 'CUSTOM_XSRF_TOKEN',
        headerName: 'X-Custom-Xsrf-Header',
      }),
    ),
  ],
};
```

### Отключение защиты XSRF {#disabling-xsrf-protection}

Если встроенный механизм защиты XSRF не работает в вашем приложении, его можно отключить с помощью возможности `withNoXsrfProtection`:

```ts
export const appConfig: ApplicationConfig = {
  providers: [provideHttpClient(withNoXsrfProtection())],
};
```

Информация о CSRF в Open Web Application Security Project \(OWASP\): [Cross-Site Request Forgery (CSRF)](https://owasp.org/www-community/attacks/csrf) и [Cross-Site Request Forgery (CSRF) Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html).
Исчерпывающий источник сведений: статья Стэнфордского университета [Robust Defenses for Cross-Site Request Forgery](https://seclab.stanford.edu/websec/csrf/csrf.pdf).

Также см. доклад Дэйва Смита [о XSRF на AngularConnect 2016](https://www.youtube.com/watch?v=9inczw6qtpY 'Cross Site Request Funkery Securing Your Angular Apps From Evil Doers').

### Включение межсайтовых скриптов (XSSI) {#cross-site-script-inclusion-xssi}

Включение межсайтовых скриптов (XSSI), также известное как уязвимость JSON, может позволить сайту злоумышленника читать данные из JSON API.
Атака работает в старых браузерах путём переопределения встроенных конструкторов объектов JavaScript с последующим включением URL API через тег `<script>`.

Атака успешна только в том случае, если возвращаемый JSON является исполняемым JavaScript.
Серверы могут предотвратить атаку, добавляя ко всем JSON-ответам префикс, делающий их неисполняемыми. По соглашению используется строка `")]}',\n"`.

Библиотека `HttpClient` Angular распознаёт это соглашение и автоматически удаляет строку `")]}',\n"` из всех ответов перед дальнейшим разбором.

Подробнее: раздел XSSI в [блоге Google о безопасности веб-сайтов](https://security.googleblog.com/2011/05/website-security-for-webmasters.html).

## Предотвращение Server-Side Request Forgery (SSRF) {#preventing-server-side-request-forgery-ssrf}

Angular включает строгую валидацию заголовков `Host`, `X-Forwarded-Host`, `X-Forwarded-Proto`, `X-Forwarded-Prefix` и `X-Forwarded-Port` в конвейере обработки запросов для предотвращения атак [Server-Side Request Forgery (SSRF)](https://developer.mozilla.org/en-US/docs/Web/Security/Attacks/SSRF) на основе заголовков.

Правила валидации:

- Заголовки `Host` и `X-Forwarded-Host` проверяются по строгому списку разрешённых значений и не могут содержать разделители пути.
- Заголовок `X-Forwarded-Port` должен быть числовым.
- Заголовок `X-Forwarded-Proto` должен иметь значение `http` или `https`.
- Заголовок `X-Forwarded-Prefix` не должен начинаться с нескольких символов `/` или `\` и не должен содержать сегменты пути `.` или `..`.

Недопустимые или запрещённые заголовки теперь вызывают запись ошибки в лог. Запросы с нераспознанными именами хостов приводят к возврату страницы с CSR, если задано `allowedHosts`; в противном случае возвращается ошибка `400 Bad Request`. Обратите внимание, что в будущем крупном релизе все нераспознанные имена хостов по умолчанию будут возвращать `400 Bad Request` независимо от настроек `allowedHosts`.

NOTE: Большинство облачных провайдеров и CDN-провайдеров автоматически валидируют эти заголовки до того, как запрос достигает источника приложения. Такая встроенная фильтрация значительно снижает практическую поверхность атаки.

### Настройка разрешённых хостов {#configuring-allowed-hosts}

Чтобы разрешить конкретные имена хостов, необходимо добавить их в список разрешённых. Это критически важно для корректной и безопасной работы приложения при развёртывании. Шаблоны поддерживают подстановочные знаки для гибкого сопоставления имён хостов.

Можно настроить опцию `allowedHosts` в `angular.json`:

```json {hideCopy}
{
  // ...
  "projects": {
    "your-project-name": {
      // ...
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "security": {
              "allowedHosts": [
                "example.com",
                "*.example.com" // разрешает все поддомены example.com
              ]
            }
            // ... другие опции
          }
        }
      }
    }
  }
}
```

Также можно настроить `allowedHosts` при инициализации движка приложения:

```typescript
const appEngine = new AngularAppEngine({
  allowedHosts: ['example.com', '*.trusted-example.com'],
});

const nodeAppEngine = new AngularNodeAppEngine({
  allowedHosts: ['example.com', '*.trusted-example.com'],
});
```

Для Node.js-варианта `AngularNodeAppEngine` также можно использовать переменную окружения `NG_ALLOWED_HOSTS` (разделённый запятыми список) для авторизации хостов.

```bash {hideDollar}
export NG_ALLOWED_HOSTS="example.com,*.trusted-example.com"
```

## Аудит Angular-приложений {#auditing-angular-applications}

Angular-приложения должны следовать тем же принципам безопасности, что и обычные веб-приложения, и должны аудироваться соответствующим образом.
Angular API, требующие аудита при проверке безопасности, — например, методы [_bypassSecurityTrust_](#trusting-safe-values) — помечены в документации как чувствительные с точки зрения безопасности.
