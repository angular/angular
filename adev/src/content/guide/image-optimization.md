# Начало работы с NgOptimizedImage

Директива `NgOptimizedImage` упрощает применение лучших практик производительности при загрузке изображений.

Директива обеспечивает приоритетную загрузку изображения [Largest Contentful Paint (LCP)](http://web.dev/lcp) с помощью:

- Автоматической установки атрибута `fetchpriority` на теге `<img>`
- Ленивой загрузки остальных изображений по умолчанию
- Автоматической генерации тега предварительного подключения в заголовке документа
- Автоматической генерации атрибута `srcset`
- Генерации [подсказки предзагрузки](https://developer.mozilla.org/docs/Web/HTML/Link_types/preload), если приложение использует SSR

Помимо оптимизации загрузки LCP-изображения, `NgOptimizedImage` применяет ряд лучших практик работы с изображениями, таких как:

- Использование [URL-адресов CDN изображений для применения оптимизаций](https://web.dev/image-cdns/#how-image-cdns-use-urls-to-indicate-optimization-options)
- Предотвращение сдвига макета путём обязательного указания `width` и `height`
- Предупреждение при неправильном указании `width` или `height`
- Предупреждение, если изображение будет визуально искажено при рендеринге

Если вы используете фоновое изображение в CSS, [начните отсюда](#how-to-migrate-your-background-image).

**ПРИМЕЧАНИЕ: Хотя директива `NgOptimizedImage` стала стабильной функцией в Angular версии 15, она была портирована и доступна как стабильная функция также в версиях 13.4.0 и 14.3.0.**

## Начало работы {#getting-started}

<docs-workflow>
<docs-step title="Import `NgOptimizedImage` directive">
Импортируйте директиву `NgOptimizedImage` из `@angular/common`:

```ts
import {NgOptimizedImage} from '@angular/common';
```

и включите её в массив `imports` standalone-компонента или NgModule:

```ts
imports: [
  NgOptimizedImage,
  // ...
],
```

</docs-step>
<docs-step title="(Optional) Set up a Loader">
Загрузчик изображений **не обязателен** для использования NgOptimizedImage, но использование его с CDN изображений открывает мощные функции производительности, включая автоматические `srcset` для ваших изображений.

Краткое руководство по настройке загрузчика можно найти в разделе [Настройка загрузчика изображений](#configuring-an-image-loader-for-ngoptimizedimage) в конце этой страницы.
</docs-step>
<docs-step title="Enable the directive">
Чтобы активировать директиву `NgOptimizedImage`, замените атрибут `src` изображения на `ngSrc`.

```html
<img ngSrc="cat.jpg" />
```

Если вы используете [встроенный сторонний загрузчик](#built-in-loaders), убедитесь, что путь базового URL не включён в `src`, так как он будет добавлен загрузчиком автоматически.
</docs-step>
<docs-step title="Mark images as `priority`">
Всегда помечайте [LCP-изображение](https://web.dev/lcp/#what-elements-are-considered) на вашей странице как `priority` для приоритизации его загрузки.

```html
<img ngSrc="cat.jpg" width="400" height="200" priority />
```

Пометка изображения как `priority` применяет следующие оптимизации:

- Устанавливает `fetchpriority=high` (подробнее о подсказках приоритета [здесь](https://web.dev/priority-hints))
- Устанавливает `loading=eager` (подробнее о встроенной ленивой загрузке [здесь](https://web.dev/browser-level-image-lazy-loading))
- Автоматически генерирует [элемент предзагрузки](https://developer.mozilla.org/docs/Web/HTML/Link_types/preload), если используется [рендеринг на сервере](guide/ssr).

Angular отображает предупреждение в режиме разработки, если LCP-элемент является изображением без атрибута `priority`. LCP-элемент страницы может варьироваться в зависимости от ряда факторов — например, от размеров экрана пользователя, поэтому страница может содержать несколько изображений, которые следует пометить как `priority`. Подробнее см. в [CSS for Web Vitals](https://web.dev/css-web-vitals/#images-and-largest-contentful-paint-lcp).
</docs-step>
<docs-step title="Include Width and Height">
Чтобы предотвратить [сдвиги макета, связанные с изображениями](https://web.dev/css-web-vitals/#images-and-layout-shifts), NgOptimizedImage требует указания высоты и ширины изображения следующим образом:

```html
<img ngSrc="cat.jpg" width="400" height="200" />
```

Для **адаптивных изображений** (изображений, стилизованных для увеличения и уменьшения относительно области просмотра) атрибуты `width` и `height` должны соответствовать внутреннему размеру файла изображения. Для адаптивных изображений также важно [задать значение `sizes`.](#responsive-images)

Для **изображений фиксированного размера** атрибуты `width` и `height` должны отражать желаемый рендеримый размер изображения. Соотношение сторон этих атрибутов всегда должно соответствовать внутреннему соотношению сторон изображения.

NOTE: Если вы не знаете размер изображений, рассмотрите использование «режима fill» для наследования размера родительского контейнера, как описано ниже.
</docs-step>
</docs-workflow>

## Использование режима `fill` {#using-fill-mode}

В случаях, когда изображение должно заполнять содержащий элемент, можно использовать атрибут `fill`. Это часто полезно, когда требуется достичь поведения «фонового изображения». Это также может быть полезно, когда вы не знаете точные ширину и высоту изображения, но у вас есть родительский контейнер с известным размером, в который нужно вписать изображение (см. «object-fit» ниже).

При добавлении атрибута `fill` к изображению не нужно и не следует включать `width` и `height`, как в этом примере:

```html
<img ngSrc="cat.jpg" fill />
```

Вы можете использовать CSS-свойство [object-fit](https://developer.mozilla.org/docs/Web/CSS/object-fit), чтобы изменить то, как изображение заполняет контейнер. При стилизации изображения с `object-fit: "contain"` изображение сохраняет своё соотношение сторон и вписывается в элемент с «почтовыми марками». При `object-fit: "cover"` элемент сохраняет своё соотношение сторон, полностью заполняет элемент, и часть контента может быть «обрезана».

Визуальные примеры см. в [документации MDN по object-fit.](https://developer.mozilla.org/docs/Web/CSS/object-fit)

Также можно стилизовать изображение с помощью [свойства object-position](https://developer.mozilla.org/docs/Web/CSS/object-position) для корректировки его положения внутри содержащего элемента.

IMPORTANT: Чтобы изображение с fill корректно отображалось, его родительский элемент **должен** иметь стиль `position: "relative"`, `position: "fixed"` или `position: "absolute"`.

## Как перейти с фонового изображения {#how-to-migrate-your-background-image}

Вот простой пошаговый процесс перехода с `background-image` на `NgOptimizedImage`. В этих шагах мы называем элемент с фоновым изображением «содержащим элементом»:

1. Удалите стиль `background-image` из содержащего элемента.
2. Убедитесь, что содержащий элемент имеет `position: "relative"`, `position: "fixed"` или `position: "absolute"`.
3. Создайте новый элемент изображения как дочерний содержащего элемента, используя `ngSrc` для включения директивы `NgOptimizedImage`.
4. Дайте этому элементу атрибут `fill`. Не включайте `height` и `width`.
5. Если вы считаете, что это изображение может быть вашим [LCP-элементом](https://web.dev/lcp/), добавьте атрибут `priority` к элементу изображения.

Вы можете настроить, как фоновое изображение заполняет контейнер, как описано в разделе [Использование режима fill](#using-fill-mode).

## Использование заполнителей {#using-placeholders}

### Автоматические заполнители {#automatic-placeholders}

NgOptimizedImage может отображать автоматический заполнитель с низким разрешением для вашего изображения, если вы используете CDN или хостинг изображений, предоставляющий автоматическое изменение размера изображений. Воспользуйтесь этой функцией, добавив атрибут `placeholder` к изображению:

```html
<img ngSrc="cat.jpg" width="400" height="200" placeholder />
```

Добавление этого атрибута автоматически запрашивает вторую, меньшую версию изображения с помощью указанного загрузчика изображений. Это маленькое изображение будет применено как стиль `background-image` с CSS-размытием во время загрузки изображения. Если загрузчик изображений не указан, изображение-заполнитель не может быть сгенерировано и будет выброшена ошибка.

Размер по умолчанию для генерируемых заполнителей составляет 30 пикселей в ширину. Этот размер можно изменить, указав значение в пикселях в провайдере `IMAGE_CONFIG`, как показано ниже:

```ts
providers: [
  {
    provide: IMAGE_CONFIG,
    useValue: {
      placeholderResolution: 40
    }
  },
],
```

Если вам нужны чёткие края вокруг размытого заполнителя, оберните изображение в содержащий `<div>` со стилем `overflow: hidden`. Пока `<div>` имеет тот же размер, что и изображение (например, используя стиль `width: fit-content`), «нечёткие края» заполнителя будут скрыты.

### Заполнители Data URL {#data-url-placeholders}

Также можно указать заполнитель с помощью base64 [data URL](https://developer.mozilla.org/docs/Web/HTTP/Basics_of_HTTP/Data_URLs) без загрузчика изображений. Формат data URL: `data:image/[imagetype];[data]`, где `[imagetype]` — формат изображения, например `png`, а `[data]` — base64-кодировка изображения. Это кодирование можно выполнить из командной строки или JavaScript. Конкретные команды см. в [документации MDN](https://developer.mozilla.org/docs/Web/HTTP/Basics_of_HTTP/Data_URLs#encoding_data_into_base64_format). Пример заполнителя data URL с усечёнными данными показан ниже:

```html
<img ngSrc="cat.jpg" width="400" height="200" placeholder="data:image/png;base64,iVBORw0K..." />
```

Однако большие data URL увеличивают размер бандлов Angular и замедляют загрузку страницы. Если невозможно использовать загрузчик изображений, команда Angular рекомендует хранить изображения-заполнители в формате base64 меньше 4 КБ и использовать их исключительно для критических изображений. В дополнение к уменьшению размеров заполнителей рассмотрите изменение форматов изображений или параметров, используемых при сохранении изображений. При очень низких разрешениях эти параметры могут существенно влиять на размер файла.

### Заполнители без размытия {#non-blurred-placeholders}

По умолчанию NgOptimizedImage применяет CSS-эффект размытия к заполнителям изображений. Чтобы отобразить заполнитель без размытия, укажите аргумент `placeholderConfig` с объектом, включающим свойство `blur`, установленное в false. Например:

```html
<img ngSrc="cat.jpg" width="400" height="200" placeholder [placeholderConfig]="{blur: false}" />
```

## Корректировка стилей изображения {#adjusting-image-styling}

В зависимости от стилей изображения, добавление атрибутов `width` и `height` может привести к тому, что изображение рендерится иначе. `NgOptimizedImage` предупреждает вас, если стили изображения рендерят его с искажённым соотношением сторон.

Обычно это можно исправить, добавив `height: auto` или `width: auto` к стилям изображения. Дополнительные сведения см. в [статье web.dev о теге `<img>`](https://web.dev/patterns/web-vitals-patterns/images/img-tag).

Если атрибуты `width` и `height` на изображении мешают задать нужный размер изображения через CSS, рассмотрите использование режима `fill` и стилизацию родительского элемента изображения.

## Функции производительности {#performance-features}

NgOptimizedImage включает ряд функций, разработанных для улучшения производительности загрузки в вашем приложении. Эти функции описаны в данном разделе.

### Добавление подсказок ресурсов {#add-resource-hints}

[Подсказка ресурса `preconnect`](https://web.dev/preconnect-and-dns-prefetch) для источника изображения гарантирует, что LCP-изображение загружается как можно быстрее.

Ссылки предварительного подключения генерируются автоматически для доменов, указанных в качестве аргумента [загрузчику](#optional-set-up-a-loader). Если источник изображения не может быть определён автоматически и для LCP-изображения не обнаружена ссылка предварительного подключения, `NgOptimizedImage` выдаст предупреждение в режиме разработки. В этом случае следует вручную добавить подсказку ресурса в `index.html`. В разделе `<head>` документа добавьте тег `link` с `rel="preconnect"`, как показано ниже:

```html
<link rel="preconnect" href="https://my.cdn.origin" />
```

Чтобы отключить предупреждения предварительного подключения, внедрите токен `PRECONNECT_CHECK_BLOCKLIST`:

```ts

providers: [
{provide: PRECONNECT_CHECK_BLOCKLIST, useValue: 'https://your-domain.com'}
],

```

Дополнительную информацию об автоматической генерации предварительных подключений см. [здесь](#why-is-a-preconnect-element-not-being-generated-for-my-image-domain).

### Запрос изображений нужного размера с автоматическим `srcset` {#request-images-at-the-correct-size-with-automatic-srcset}

Определение [атрибута `srcset`](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/srcset) гарантирует, что браузер запрашивает изображение нужного размера для области просмотра пользователя и не тратит время на загрузку слишком большого изображения. `NgOptimizedImage` генерирует подходящий `srcset` для изображения на основе наличия и значения [атрибута `sizes`](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/sizes) на теге изображения.

#### Изображения фиксированного размера {#fixed-size-images}

Если изображение должно быть «фиксированного» размера (т. е. одинакового размера на всех устройствах, кроме [плотности пикселей](https://web.dev/codelab-density-descriptors/)), нет необходимости задавать атрибут `sizes`. `srcset` может быть сгенерирован автоматически из атрибутов ширины и высоты изображения без дополнительных входных данных.

Пример сгенерированного srcset:

```html
<img ... srcset="image-400w.jpg 1x, image-800w.jpg 2x" />
```

#### Адаптивные изображения {#responsive-images}

Если изображение должно быть адаптивным (т. е. увеличиваться и уменьшаться в зависимости от размера области просмотра), вам нужно определить [атрибут `sizes`](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/sizes) для генерации `srcset`.

Если вы раньше не использовали `sizes`, хорошей отправной точкой является установка его на основе ширины области просмотра. Например, если CSS заставляет изображение занимать 100% ширины области просмотра, установите `sizes` в `100vw`, и браузер выберет изображение в `srcset`, ближайшее к ширине области просмотра (с учётом плотности пикселей). Если изображение, вероятно, займёт только половину экрана (например, в боковой панели), установите `sizes` в `50vw`, чтобы браузер выбрал меньшее изображение. И так далее.

Если вышеописанное не охватывает нужное поведение изображения, обратитесь к документации по [расширенным значениям sizes](#advanced-sizes-values).

Обратите внимание, что `NgOptimizedImage` автоматически добавляет `"auto"` к указанному значению `sizes`. Это оптимизация, повышающая точность выбора srcset в браузерах, поддерживающих `sizes="auto"`, и игнорируемая браузерами, которые его не поддерживают.

По умолчанию адаптивные точки останова:

`[16, 32, 48, 64, 96, 128, 256, 384, 640, 750, 828, 1080, 1200, 1920, 2048, 3840]`

Если вы хотите настроить эти точки останова, сделайте это с помощью провайдера `IMAGE_CONFIG`:

```ts
providers: [
  {
    provide: IMAGE_CONFIG,
    useValue: {
      breakpoints: [16, 48, 96, 128, 384, 640, 750, 828, 1080, 1200, 1920]
    }
  },
],
```

Если вы хотите вручную определить атрибут `srcset`, укажите его с помощью атрибута `ngSrcset`:

```html
<img ngSrc="hero.jpg" ngSrcset="100w, 200w, 300w" />
```

При наличии атрибута `ngSrcset` `NgOptimizedImage` генерирует и устанавливает `srcset` на основе включённых размеров. Не включайте имена файлов изображений в `ngSrcset` — директива определяет эту информацию из `ngSrc`. Директива поддерживает как дескрипторы ширины (например, `100w`), так и дескрипторы плотности (например, `1x`).

```html
<img ngSrc="hero.jpg" ngSrcset="100w, 200w, 300w" sizes="50vw" />
```

### Отключение автоматической генерации srcset {#disabling-automatic-srcset-generation}

Чтобы отключить генерацию srcset для одного изображения, добавьте атрибут `disableOptimizedSrcset` к изображению:

```html
<img ngSrc="about.jpg" disableOptimizedSrcset />
```

### Отключение ленивой загрузки изображений {#disabling-image-lazy-loading}

По умолчанию `NgOptimizedImage` устанавливает `loading=lazy` для всех изображений, не помеченных как `priority`. Вы можете отключить это поведение для изображений без приоритета, установив атрибут `loading`. Этот атрибут принимает значения: `eager`, `auto` и `lazy`. [Подробнее см. в документации стандартного атрибута `loading` изображения](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/loading#value).

```html
<img ngSrc="cat.jpg" width="400" height="200" loading="eager" />
```

### Управление декодированием изображений {#controlling-image-decoding}

По умолчанию `NgOptimizedImage` устанавливает `decoding="auto"` для всех изображений. Это позволяет браузеру выбирать оптимальное время для декодирования изображения после его загрузки. Когда изображение помечено как `priority`, Angular автоматически устанавливает `decoding="sync"`, чтобы изображение было декодировано и отображено как можно раньше, помогая улучшить производительность **Largest Contentful Paint (LCP)**.

Вы по-прежнему можете переопределить это поведение, явно задав атрибут `decoding`.
[Подробнее см. в документации стандартного атрибута `decoding` изображения](https://developer.mozilla.org/docs/Web/HTML/Element/img#decoding).

```html
<!-- Default: decoding is 'auto' -->
<img ngSrc="gallery/landscape.jpg" width="1200" height="800" />

<!-- Decode the image asynchronously to avoid blocking the main thread.-->
<img ngSrc="gallery/preview.jpg" width="600" height="400" decoding="async" />

<!-- Priority images automatically use decoding="sync" -->
<img ngSrc="awesome.jpg" width="500" height="625" priority />

<!-- Decode immediately (can block) when you need the pixels right away -->
<img ngSrc="hero.jpg" width="1600" height="900" decoding="sync" />
```

**Допустимые значения**

- `auto` (по умолчанию): позволяет браузеру выбрать оптимальную стратегию.
- `async`: декодирует изображение асинхронно, по возможности избегая блокировки основного потока.
- `sync`: декодирует изображение немедленно; может блокировать рендеринг, но гарантирует готовность пикселей сразу после доступности изображения.

### Расширенные значения 'sizes' {#advanced-sizes-values}

Возможно, вы захотите, чтобы изображения отображались с разной шириной на экранах разного размера. Распространённый пример такого паттерна — сетчатый или колоночный макет, отображающий один столбец на мобильных устройствах и два столбца на более крупных устройствах. Вы можете зафиксировать это поведение в атрибуте `sizes`, используя синтаксис «медиазапроса», например:

```html
<img ngSrc="cat.jpg" width="400" height="200" sizes="(max-width: 768px) 100vw, 50vw" />
```

Атрибут `sizes` в приведённом выше примере означает: «Я ожидаю, что это изображение займёт 100 процентов ширины экрана на устройствах шириной менее 768px. В противном случае я ожидаю, что оно займёт 50 процентов ширины экрана».

Дополнительные сведения об атрибуте `sizes` см. на [web.dev](https://web.dev/learn/design/responsive-images/#sizes) или [mdn](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/sizes).

## Настройка загрузчика изображений для `NgOptimizedImage` {#configuring-an-image-loader-for-ngoptimizedimage}

«Загрузчик» — это функция, генерирующая [URL для трансформации изображения](https://web.dev/image-cdns/#how-image-cdns-use-urls-to-indicate-optimization-options) для заданного файла изображения. При необходимости `NgOptimizedImage` устанавливает трансформации размера, формата и качества изображения.

`NgOptimizedImage` предоставляет как универсальный загрузчик, не применяющий трансформаций, так и загрузчики для различных сторонних сервисов изображений. Также поддерживается написание собственного загрузчика.

| Тип загрузчика                              | Поведение                                                                                                                                                                                                                      |
| :------------------------------------------ | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Универсальный загрузчик                     | URL, возвращаемый универсальным загрузчиком, всегда соответствует значению `src`. Иными словами, этот загрузчик не применяет трансформаций. Основной сценарий использования — сайты, которые используют Angular для обслуживания изображений. |
| Загрузчики для сторонних сервисов изображений | URL, возвращаемый загрузчиками для сторонних сервисов, следует соглашениям API, используемым конкретным сервисом.                                                                                                              |
| Пользовательские загрузчики                 | Поведение пользовательского загрузчика определяется его разработчиком. Следует использовать пользовательский загрузчик, если ваш сервис изображений не поддерживается встроенными загрузчиками `NgOptimizedImage`.             |

На основе сервисов изображений, обычно используемых с Angular-приложениями, `NgOptimizedImage` предоставляет загрузчики, предварительно настроенные для следующих сервисов:

| Сервис изображений        | Angular API               | Документация                                                                |
| :------------------------ | :------------------------ | :-------------------------------------------------------------------------- |
| Cloudflare Image Resizing | `provideCloudflareLoader` | [Документация](https://developers.cloudflare.com/images/image-resizing/)    |
| Cloudinary                | `provideCloudinaryLoader` | [Документация](https://cloudinary.com/documentation/resizing_and_cropping)  |
| ImageKit                  | `provideImageKitLoader`   | [Документация](https://docs.imagekit.io/)                                   |
| Imgix                     | `provideImgixLoader`      | [Документация](https://docs.imgix.com/)                                     |
| Netlify                   | `provideNetlifyLoader`    | [Документация](https://docs.netlify.com/image-cdn/overview/)                |

Для использования **универсального загрузчика** никаких дополнительных изменений кода не требуется. Это поведение по умолчанию.

### Встроенные загрузчики {#built-in-loaders}

Чтобы использовать существующий загрузчик для **стороннего сервиса изображений**, добавьте фабрику провайдера для выбранного сервиса в массив `providers`. В примере ниже используется загрузчик Imgix:

```ts
providers: [
  provideImgixLoader('https://my.base.url/'),
],
```

Базовый URL для ресурсов изображений должен передаваться в фабрику провайдера в качестве аргумента. Для большинства сайтов этот базовый URL должен соответствовать одному из следующих шаблонов:

- <https://yoursite.yourcdn.com>
- <https://subdomain.yoursite.com>
- <https://subdomain.yourcdn.com/yoursite>

Подробнее о структуре базового URL можно узнать в документации соответствующего CDN-провайдера.

### Пользовательские загрузчики {#custom-loaders}

Чтобы использовать **пользовательский загрузчик**, укажите функцию загрузчика в качестве значения для DI-токена `IMAGE_LOADER`. В примере ниже пользовательская функция загрузчика возвращает URL, начинающийся с `https://example.com`, который включает `src`, `width` и `height` в качестве URL-параметров.

```ts
providers: [
  {
    provide: IMAGE_LOADER,
    useValue: (config: ImageLoaderConfig) => {
      return `https://example.com/images?src=${config.src}&width=${config.width}&height=${config.height}`;
    },
  },
],
```

Функция загрузчика для директивы `NgOptimizedImage` принимает объект типа `ImageLoaderConfig` (из `@angular/common`) в качестве аргумента и возвращает абсолютный URL ресурса изображения. Объект `ImageLoaderConfig` содержит свойство `src` и необязательные свойства `width`, `height` и `loaderParams`.

NOTE: Хотя свойство `width` может присутствовать не всегда, пользовательский загрузчик должен использовать его для поддержки запроса изображений различной ширины, чтобы `ngSrcset` работал корректно.

### Свойство `loaderParams` {#the-loaderparams-property}

Директива `NgOptimizedImage` поддерживает дополнительный атрибут `loaderParams`, специально разработанный для поддержки пользовательских загрузчиков. Атрибут `loaderParams` принимает объект с любыми свойствами в качестве значения и сам по себе ничего не делает. Данные в `loaderParams` добавляются к объекту `ImageLoaderConfig`, передаваемому пользовательскому загрузчику, и могут использоваться для управления поведением загрузчика.

Распространённый сценарий использования `loaderParams` — управление расширенными функциями CDN изображений.

### Использование свойства `transform` со встроенными загрузчиками {#using-the-transform-property-with-built-in-loaders}

Встроенные загрузчики для Cloudinary, Cloudflare, ImageKit и Imgix поддерживают специальное свойство `transform` внутри `loaderParams`. Это свойство позволяет применять пользовательские трансформации изображений, предоставляемые вашим CDN.

Свойство `transform` принимает два формата:

#### Строковый формат {#string-format}

Укажите трансформации в виде строки с разделителями-запятыми, используя синтаксис трансформации вашего CDN:

```html
<img
  ngSrc="my-image.jpg"
  width="400"
  height="300"
  [loaderParams]="{transform: 'e_grayscale,r_10'}"
/>
```

#### Объектный формат {#object-format}

Укажите трансформации в виде объекта с парами ключ-значение.

```html
<img
  ngSrc="my-image.jpg"
  width="400"
  height="300"
  [loaderParams]="{transform: {e: 'grayscale', r: 10}}"
/>
```

NOTE: Свойство `transform` не поддерживается загрузчиком Netlify, поскольку CDN Netlify не предоставляет пользовательские параметры трансформации.

### Пример пользовательского загрузчика {#example-custom-loader}

Ниже показан пример функции пользовательского загрузчика. Эта функция конкатенирует `src`, `width` и `height`, и использует `loaderParams` для управления пользовательской функцией CDN для скруглённых углов:

```ts
const myCustomLoader = (config: ImageLoaderConfig) => {
  let url = `https://example.com/images/${config.src}?`;
  let queryParams = [];
  if (config.width) {
    queryParams.push(`w=${config.width}`);
  }
  if (config.height) {
    queryParams.push(`h=${config.height}`);
  }
  if (config.loaderParams?.roundedCorners) {
    queryParams.push('mask=corners&corner-radius=5');
  }
  return url + queryParams.join('&');
};
```

Обратите внимание, что в приведённом примере мы придумали имя свойства 'roundedCorners' для управления функцией нашего пользовательского загрузчика. Затем мы можем использовать эту функцию при создании изображения следующим образом:

```html
<img ngSrc="profile.jpg" width="300" height="300" [loaderParams]="{roundedCorners: true}" />
```

## Часто задаваемые вопросы {#frequently-asked-questions}

### Поддерживает ли NgOptimizedImage CSS-свойство `background-image`? {#does-ngoptimizedimage-support-the-background-image-css-property}

NgOptimizedImage не поддерживает CSS-свойство `background-image` напрямую, но разработана для удобной поддержки использования изображения в качестве фона другого элемента.

Пошаговый процесс перехода с `background-image` на `NgOptimizedImage` см. в разделе [Как перейти с фонового изображения](#how-to-migrate-your-background-image) выше.

### Почему нельзя использовать `src` с `NgOptimizedImage`? {#why-cant-i-use-src-with-ngoptimizedimage}

Атрибут `ngSrc` был выбран в качестве триггера для NgOptimizedImage по техническим соображениям, связанным со способом загрузки изображений браузером. NgOptimizedImage программно изменяет атрибут `loading` — если браузер увидит атрибут `src` до этих изменений, он начнёт жадно загружать файл изображения, и изменения загрузки будут проигнорированы.

### Почему для домена моего изображения не генерируется элемент preconnect? {#why-is-a-preconnect-element-not-being-generated-for-my-image-domain}

Генерация preconnect выполняется на основе статического анализа вашего приложения. Это означает, что домен изображения должен быть непосредственно включён в параметр загрузчика, как в следующем примере:

```ts
providers: [
  provideImgixLoader('https://my.base.url/'),
],
```

Если вы используете переменную для передачи строки домена загрузчику или не используете загрузчик, статический анализ не сможет определить домен, и ссылка preconnect не будет сгенерирована. В этом случае следует вручную добавить ссылку preconnect в заголовок документа, как [описано выше](#add-resource-hints).

### Можно ли использовать два разных домена изображений на одной странице? {#can-i-use-two-different-image-domains-in-the-same-page}

Паттерн провайдера [загрузчиков изображений](#configuring-an-image-loader-for-ngoptimizedimage) разработан максимально простым для распространённого случая использования одного CDN изображений в компоненте. Однако вполне возможно управлять несколькими CDN изображений с помощью одного провайдера.

Для этого рекомендуется написать [пользовательский загрузчик изображений](#custom-loaders), использующий [свойство `loaderParams`](#the-loaderparams-property) для передачи флага, указывающего, какой CDN изображений следует использовать, а затем вызывающий соответствующий загрузчик на основе этого флага.

### Можно ли добавить новый встроенный загрузчик для предпочитаемого CDN? {#can-you-add-a-new-built-in-loader-for-my-preferred-cdn}

По соображениям обслуживания мы в настоящее время не планируем поддерживать дополнительные встроенные загрузчики в репозитории Angular. Вместо этого мы поощряем разработчиков публиковать дополнительные загрузчики изображений как сторонние пакеты.

### Можно ли использовать это с тегом `<picture>`? {#can-i-use-this-with-the-picture-tag}

Нет, но это есть в нашем плане разработки, так что следите за новостями.

Если вы ждёте этой функции, пожалуйста, проголосуйте за Github issue [здесь](https://github.com/angular/angular/issues/56594).

### Как найти LCP-изображение с помощью Chrome DevTools? {#how-do-i-find-my-lcp-image-with-chrome-devtools}

1. На вкладке производительности Chrome DevTools нажмите кнопку «start profiling and reload page» в верхнем левом углу. Она выглядит как значок обновления страницы.

2. Это запустит снимок профилирования вашего Angular-приложения.

3. Как только результат профилирования будет доступен, выберите «LCP» в разделе тайминга.

4. В нижней панели должна появиться сводная запись. Вы можете найти LCP-элемент в строке «related node». Нажав на него, вы откроете элемент в панели Elements.

<img alt="LCP in the Chrome DevTools" src="assets/images/guide/image-optimization/devtools-lcp.png">

NOTE: Это определяет LCP-элемент только в области просмотра тестируемой страницы. Также рекомендуется использовать эмуляцию мобильных устройств для определения LCP-элемента на небольших экранах.
