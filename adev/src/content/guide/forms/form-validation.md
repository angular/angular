# Валидация ввода в формах {#validating-form-input}

Вы можете повысить качество данных, проверяя ввод пользователя на точность и полноту.
Эта страница показывает, как валидировать ввод пользователя из UI и отображать полезные сообщения об ошибках в реактивных и шаблонных формах.

## Валидация ввода в шаблонных формах {#validating-input-in-template-driven-forms}

Чтобы добавить валидацию в шаблонную форму, используйте те же атрибуты валидации, что и при [нативной HTML валидации форм](https://developer.mozilla.org/docs/Web/Guide/HTML/HTML5/Constraint_validation).
Angular использует директивы для сопоставления этих атрибутов с функциями-валидаторами в фреймворке.

Каждый раз, когда значение элемента управления формой меняется, Angular запускает валидацию и генерирует либо список ошибок валидации, приводящих к статусу `INVALID`, либо null, что соответствует статусу VALID.

Затем вы можете проверить состояние элемента управления, экспортировав `ngModel` в локальную переменную шаблона.
В следующем примере `NgModel` экспортируется в переменную `name`:

<docs-code header="actor-form-template.component.html (name)" path="adev/src/content/examples/form-validation/src/app/template/actor-form-template.component.html" region="name-with-error-msg"/>

Обратите внимание на следующие особенности, показанные в примере.

- Элемент `<input>` имеет атрибуты HTML валидации: `required` и `minlength`.
  Также он содержит пользовательскую директиву валидатора, `forbiddenName`.
  Подробнее см. раздел [Пользовательские валидаторы](#defining-custom-validators).

- `#name="ngModel"` экспортирует `NgModel` в локальную переменную `name`.
  `NgModel` отражает многие свойства своего базового экземпляра `FormControl`, поэтому вы можете использовать это в шаблоне для проверки состояния элемента управления, такого как `valid` и `dirty`.
  Полный список свойств элементов управления см. в справочнике API [AbstractControl](api/forms/AbstractControl).
  - Внешний `@if` показывает набор вложенных сообщений только если `name` недействителен и элемент управления является либо `dirty`, либо `touched`.

  - Каждый вложенный `@if` может отображать пользовательское сообщение для одной из возможных ошибок валидации.
    Есть сообщения для `required`, `minlength` и `forbiddenName`.

HELPFUL: Чтобы предотвратить отображение ошибок валидатором до того, как пользователь имел возможность отредактировать форму, следует проверять состояние `dirty` или `touched` элемента управления.

- Когда пользователь изменяет значение в отслеживаемом поле, элемент управления помечается как "dirty"
- Когда пользователь переключает фокус с элемента управления формы, он помечается как "touched"

## Валидация ввода в реактивных формах {#validating-input-in-reactive-forms}

В реактивной форме источником истины является класс компонента.
Вместо добавления валидаторов через атрибуты в шаблоне, вы добавляете функции-валидаторы непосредственно в модель элемента управления формой в классе компонента.
Angular затем вызывает эти функции каждый раз, когда значение элемента управления меняется.

### Функции-валидаторы {#validator-functions}

Функции-валидаторы могут быть синхронными или асинхронными.

| Тип валидатора     | Подробности                                                                                                                                                                                                                               |
| :----------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Sync валидаторы    | Синхронные функции, которые принимают экземпляр элемента управления и немедленно возвращают набор ошибок валидации или `null`. Передаются вторым аргументом при создании `FormControl`.                                                   |
| Async валидаторы   | Асинхронные функции, которые принимают экземпляр элемента управления и возвращают Promise или Observable, которые позже генерируют набор ошибок валидации или `null`. Передаются третьим аргументом при создании `FormControl`. |

Из соображений производительности Angular запускает асинхронные валидаторы только если все синхронные валидаторы проходят.
Каждый должен завершиться до установки ошибок.

### Встроенные функции-валидаторы {#built-in-validator-functions}

Вы можете [написать собственные функции-валидаторы](#defining-custom-validators) или использовать встроенные валидаторы Angular.

Те же встроенные валидаторы, которые доступны как атрибуты в шаблонных формах, например `required` и `minlength`, доступны для использования как функции из класса `Validators`.
Полный список встроенных валидаторов см. в справочнике API [Validators](api/forms/Validators).

Чтобы обновить форму актёра до реактивной, используйте некоторые из тех же
встроенных валидаторов — на этот раз в виде функций, как в следующем примере.

## Определение пользовательских валидаторов {#defining-custom-validators}

Встроенные валидаторы не всегда соответствуют точному сценарию использования вашего приложения, поэтому иногда требуется создать пользовательский валидатор.

Рассмотрим функцию `forbiddenNameValidator` из предыдущего примера.
Вот как выглядит определение этой функции.

<docs-code header="forbidden-name.directive.ts (forbiddenNameValidator)" path="adev/src/content/examples/form-validation/src/app/shared/forbidden-name.directive.ts" region="custom-validator"/>

Функция является фабрикой, которая принимает регулярное выражение для обнаружения _конкретного_ запрещённого имени и возвращает функцию-валидатор.

В этом примере запрещённое имя — "bob", поэтому валидатор отклоняет любое имя актёра, содержащее "bob".
В другом месте можно отклонить "alice" или любое имя, которое соответствует настроенному регулярному выражению.

Фабрика `forbiddenNameValidator` возвращает настроенную функцию-валидатор.
Эта функция принимает объект элемента управления Angular и возвращает _либо_ null, если значение элемента управления действительно, _либо_ объект ошибки валидации.
Объект ошибки валидации обычно имеет свойство с именем, соответствующим ключу валидации, `'forbiddenName'`, и значением — произвольный словарь значений, которые можно вставить в сообщение об ошибке, `{name}`.

Пользовательские асинхронные валидаторы аналогичны синхронным, но вместо этого должны возвращать Promise или Observable, которые позже генерируют null или объект ошибки валидации.
В случае Observable она должна завершиться, после чего форма использует последнее сгенерированное значение для валидации.

## Кросс-поля валидация {#cross-field-validation}

Кросс-поля валидатор — это [пользовательский валидатор](#defining-custom-validators 'Читать о пользовательских валидаторах'), который сравнивает значения разных полей в форме и принимает или отклоняет их в совокупности.
Например, у вас может быть форма, предлагающая взаимоисключающие варианты, так что пользователь может выбрать A или B, но не оба.
Некоторые значения полей могут также зависеть от других: пользователю может быть разрешено выбрать B только если A тоже выбрано.

Следующие примеры кросс-валидации показывают, как:

- Валидировать ввод реактивных или шаблонных форм на основе значений двух родственных элементов управления,
- Показывать описательное сообщение об ошибке после того, как пользователь взаимодействовал с формой и валидация не прошла.

В примерах используется кросс-валидация для проверки того, что актёры не используют одно и то же имя в своей роли при заполнении формы актёра.
Валидаторы делают это, проверяя, что имена и роли актёров не совпадают.

### Добавление кросс-валидации в реактивные формы {#adding-cross-validation-to-reactive-forms}

Форма имеет следующую структуру:

```ts
const actorForm = new FormGroup({
  'name': new FormControl(),
  'role': new FormControl(),
  'skill': new FormControl(),
});
```

Обратите внимание, что `name` и `role` — дочерние элементы управления.
Чтобы оценить оба элемента управления в одном пользовательском валидаторе, необходимо выполнить валидацию в общем родительском элементе управления: `FormGroup`.
Вы запрашиваете у `FormGroup` его дочерние элементы управления, чтобы сравнить их значения.

Чтобы добавить валидатор к `FormGroup`, передайте новый валидатор вторым аргументом при создании.

```ts
const actorForm = new FormGroup(
  {
    'name': new FormControl(),
    'role': new FormControl(),
    'skill': new FormControl(),
  },
  {validators: unambiguousRoleValidator},
);
```

Код валидатора выглядит следующим образом.

<docs-code header="unambiguous-role.directive.ts" path="adev/src/content/examples/form-validation/src/app/shared/unambiguous-role.directive.ts" region="cross-validation-validator"/>

Валидатор `unambiguousRoleValidator` реализует интерфейс `ValidatorFn`.
Он принимает объект элемента управления Angular в качестве аргумента и возвращает null, если форма действительна, или `ValidationErrors` в противном случае.

Валидатор получает дочерние элементы управления, вызывая метод [get](api/forms/AbstractControl#get) у `FormGroup`, а затем сравнивает значения элементов управления `name` и `role`.

Если значения не совпадают, роль однозначна, оба действительны, и валидатор возвращает null.
Если они совпадают, роль актёра неоднозначна, и валидатор должен пометить форму как недействительную, возвращая объект ошибки.

## Создание асинхронных валидаторов {#creating-asynchronous-validators}

Асинхронные валидаторы реализуют интерфейсы `AsyncValidatorFn` и `AsyncValidator`.
Они очень похожи на свои синхронные аналоги, со следующими отличиями.

- Функции `validate()` должны возвращать Promise или Observable,
- Возвращаемый Observable должен быть конечным, то есть в какой-то момент он должен завершиться.
  Чтобы преобразовать бесконечный Observable в конечный, пропустите Observable через оператор фильтрации, такой как `first`, `last`, `take` или `takeUntil`.

Асинхронная валидация происходит после синхронной и выполняется только если синхронная валидация прошла успешно.
Эта проверка позволяет формам избегать потенциально дорогостоящих асинхронных процессов валидации \(например, HTTP-запроса\), если базовые методы валидации уже обнаружили недопустимый ввод.

После начала асинхронной валидации элемент управления формой входит в состояние `pending`.
Проверяйте свойство `pending` элемента управления и используйте его для визуальной обратной связи о текущей операции валидации.

Распространённый UI-паттерн — показывать спиннер во время выполнения асинхронной валидации.
В следующем примере показано, как этого добиться в шаблонной форме.

```angular-html
<input [(ngModel)]="name" #model="ngModel" appSomeAsyncValidator />

@if (model.pending) {
  <app-spinner />
}
```

### Реализация пользовательского асинхронного валидатора {#implementing-a-custom-async-validator}

В следующем примере асинхронный валидатор гарантирует, что актёры назначены на роль, которая ещё не занята.
Новые актёры постоянно проходят прослушивание, а старые уходят на пенсию, поэтому список доступных ролей нельзя получить заранее.
Для проверки потенциального назначения на роль валидатор должен инициировать асинхронную операцию для обращения к центральной базе данных всех актёров, занятых в настоящее время.

Следующий код создаёт класс валидатора `UniqueRoleValidator`, реализующий интерфейс `AsyncValidator`.

<docs-code header="role.directive.ts" path="adev/src/content/examples/form-validation/src/app/shared/role.directive.ts" region="async-validator"/>

Свойство `actorsService` инициализируется экземпляром токена `ActorsService`, который определяет следующий интерфейс.

```ts
interface ActorsService {
  isRoleTaken: (role: string) => Observable<boolean>;
}
```

В реальном приложении `ActorsService` отвечал бы за выполнение HTTP-запроса к базе данных актёров для проверки доступности роли.
С точки зрения валидатора фактическая реализация сервиса не важна, поэтому пример может просто кодировать против интерфейса `ActorsService`.

Когда валидация начинается, `UniqueRoleValidator` делегирует метод `isRoleTaken()` из `ActorsService` с текущим значением элемента управления.
В этот момент элемент управления помечается как `pending` и остаётся в этом состоянии до завершения цепочки Observable, возвращённой методом `validate()`.

Метод `isRoleTaken()` отправляет HTTP-запрос, который проверяет доступность роли, и возвращает `Observable<boolean>` в качестве результата.
Метод `validate()` пропускает ответ через оператор `map` и преобразует его в результат валидации.

Метод затем, как любой валидатор, возвращает `null`, если форма действительна, и `ValidationErrors`, если нет.
Этот валидатор обрабатывает потенциальные ошибки с помощью оператора `catchError`.
В данном случае валидатор расценивает ошибку `isRoleTaken()` как успешную валидацию, поскольку неудача при выполнении запроса валидации не обязательно означает, что роль недействительна.
Вы можете обработать ошибку иначе и вместо этого вернуть объект `ValidationError`.

Через некоторое время цепочка Observable завершается, и асинхронная валидация выполнена.
Флаг `pending` устанавливается в `false`, и обновляется действительность формы.

### Оптимизация производительности асинхронных валидаторов {#optimizing-performance-of-async-validators}

По умолчанию все валидаторы запускаются после каждого изменения значения формы.
С синхронными валидаторами это обычно не оказывает заметного влияния на производительность приложения.
Однако асинхронные валидаторы, как правило, выполняют какой-либо HTTP-запрос для валидации элемента управления.
Отправка HTTP-запроса после каждого нажатия клавиши может создать нагрузку на API бэкенда и по возможности должна быть избегаема.

Вы можете задержать обновление действительности формы, изменив свойство `updateOn` с `change` (по умолчанию) на `submit` или `blur`.

Для шаблонных форм задайте свойство в шаблоне.

```angular-html
<input [(ngModel)]="name" [ngModelOptions]="{updateOn: 'blur'}" />
```

Для реактивных форм задайте свойство в экземпляре `FormControl`.

```ts
new FormControl('', {updateOn: 'blur'});
```

## Взаимодействие с нативной HTML валидацией форм {#interaction-with-native-html-form-validation}

По умолчанию Angular отключает [нативную HTML валидацию форм](https://developer.mozilla.org/docs/Web/Guide/HTML/Constraint_validation), добавляя атрибут `novalidate` на охватывающий элемент `<form>`, и использует директивы для сопоставления этих атрибутов с функциями-валидаторами в фреймворке.
Если вы хотите использовать нативную валидацию **совместно** с валидацией Angular, вы можете повторно включить её с помощью директивы `ngNativeValidate`.
Подробнее см. в [документации API](api/forms/NgForm#native-dom-validation-ui).
