# Валидация ввода формы {#validating-form-input}

Можно улучшить общее качество данных, проверяя пользовательский ввод на точность и полноту.
На этой странице показано, как валидировать пользовательский ввод из UI и отображать полезные сообщения о валидации — в реактивных формах и формах на основе шаблонов.

## Валидация ввода в формах на основе шаблонов {#validating-input-in-template-driven-forms}

Для добавления валидации в форму на основе шаблонов используются те же атрибуты валидации, что и в [нативной валидации HTML-форм](https://developer.mozilla.org/docs/Web/Guide/HTML/HTML5/Constraint_validation).
Angular использует директивы для сопоставления этих атрибутов с функциями-валидаторами во фреймворке.

При каждом изменении значения элемента управления формы Angular запускает валидацию и генерирует либо список ошибок валидации, что приводит к статусу `INVALID`, либо null — к статусу VALID.

Затем можно проверить состояние элемента управления, экспортировав `ngModel` в локальную переменную шаблона.
Следующий пример экспортирует `NgModel` в переменную с именем `name`:

<docs-code header="actor-form-template.component.html (name)" path="adev/src/content/examples/form-validation/src/app/template/actor-form-template.component.html" region="name-with-error-msg"/>

Обратите внимание на следующие особенности, проиллюстрированные в примере.

- Элемент `<input>` содержит атрибуты HTML-валидации: `required` и `minlength`.
  Также присутствует пользовательская директива-валидатор `forbiddenName`.
  Подробнее см. в разделе [Пользовательские валидаторы](#defining-custom-validators).

- `#name="ngModel"` экспортирует `NgModel` в локальную переменную с именем `name`.
  `NgModel` отражает многие свойства своего базового экземпляра `FormControl`, поэтому его можно использовать в шаблоне для проверки состояний элемента управления, таких как `valid` и `dirty`.
  Полный список свойств элемента управления см. в справочнике API [AbstractControl](api/forms/AbstractControl).
  - Внешний `@if` отображает набор вложенных сообщений, но только если `name` недействителен и элемент управления является `dirty` или `touched`.

  - Каждый вложенный `@if` может выводить пользовательское сообщение для одной из возможных ошибок валидации.
    Есть сообщения для `required`, `minlength` и `forbiddenName`.

HELPFUL: Чтобы валидатор не отображал ошибки до того, как пользователь успел отредактировать форму, следует проверять состояния `dirty` или `touched` в элементе управления.

- Когда пользователь изменяет значение в отслеживаемом поле, элемент управления помечается как «dirty» (грязный)
- Когда пользователь теряет фокус на элементе управления формы, он помечается как «touched» (затронутый)

## Валидация ввода в реактивных формах {#validating-input-in-reactive-forms}

В реактивной форме источником истины является класс компонента.
Вместо добавления валидаторов через атрибуты в шаблоне функции-валидаторы добавляются непосредственно в модель элемента управления формы в классе компонента.
Angular затем вызывает эти функции при каждом изменении значения элемента управления.

### Функции-валидаторы {#validator-functions}

Функции-валидаторы могут быть синхронными или асинхронными.

| Тип валидатора   | Подробности                                                                                                                                                                                                                                                              |
| :--------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Синхронные       | Синхронные функции, принимающие экземпляр элемента управления и немедленно возвращающие набор ошибок валидации или `null`. Передаются как второй аргумент при создании `FormControl`.                                                                                  |
| Асинхронные      | Асинхронные функции, принимающие экземпляр элемента управления и возвращающие Promise или Observable, которые впоследствии генерируют набор ошибок валидации или `null`. Передаются как третий аргумент при создании `FormControl`.                                   |

По соображениям производительности Angular запускает асинхронные валидаторы только при успешном прохождении всех синхронных валидаторов.
Каждый должен завершиться до установки ошибок.

### Встроенные функции-валидаторы {#built-in-validator-functions}

Можно [написать собственные функции-валидаторы](#defining-custom-validators) или использовать встроенные валидаторы Angular.

Те же встроенные валидаторы, что доступны в виде атрибутов в формах на основе шаблонов, например `required` и `minlength`, доступны для использования как функции из класса `Validators`.
Полный список встроенных валидаторов см. в справочнике API [Validators](api/forms/Validators).

Для обновления формы актёра до реактивной используйте некоторые из тех же встроенных валидаторов — на этот раз в виде функций, как в следующем примере.

## Определение пользовательских валидаторов {#defining-custom-validators}

Встроенные валидаторы не всегда соответствуют конкретному случаю использования приложения, поэтому иногда необходимо создавать пользовательские валидаторы.

Рассмотрим функцию `forbiddenNameValidator` из предыдущего примера.
Вот как выглядит её определение.

<docs-code header="forbidden-name.directive.ts (forbiddenNameValidator)" path="adev/src/content/examples/form-validation/src/app/shared/forbidden-name.directive.ts" region="custom-validator"/>

Функция является фабрикой, принимающей регулярное выражение для обнаружения _конкретного_ запрещённого имени и возвращающей функцию-валидатор.

В этом примере запрещённое имя — «bob», поэтому валидатор отклоняет любое имя актёра, содержащее «bob».
В другом месте это могло бы быть «alice» или любое имя, совпадающее с настраиваемым регулярным выражением.

Фабрика `forbiddenNameValidator` возвращает настроенную функцию-валидатор.
Эта функция принимает объект элемента управления Angular и возвращает _либо_ null, если значение элемента управления действительно, _либо_ объект ошибки валидации.
Объект ошибки валидации обычно имеет свойство, имя которого является ключом валидации `'forbiddenName'`, а значение — произвольным словарём значений, которые можно вставить в сообщение об ошибке `{name}`.

Пользовательские асинхронные валидаторы похожи на синхронные, но вместо этого они должны возвращать Promise или Observable, которые впоследствии генерируют null или объект ошибки валидации.
В случае Observable он должен завершиться, и тогда форма использует последнее сгенерированное значение для валидации.

## Перекрёстная валидация {#cross-field-validation}

Перекрёстный валидатор — это [пользовательский валидатор](#defining-custom-validators 'Read about custom validators'), который сравнивает значения разных полей формы и принимает или отклоняет их в комбинации.
Например, форма может предлагать взаимоисключающие варианты, при которых пользователь может выбрать A или B, но не оба.
Некоторые значения полей могут также зависеть от других; пользователю может быть разрешено выбрать B только если также выбран A.

Следующие примеры перекрёстной валидации показывают, как:

- Валидировать ввод реактивной или основанной на шаблонах формы на основе значений двух родственных элементов управления,
- Показывать описательное сообщение об ошибке после взаимодействия пользователя с формой и неудачной валидации.

В примерах используется перекрёстная валидация, чтобы гарантировать, что актёры не используют одно и то же имя в своей роли при заполнении формы актёра.
Валидаторы проверяют это, убеждаясь, что имена актёров и роли не совпадают.

### Добавление перекрёстной валидации в реактивные формы {#adding-cross-validation-to-reactive-forms}

Форма имеет следующую структуру:

```ts
const actorForm = new FormGroup({
  'name': new FormControl(),
  'role': new FormControl(),
  'skill': new FormControl(),
});
```

Обратите внимание, что `name` и `role` — родственные элементы управления.
Для оценки обоих элементов управления в одном пользовательском валидаторе необходимо выполнить валидацию в общем предке: `FormGroup`.
`FormGroup` запрашивается для его дочерних элементов управления, чтобы можно было сравнить их значения.

Для добавления валидатора в `FormGroup` передайте новый валидатор как второй аргумент при создании.

```ts
const actorForm = new FormGroup(
  {
    'name': new FormControl(),
    'role': new FormControl(),
    'skill': new FormControl(),
  },
  {validators: unambiguousRoleValidator},
);
```

Код валидатора выглядит следующим образом.

<docs-code header="unambiguous-role.directive.ts" path="adev/src/content/examples/form-validation/src/app/shared/unambiguous-role.directive.ts" region="cross-validation-validator"/>

Валидатор `unambiguousRoleValidator` реализует интерфейс `ValidatorFn`.
Он принимает объект элемента управления Angular в качестве аргумента и возвращает null, если форма действительна, или `ValidationErrors` в противном случае.

Валидатор получает дочерние элементы управления, вызывая метод [get](api/forms/AbstractControl#get) `FormGroup`, затем сравнивает значения элементов управления `name` и `role`.

Если значения не совпадают, роль однозначна, оба действительны, и валидатор возвращает null.
Если они совпадают, роль актёра неоднозначна, и валидатор должен пометить форму как недействительную, возвращая объект ошибки.

## Создание асинхронных валидаторов {#creating-asynchronous-validators}

Асинхронные валидаторы реализуют интерфейсы `AsyncValidatorFn` и `AsyncValidator`.
Они очень похожи на синхронные аналоги со следующими отличиями.

- Функции `validate()` должны возвращать Promise или Observable,
- Возвращаемый Observable должен быть конечным, то есть должен завершиться в какой-то момент.
  Для преобразования бесконечного Observable в конечный используйте фильтрующий оператор, например `first`, `last`, `take` или `takeUntil`.

Асинхронная валидация происходит после синхронной и выполняется только при успешной синхронной валидации.
Эта проверка позволяет формам избежать потенциально дорогостоящих асинхронных процессов валидации \(например, HTTP-запроса\), если более базовые методы валидации уже нашли недействительный ввод.

После начала асинхронной валидации элемент управления формы переходит в состояние `pending`.
Проверяйте свойство `pending` элемента управления и используйте его для визуальной обратной связи о текущей операции валидации.

Типичный паттерн UI — показывать спиннер во время выполнения асинхронной валидации.
Следующий пример показывает, как достичь этого в форме на основе шаблонов.

```angular-html
<input [(ngModel)]="name" #model="ngModel" appSomeAsyncValidator />

@if (model.pending) {
  <app-spinner />
}
```

### Реализация пользовательского асинхронного валидатора {#implementing-a-custom-async-validator}

В следующем примере асинхронный валидатор гарантирует, что актёры претендуют на роль, которая ещё не занята.
Новые актёры постоянно проходят прослушивания, а старые уходят на пенсию, поэтому список доступных ролей нельзя получить заранее.
Для валидации потенциальной записи роли валидатор должен инициировать асинхронную операцию для обращения к центральной базе данных всех текущих актёров.

Следующий код создаёт класс валидатора `UniqueRoleValidator`, реализующий интерфейс `AsyncValidator`.

<docs-code header="role.directive.ts" path="adev/src/content/examples/form-validation/src/app/shared/role.directive.ts" region="async-validator"/>

Свойство `actorsService` инициализируется экземпляром токена `ActorsService`, который определяет следующий интерфейс.

```ts
interface ActorsService {
  isRoleTaken: (role: string) => Observable<boolean>;
}
```

В реальном приложении `ActorsService` отвечал бы за выполнение HTTP-запроса к базе данных актёров для проверки доступности роли.
С точки зрения валидатора конкретная реализация сервиса не важна, поэтому в примере можно писать код против интерфейса `ActorsService`.

По мере начала валидации `UniqueRoleValidator` делегирует метод `isRoleTaken()` сервиса `ActorsService` с текущим значением элемента управления.
В этот момент элемент управления помечается как `pending` и остаётся в этом состоянии до завершения цепочки Observable, возвращаемой методом `validate()`.

Метод `isRoleTaken()` выполняет HTTP-запрос, проверяя доступность роли, и возвращает `Observable<boolean>` в качестве результата.
Метод `validate()` преобразует ответ через оператор `map` и трансформирует его в результат валидации.

Затем метод, как любой валидатор, возвращает `null` при действительности формы и `ValidationErrors` в противном случае.
Этот валидатор обрабатывает любые потенциальные ошибки с помощью оператора `catchError`.
В данном случае валидатор рассматривает ошибку `isRoleTaken()` как успешную валидацию, поскольку неудача запроса не обязательно означает, что роль недействительна.
Можно обработать ошибку иначе и вернуть объект `ValidationError` вместо этого.

Через некоторое время цепочка Observable завершается, и асинхронная валидация завершена.
Флаг `pending` устанавливается в `false`, и валидность формы обновляется.

### Оптимизация производительности асинхронных валидаторов {#optimizing-performance-of-async-validators}

По умолчанию все валидаторы запускаются при каждом изменении значения формы.
Для синхронных валидаторов это обычно не оказывает заметного влияния на производительность приложения.
Асинхронные валидаторы, однако, обычно выполняют HTTP-запросы для валидации элемента управления.
Отправка HTTP-запроса при каждом нажатии клавиши может создать нагрузку на бэкенд API и по возможности следует избегать этого.

Можно задержать обновление валидности формы, изменив свойство `updateOn` с `change` (по умолчанию) на `submit` или `blur`.

В формах на основе шаблонов задайте свойство в шаблоне.

```angular-html
<input [(ngModel)]="name" [ngModelOptions]="{updateOn: 'blur'}" />
```

В реактивных формах задайте свойство в экземпляре `FormControl`.

```ts
new FormControl('', {updateOn: 'blur'});
```

## Взаимодействие с нативной валидацией HTML-форм {#interaction-with-native-html-form-validation}

По умолчанию Angular отключает [нативную валидацию HTML-форм](https://developer.mozilla.org/docs/Web/Guide/HTML/Constraint_validation), добавляя атрибут `novalidate` на вмещающий элемент `<form>`, и использует директивы для сопоставления этих атрибутов с функциями-валидаторами во фреймворке.
Если вы хотите использовать нативную валидацию **в сочетании** с валидацией на основе Angular, можно повторно включить её с помощью директивы `ngNativeValidate`.
Подробнее см. в [документации API](api/forms/NgForm#native-dom-validation-ui).
