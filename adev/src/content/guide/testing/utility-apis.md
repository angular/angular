# API утилит тестирования

ПРИМЕЧАНИЕ: Пока это руководство обновляется для Vitest, некоторые описания и примеры API утилит представлены в контексте Karma/Jasmine. Мы активно работаем над предоставлением эквивалентов для Vitest и обновлёнными рекомендациями там, где это применимо.

На этой странице описаны наиболее полезные возможности Angular для тестирования.

Утилиты тестирования Angular включают `TestBed`, `ComponentFixture` и набор функций для управления тестовой средой.
Классы [`TestBed`](#testbed-class-summary) и [`ComponentFixture`](#the-componentfixture) рассматриваются отдельно.

Вот краткое описание автономных функций в порядке вероятной полезности:

| Функция      | Подробности                                                                                                                                                                                                                                                       |
| :----------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| [`inject`]   | Внедряет один или несколько сервисов из текущего инжектора `TestBed` в тестовую функцию. Не может внедрять сервис, предоставляемый самим компонентом. См. обсуждение [debugElement.injector](guide/testing/components-scenarios#get-injected-services). |
| `getTestBed` | Получает текущий экземпляр `TestBed`. Обычно это не нужно, поскольку статических методов класса `TestBed` достаточно. Экземпляр `TestBed` предоставляет несколько редко используемых членов, недоступных в виде статических методов.                             |

Для работы со сложными асинхронными сценариями или тестирования устаревших приложений на основе Zone.js см. руководство [Утилиты тестирования Zone.js](guide/testing/zone-js-testing-utilities).

## Краткое описание класса `TestBed` {#testbed-class-summary}

Класс `TestBed` является одной из основных утилит тестирования Angular.
Его API достаточно обширен и может показаться сложным, пока вы не изучите его постепенно, шаг за шагом.
Сначала прочитайте начало этого руководства, чтобы получить базовые знания, прежде чем пытаться освоить весь API.

Определение модуля, передаваемое в `configureTestingModule`, является подмножеством свойств метаданных `@NgModule`.

```ts
type TestModuleMetadata = {
  providers?: any[];
  declarations?: any[];
  imports?: any[];
  schemas?: Array<SchemaMetadata | any[]>;
};
```

Каждый метод переопределения принимает `MetadataOverride<T>`, где `T` — тип метаданных, соответствующий методу, то есть параметр `@NgModule`, `@Component`, `@Directive` или `@Pipe`.

```ts
type MetadataOverride<T> = {
  add?: Partial<T>;
  remove?: Partial<T>;
  set?: Partial<T>;
};
```

API `TestBed` состоит из статических методов класса, которые либо обновляют, либо ссылаются на _глобальный_ экземпляр `TestBed`.

Внутри все статические методы оборачивают методы текущего экземпляра `TestBed` времени выполнения, который также возвращается функцией `getTestBed()`.

Вызывайте методы `TestBed` _внутри_ `beforeEach()`, чтобы обеспечить чистое состояние перед каждым отдельным тестом.

Вот наиболее важные статические методы в порядке вероятной полезности.

| Методы                   | Подробности                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| :----------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `configureTestingModule` | Заглушки тестирования устанавливают [начальную тестовую среду](guide/testing) и модуль тестирования по умолчанию. Модуль тестирования по умолчанию настроен с базовыми объявлениями и некоторыми заменами сервисов Angular, необходимыми каждому тестировщику. <br /> Вызовите `configureTestingModule`, чтобы уточнить конфигурацию модуля тестирования для конкретного набора тестов, добавляя и удаляя импорты, объявления \(компонентов, директив и пайпов\) и провайдеры. |
| `compileComponents`      | Асинхронно компилирует модуль тестирования после завершения его настройки. **Необходимо** вызывать этот метод, если _любой_ из компонентов тестового модуля имеет асинхронно загружаемые ресурсы (например, блоки @defer). <br /> После вызова `compileComponents` конфигурация `TestBed` замораживается на время текущей спецификации.                                                                                                                                          |
| `createComponent<T>`     | Создаёт экземпляр компонента типа `T` на основе текущей конфигурации `TestBed`. После вызова `createComponent` конфигурация `TestBed` замораживается на время текущей спецификации.                                                                                                                                                                                                                                                                                              |
| `overrideComponent`      | Заменяет метаданные для указанного класса компонента, который может находиться глубоко во вложенном внутреннем модуле.                                                                                                                                                                                                                                                                                                                                                            |
| `overrideDirective`      | Заменяет метаданные для указанного класса директивы, который может находиться глубоко во вложенном внутреннем модуле.                                                                                                                                                                                                                                                                                                                                                             |
| `overridePipe`           | Заменяет метаданные для указанного класса пайпа, который может находиться глубоко во вложенном внутреннем модуле.                                                                                                                                                                                                                                                                                                                                                                 |
| `overrideModule`         | Заменяет метаданные для указанного `NgModule`. Модули могут импортировать другие модули. Метод `overrideModule` может проникнуть глубоко в текущий тестовый модуль для изменения одного из таких внутренних модулей.                                                                                                                                                                                                                                                              |

|
`inject` | Получает сервис из текущего инжектора `TestBed`. Функция `inject` часто достаточна для этой цели. Но `inject` выбрасывает ошибку, если не может предоставить сервис. <br /> Что если сервис необязателен? <br /> Метод `TestBed.inject()` принимает необязательный второй параметр — объект для возврата, если Angular не может найти провайдер \(`null` в этом примере\): `expect(TestBed.inject(NotProvided, null)).toBeNull();` После вызова `TestBed.inject` конфигурация `TestBed` замораживается на время текущей спецификации. |
|
`initTestEnvironment` | Инициализирует тестовую среду для всего тестового запуска. <br /> Заглушки тестирования вызывают его за вас, поэтому вам редко нужно делать это самостоятельно. <br /> Вызывайте этот метод _ровно один раз_. Чтобы изменить это значение по умолчанию в середине тестового запуска, сначала вызовите `resetTestEnvironment`. <br /> Укажите фабрику компилятора Angular, `PlatformRef` и модуль тестирования Angular по умолчанию. Альтернативы для платформ, отличных от браузера, доступны в общей форме `@angular/platform-<platform_name>/testing/<platform_name>`. |
| `resetTestEnvironment` | Сбрасывает начальную тестовую среду, включая модуль тестирования по умолчанию. |

Несколько методов экземпляра `TestBed` не охвачены статическими методами _класса_ `TestBed`.
Они редко нужны.

## `ComponentFixture` {#the-componentfixture}

`TestBed.createComponent<T>` создаёт экземпляр компонента `T` и возвращает строго типизированный `ComponentFixture` для этого компонента.

Свойства и методы `ComponentFixture` предоставляют доступ к компоненту, его представлению в DOM и аспектам его Angular-окружения.

### Свойства `ComponentFixture` {#componentfixture-properties}

Вот наиболее важные свойства для тестировщиков в порядке вероятной полезности.

| Свойства            | Подробности                                                                                                                                                                                                                                                                                  |
| :------------------ | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `componentInstance` | Экземпляр класса компонента, созданный `TestBed.createComponent`.                                                                                                                                                                                                                            |
| `debugElement`      | `DebugElement`, связанный с корневым элементом компонента. <br /> `debugElement` предоставляет информацию о компоненте и его DOM-элементе во время тестирования и отладки. Это критически важное свойство для тестировщиков. Наиболее интересные члены описаны [ниже](#debugelement). |
| `nativeElement`     | Нативный DOM-элемент в корне компонента.                                                                                                                                                                                                                                                     |
| `changeDetectorRef` | `ChangeDetectorRef` для компонента. <br /> `ChangeDetectorRef` наиболее ценен при тестировании компонента, использующего метод `ChangeDetectionStrategy.OnPush`, или когда обнаружение изменений компонента находится под вашим программным управлением.                                    |

### Методы `ComponentFixture` {#componentfixture-methods}

Методы _fixture_ заставляют Angular выполнять определённые задачи на дереве компонентов.
Вызывайте эти методы для активации поведения Angular в ответ на смоделированные действия пользователя.

Вот наиболее полезные методы для тестировщиков.

| Методы              | Подробности                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| :------------------ | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `detectChanges`     | Запускает цикл обнаружения изменений для компонента. <br /> Вызывайте его для инициализации компонента \(он вызывает `ngOnInit`\) и после изменения привязанных к данным значений свойств компонента в тестовом коде. Angular не видит изменения `personComponent.name` и не обновит привязку `name`, пока вы не вызовете `detectChanges`. <br /> После выполняет `checkNoChanges` для подтверждения отсутствия циклических обновлений, если только не вызвано как `detectChanges(false)`.                                                                       |
| `autoDetectChanges` | Установите в `true`, если хотите, чтобы fixture автоматически обнаруживал изменения. <br /> При включённом автоопределении тестовый fixture вызывает `detectChanges` сразу после создания компонента. Затем он прослушивает соответствующие события зоны и вызывает `detectChanges` соответственно. Если тестовый код напрямую изменяет значения свойств компонента, вероятно, всё равно нужно вызвать `fixture.detectChanges` для активации обновлений привязки данных. <br /> По умолчанию `false`. Тестировщики, предпочитающие точный контроль, обычно оставляют `false`. |
| `checkNoChanges`    | Выполняет обнаружение изменений, чтобы убедиться в отсутствии ожидающих изменений. Выбрасывает исключение, если они есть.                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `isStable`          | Если fixture в данный момент _стабилен_, возвращает `true`. Если есть незавершённые асинхронные задачи, возвращает `false`.                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `whenStable`        | Возвращает промис, который разрешается, когда fixture становится стабильным. <br /> Для продолжения тестирования после завершения асинхронной активности или асинхронного обнаружения изменений привяжитесь к этому промису. См. [whenStable](guide/testing/components-scenarios#whenstable).                                                                                                                                                                                                                                                                   |
| `destroy`           | Инициирует уничтожение компонента.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |

#### `DebugElement` {#debugelement}

`DebugElement` предоставляет важную информацию о представлении компонента в DOM.

Из `DebugElement` корневого тестового компонента, возвращаемого `fixture.debugElement`, можно обходить \(и запрашивать\) всё дерево элементов и компонентов fixture.

Вот наиболее полезные члены `DebugElement` для тестировщиков в приблизительном порядке полезности:

| Члены                 | Подробности                                                                                                                                                                                                                                                                                                                                                |
| :-------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `nativeElement`       | Соответствующий DOM-элемент в браузере.                                                                                                                                                                                                                                                                                                                    |
| `query`               | Вызов `query(predicate: Predicate<DebugElement>)` возвращает первый `DebugElement`, соответствующий предикату на любой глубине поддерева.                                                                                                                                                                                                                  |
| `queryAll`            | Вызов `queryAll(predicate: Predicate<DebugElement>)` возвращает все `DebugElements`, соответствующие предикату на любой глубине поддерева.                                                                                                                                                                                                                 |
| `injector`            | Инжектор зависимостей хост-элемента. Например, инжектор экземпляра компонента корневого элемента.                                                                                                                                                                                                                                                          |
| `componentInstance`   | Собственный экземпляр компонента элемента, если он есть.                                                                                                                                                                                                                                                                                                   |
| `context`             | Объект, предоставляющий родительский контекст для этого элемента. Зачастую это экземпляр компонента-предка, управляющего этим элементом. <br /> Когда элемент повторяется в блоке `@for`, контекстом является `RepeaterContext`, чьё свойство `$implicit` — значение экземпляра строки. Например, `hero` в `@for(hero of heroes; ...)`.                    |
| `children`            | Непосредственные дочерние `DebugElement`. Обходите дерево, спускаясь через `children`. `DebugElement` также имеет `childNodes` — список объектов `DebugNode`. `DebugElement` является производным от `DebugNode`, и узлов зачастую больше, чем элементов. Тестировщики обычно могут игнорировать простые узлы.                                             |
| `parent`              | Родительский `DebugElement`. `null`, если это корневой элемент.                                                                                                                                                                                                                                                                                            |
| `name`                | Имя тега элемента, если это элемент.                                                                                                                                                                                                                                                                                                                       |
| `triggerEventHandler` | Запускает событие по его имени, если в коллекции `listeners` элемента есть соответствующий обработчик. Второй параметр — _объект события_, ожидаемый обработчиком. <br /> Если у события нет обработчика или есть другая проблема, рассмотрите вызов `nativeElement.dispatchEvent(eventObject)`.                                                           |
| `listeners`           | Обратные вызовы, привязанные к свойствам `@Output` компонента и/или свойствам событий элемента.                                                                                                                                                                                                                                                            |
| `providerTokens`      | Токены поиска инжектора этого компонента. Включает сам компонент и токены, перечисленные в его метаданных `providers`.                                                                                                                                                                                                                                     |
| `source`              | Где найти этот элемент в шаблоне исходного компонента.                                                                                                                                                                                                                                                                                                     |
| `references`          | Словарь объектов, связанных с локальными переменными шаблона \(например, `#foo`\), с ключами по имени локальной переменной.                                                                                                                                                                                                                                |

Методы `DebugElement.query(predicate)` и `DebugElement.queryAll(predicate)` принимают предикат, фильтрующий поддерево исходного элемента для соответствующих `DebugElement`.

Предикат — это любой метод, принимающий `DebugElement` и возвращающий _истинное_ значение.
В следующем примере находятся все `DebugElements` со ссылкой на локальную переменную шаблона с именем "content":

```ts
// Filter for DebugElements with a #content reference
const contentRefs = el.queryAll((de) => de.references['content']);
```

Класс Angular `By` имеет три статических метода для распространённых предикатов:

| Статический метод         | Подробности                                                                      |
| :------------------------ | :------------------------------------------------------------------------------- |
| `By.all`                  | Возвращает все элементы.                                                         |
| `By.css(selector)`        | Возвращает элементы с соответствующими CSS-селекторами.                          |
| `By.directive(directive)` | Возвращает элементы, соответствующие экземпляру класса директивы в Angular.      |

```ts
// Can find DebugElement either by css selector or by directive
const h2 = fixture.debugElement.query(By.css('h2'));
const directive = fixture.debugElement.query(By.directive(Highlight));
```
