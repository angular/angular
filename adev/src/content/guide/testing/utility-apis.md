# API утилит тестирования {#testing-utility-apis}

NOTE: Пока это руководство обновляется для Vitest, некоторые описания и примеры API утилит представлены в контексте Karma/Jasmine. Мы активно работаем над добавлением эквивалентов для Vitest и обновлением соответствующих разделов.

На этой странице описаны наиболее полезные функции тестирования Angular.

Утилиты тестирования Angular включают `TestBed`, `ComponentFixture` и набор функций для управления тестовой средой.
Классы [`TestBed`](#testbed-class-summary) и [`ComponentFixture`](#the-componentfixture) рассмотрены отдельно.

Краткое описание автономных функций в порядке вероятной полезности:

| Функция      | Описание                                                                                                                                                                                                                                                               |
| :----------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| [`inject`]   | Внедряет один или несколько сервисов из текущего инжектора `TestBed` в тестовую функцию. Не может внедрить сервис, предоставляемый самим компонентом. Смотрите обсуждение [debugElement.injector](guide/testing/components-scenarios#get-injected-services). |
| `getTestBed` | Получает текущий экземпляр `TestBed`. Обычно не нужен, так как статических методов класса `TestBed` достаточно. Экземпляр `TestBed` раскрывает несколько редко используемых членов, недоступных как статические методы.                                                |

Для обработки сложных асинхронных сценариев или тестирования устаревших приложений на основе Zone.js смотрите руководство [Утилиты тестирования Zone.js](guide/testing/zone-js-testing-utilities).

## Краткое описание класса `TestBed` {#testbed-class-summary}

Класс `TestBed` — одна из основных утилит тестирования Angular.
Его API довольно обширен и поначалу может показаться сложным, поэтому изучайте его постепенно.
Сначала прочитайте начало этого руководства, чтобы освоить основы, прежде чем пытаться изучить полный API.

Определение модуля, передаваемое в `configureTestingModule`, является подмножеством метаданных `@NgModule`.

```ts
type TestModuleMetadata = {
  providers?: any[];
  declarations?: any[];
  imports?: any[];
  schemas?: Array<SchemaMetadata | any[]>;
};
```

Каждый метод переопределения принимает `MetadataOverride<T>`, где `T` — тип метаданных, соответствующий методу, то есть параметр `@NgModule`, `@Component`, `@Directive` или `@Pipe`.

```ts
type MetadataOverride<T> = {
  add?: Partial<T>;
  remove?: Partial<T>;
  set?: Partial<T>;
};
```

API `TestBed` состоит из статических методов класса, которые обновляют или обращаются к _глобальному_ экземпляру `TestBed`.

Внутренне все статические методы охватывают методы текущего экземпляра `TestBed` времени выполнения, который также возвращается функцией `getTestBed()`.

Вызывайте методы `TestBed` _внутри_ `beforeEach()`, чтобы обеспечить чистое начало перед каждым тестом.

Вот наиболее важные статические методы в порядке вероятной полезности.

| Методы                   | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| :----------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `configureTestingModule` | Тестовые прослойки создают [начальную тестовую среду](guide/testing) и тестовый модуль по умолчанию. Модуль по умолчанию настроен с базовыми объявлениями и некоторыми заменами Angular-сервисов, необходимыми каждому тестировщику. <br /> Вызывайте `configureTestingModule` для уточнения конфигурации тестового модуля под конкретный набор тестов путём добавления и удаления импортов, объявлений \(компонентов, директив и Pipe\) и провайдеров. |
| `compileComponents`      | Асинхронно компилирует тестовый модуль после завершения его настройки. **Необходимо** вызывать этот метод, если _любой_ из компонентов тестового модуля имеет асинхронно загружаемые ресурсы (например, блоки @defer). <br /> После вызова `compileComponents` конфигурация `TestBed` замораживается на время текущей спецификации.                                                                                                                                          |
| `createComponent<T>`     | Создаёт экземпляр компонента типа `T` на основе текущей конфигурации `TestBed`. После вызова `createComponent` конфигурация `TestBed` замораживается на время текущей спецификации.                                                                                                                                                                                                                                                                                                       |
| `overrideComponent`      | Заменяет метаданные для указанного класса компонента, который может быть глубоко вложен во внутренний модуль.                                                                                                                                                                                                                                                                                                                                                                             |
| `overrideDirective`      | Заменяет метаданные для указанного класса директивы, который может быть глубоко вложен во внутренний модуль.                                                                                                                                                                                                                                                                                                                                                                              |
| `overridePipe`           | Заменяет метаданные для указанного класса Pipe, который может быть глубоко вложен во внутренний модуль.                                                                                                                                                                                                                                                                                                                                                                                   |
| `overrideModule`         | Заменяет метаданные для указанного `NgModule`. Помните, что модули могут импортировать другие модули. Метод `overrideModule` может проникать глубоко в текущий тестовый модуль для изменения одного из внутренних модулей.                                                                                                                                                                                                                                                                |

|
`inject` | Получает сервис из текущего инжектора `TestBed`. Для этой цели часто достаточно функции `inject`. Но `inject` выбросит ошибку, если не сможет предоставить сервис. <br /> Что если сервис необязателен? <br /> Метод `TestBed.inject()` принимает необязательный второй параметр — объект для возврата, если Angular не может найти провайдер \(`null` в этом примере\): `expect(TestBed.inject(NotProvided, null)).toBeNull();` После вызова `TestBed.inject` конфигурация `TestBed` замораживается на время текущей спецификации. |
|
`initTestEnvironment` | Инициализирует тестовую среду для всего тестового запуска. <br /> Тестовые прослойки вызывают его за вас, поэтому редко нужно делать это самостоятельно. <br /> Вызывайте этот метод _ровно один раз_. Чтобы изменить значение по умолчанию в середине тестового запуска, сначала вызовите `resetTestEnvironment`. <br /> Укажите фабрику компилятора Angular, `PlatformRef` и тестовый модуль Angular по умолчанию. Альтернативы для небраузерных платформ доступны в общем виде `@angular/platform-<platform_name>/testing/<platform_name>`. |
| `resetTestEnvironment` | Сбрасывает начальную тестовую среду, включая тестовый модуль по умолчанию. |

Несколько методов экземпляра `TestBed` не охвачены статическими методами _класса_ `TestBed`.
Они редко нужны.

## `ComponentFixture` {#the-componentfixture}

`TestBed.createComponent<T>` создаёт экземпляр компонента `T` и возвращает строго типизированный `ComponentFixture` для этого компонента.

Свойства и методы `ComponentFixture` обеспечивают доступ к компоненту, его DOM-представлению и аспектам его Angular-среды.

### Свойства `ComponentFixture` {#componentfixture-properties}

Вот наиболее важные свойства для тестировщиков в порядке вероятной полезности.

| Свойства            | Описание                                                                                                                                                                                                                                                                               |
| :------------------ | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `componentInstance` | Экземпляр класса компонента, созданный `TestBed.createComponent`.                                                                                                                                                                                                                      |
| `debugElement`      | `DebugElement`, связанный с корневым элементом компонента. <br /> `debugElement` даёт представление о компоненте и его DOM-элементе во время тестирования и отладки. Это критически важное свойство для тестировщиков. Наиболее интересные члены рассмотрены [ниже](#debugelement). |
| `nativeElement`     | Нативный DOM-элемент в корне компонента.                                                                                                                                                                                                                                               |
| `changeDetectorRef` | `ChangeDetectorRef` для компонента. <br /> `ChangeDetectorRef` наиболее ценен при тестировании компонента с методом `ChangeDetectionStrategy.OnPush` или когда обнаружение изменений компонента находится под программным контролем.                                                    |

### Методы `ComponentFixture` {#componentfixture-methods}

Методы _fixture_ заставляют Angular выполнять определённые задачи в дереве компонентов.
Вызывайте эти методы для инициирования поведения Angular в ответ на имитируемые действия пользователя.

Вот наиболее полезные методы для тестировщиков.

| Методы              | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| :------------------ | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `detectChanges`     | Запускает цикл обнаружения изменений для компонента. <br /> Вызывайте для инициализации компонента \(вызывает `ngOnInit`\) и после изменения значений привязанных свойств компонента в тестовом коде. Angular не видит, что вы изменили `personComponent.name`, и не обновит привязку `name`, пока не будет вызван `detectChanges`. <br /> После этого выполняет `checkNoChanges` для подтверждения отсутствия циклических обновлений, если только не вызван как `detectChanges(false)`;                                                               |
| `autoDetectChanges` | Установите в `true`, если хотите, чтобы fixture автоматически обнаруживал изменения. <br /> При autodetect равном `true` fixture вызывает `detectChanges` сразу после создания компонента. Затем прослушивает соответствующие события зоны и вызывает `detectChanges` соответственно. Если тестовый код напрямую изменяет значения свойств компонента, вероятно, всё равно нужно вызвать `fixture.detectChanges` для запуска обновлений привязки данных. <br /> По умолчанию: `false`. Тестировщики, предпочитающие точный контроль, как правило, оставляют его `false`. |
| `checkNoChanges`    | Выполняет запуск обнаружения изменений, чтобы убедиться в отсутствии ожидающих изменений. При их наличии выбрасывает исключение.                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `isStable`          | Возвращает `true`, если fixture в настоящее время _стабилен_. Возвращает `false`, если есть незавершённые асинхронные задачи.                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `whenStable`        | Возвращает промис, который разрешается, когда fixture становится стабильным. <br /> Используйте этот промис для продолжения тестирования после завершения асинхронной активности или асинхронного обнаружения изменений. Смотрите [whenStable](guide/testing/components-scenarios#whenstable).                                                                                                                                                                                                                                                          |
| `destroy`           | Инициирует уничтожение компонента.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |

#### `DebugElement` {#debugelement}

`DebugElement` предоставляет важную информацию о DOM-представлении компонента.

Из `DebugElement` корневого тестового компонента, возвращаемого `fixture.debugElement`, можно обходить \(и запрашивать\) всё дерево элементов и компонентов fixture.

Вот наиболее полезные члены `DebugElement` для тестировщиков примерно в порядке полезности:

| Члены                 | Описание                                                                                                                                                                                                                                                                                                                                                    |
| :-------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `nativeElement`       | Соответствующий DOM-элемент в браузере                                                                                                                                                                                                                                                                                                                      |
| `query`               | Вызов `query(predicate: Predicate<DebugElement>)` возвращает первый `DebugElement`, соответствующий предикату на любой глубине в поддереве.                                                                                                                                                                                                                 |
| `queryAll`            | Вызов `queryAll(predicate: Predicate<DebugElement>)` возвращает все `DebugElement`, соответствующие предикату на любой глубине в поддереве.                                                                                                                                                                                                                 |
| `injector`            | Инжектор зависимостей хоста. Например, инжектор экземпляра компонента корневого элемента.                                                                                                                                                                                                                                                                   |
| `componentInstance`   | Собственный экземпляр компонента элемента, если он есть.                                                                                                                                                                                                                                                                                                    |
| `context`             | Объект, предоставляющий родительский контекст для этого элемента. Часто экземпляр компонента-предка, управляющего этим элементом. <br /> Когда элемент повторяется в блоке `@for`, контекст — это `RepeaterContext`, чьё свойство `$implicit` является значением экземпляра строки. Например, `hero` в `@for(hero of heroes; ...)`. |
| `children`            | Непосредственные дочерние `DebugElement`. Обход дерева выполняется через `children`. У `DebugElement` также есть `childNodes` — список объектов `DebugNode`. `DebugElement` наследуется от `DebugNode`, и узлов часто больше, чем элементов. Тестировщики обычно игнорируют обычные узлы.                                                                    |
| `parent`              | Родительский `DebugElement`. Null, если это корневой элемент.                                                                                                                                                                                                                                                                                               |
| `name`                | Имя тега элемента, если это элемент.                                                                                                                                                                                                                                                                                                                        |
| `triggerEventHandler` | Запускает событие по его имени, если в коллекции `listeners` элемента есть соответствующий обработчик. Второй параметр — _объект события_, ожидаемый обработчиком. <br /> Если у события нет обработчика или есть другая проблема, рассмотрите вызов `nativeElement.dispatchEvent(eventObject)`.                                                            |
| `listeners`           | Обратные вызовы, привязанные к свойствам `@Output` компонента и/или свойствам событий элемента.                                                                                                                                                                                                                                                             |
| `providerTokens`      | Токены поиска инжектора этого компонента. Включает сам компонент плюс токены, перечисленные компонентом в метаданных `providers`.                                                                                                                                                                                                                            |
| `source`              | Где найти этот элемент в исходном шаблоне компонента.                                                                                                                                                                                                                                                                                                       |
| `references`          | Словарь объектов, связанных с локальными переменными шаблона \(например, `#foo`\), с ключами по именам локальных переменных.                                                                                                                                                                                                                                |

Методы `DebugElement.query(predicate)` и `DebugElement.queryAll(predicate)` принимают предикат, который фильтрует поддерево исходного элемента в поиске соответствующих `DebugElement`.

Предикат — это любой метод, принимающий `DebugElement` и возвращающий _истинное_ значение.
Следующий пример находит все `DebugElement` со ссылкой на локальную переменную шаблона «content»:

```ts
// Filter for DebugElements with a #content reference
const contentRefs = el.queryAll((de) => de.references['content']);
```

Класс Angular `By` имеет три статических метода для распространённых предикатов:

| Статический метод         | Описание                                                                          |
| :------------------------ | :-------------------------------------------------------------------------------- |
| `By.all`                  | Возвращает все элементы                                                           |
| `By.css(selector)`        | Возвращает элементы с соответствующими CSS-селекторами                            |
| `By.directive(directive)` | Возвращает элементы, которым Angular сопоставил экземпляр класса директивы       |

```ts
// Can find DebugElement either by css selector or by directive
const h2 = fixture.debugElement.query(By.css('h2'));
const directive = fixture.debugElement.query(By.directive(Highlight));
```
