# Оптимизация размера клиентского приложения с помощью облегчённых токенов внедрения

На этой странице представлен концептуальный обзор техники внедрения зависимостей, рекомендованной для разработчиков библиотек.
Проектирование библиотеки с использованием _облегчённых токенов внедрения_ помогает оптимизировать размер бандла клиентских приложений, использующих вашу библиотеку.

Вы можете управлять структурой зависимостей между компонентами и injectable-сервисами для оптимизации размера бандла с помощью tree-shakable провайдеров.
Обычно это гарантирует, что если предоставленный компонент или сервис никогда реально не используется приложением, компилятор может удалить его код из бандла.

Из-за особенностей хранения токенов внедрения в Angular возможно, что такой неиспользуемый компонент или сервис всё равно окажется в бандле.
На этой странице описывается паттерн проектирования внедрения зависимостей, который поддерживает правильный tree-shaking с использованием облегчённых токенов внедрения.

Паттерн облегчённых токенов внедрения особенно важен для разработчиков библиотек.
Он гарантирует, что когда приложение использует лишь часть возможностей вашей библиотеки, неиспользуемый код может быть исключён из бандла клиентского приложения.

Когда приложение использует вашу библиотеку, могут быть некоторые сервисы, которые библиотека предоставляет, но клиентское приложение не использует.
В этом случае разработчик приложения должен ожидать, что этот сервис будет подвергнут tree-shaking и не увеличит размер скомпилированного приложения.
Поскольку разработчик приложения не может знать о проблеме tree-shaking в библиотеке и не может её устранить, ответственность за это лежит на разработчике библиотеки.
Чтобы предотвратить сохранение неиспользуемых компонентов, ваша библиотека должна использовать паттерн облегчённых токенов внедрения.

## Когда токены сохраняются {#when-tokens-are-retained}

Для лучшего понимания условий, при которых происходит сохранение токенов, рассмотрим библиотеку, предоставляющую компонент library-card.
Этот компонент содержит тело и может содержать необязательный заголовок:

```html
<lib-card>
  <lib-header>…</lib-header>
</lib-card>
```

В вероятной реализации компонент `<lib-card>` использует `contentChild` или `contentChildren` для получения `<lib-header>` и `<lib-body>`, как в следующем примере:

```ts {highlight: [14]}
import {Component, contentChild} from '@angular/core';

@Component({
  selector: 'lib-header',
  …,
})
class LibHeader {}

@Component({
  selector: 'lib-card',
  …,
})
class LibCard {
  readonly header = contentChild(LibHeader);
}
```

Поскольку `<lib-header>` является необязательным, элемент может появляться в шаблоне в минимальной форме `<lib-card />`.
В этом случае `<lib-header>` не используется, и можно ожидать его удаления через tree-shaking, однако этого не происходит.
Это связано с тем, что `LibCard` фактически содержит две ссылки на `LibHeader`:

```ts
readonly header = contentChild(LibHeader);
```

- Одна из этих ссылок находится в _позиции типа_ — то есть указывает `LibHeader` в качестве типа: `readonly header: Signal<LibHeader|undefined>`.
- Другая ссылка находится в _позиции значения_ — то есть `LibHeader` является значением, передаваемым в функцию `contentChild`: `contentChild(LibHeader)`.

Компилятор обрабатывает ссылки на токены в этих позициях по-разному:

- Компилятор стирает ссылки в _позиции типа_ после преобразования из TypeScript, поэтому они не влияют на tree-shaking.
- Компилятор должен сохранять ссылки в _позиции значения_ во время выполнения, что **предотвращает** удаление компонента через tree-shaking.

В примере компилятор сохраняет токен `LibHeader`, который встречается в позиции значения.
Это предотвращает удаление компонента через tree-shaking, даже если приложение фактически не использует `<lib-header>` нигде.
Если код, шаблон и стили `LibHeader` в сумме становятся слишком большими, их ненужное включение может значительно увеличить размер клиентского приложения.

## Когда использовать паттерн облегчённых токенов внедрения {#when-to-use-the-lightweight-injection-token-pattern}

Проблема tree-shaking возникает, когда компонент используется в качестве токена внедрения.
Это может происходить в двух случаях:

- Токен используется в позиции значения [контентного запроса](guide/components/queries#content-queries).
- Токен используется с функцией `inject`.

В следующем примере оба использования токена `CustomOther` приводят к его сохранению, предотвращая его удаление через tree-shaking, когда он не используется:

```ts {highlight: [[2],[4]]}
class App {
  private readonly other = inject(CustomOther, {optional: true});

  readonly header = contentChild(CustomOther);
}
```

Хотя токены, используемые только как спецификаторы типов, удаляются при преобразовании в JavaScript, все токены, используемые для внедрения зависимостей, необходимы во время выполнения.
При использовании `inject(CustomOther)` `CustomOther` передаётся как аргумент значения.
Теперь токен находится в позиции значения, что заставляет tree-shaker сохранять ссылку.

HELPFUL: Библиотеки должны использовать [tree-shakable провайдеры](guide/di/defining-dependency-providers) для всех сервисов, предоставляя зависимости на корневом уровне, а не в компонентах или модулях.

## Использование облегчённых токенов внедрения {#using-lightweight-injection-tokens}

Паттерн облегчённых токенов внедрения состоит в использовании небольшого абстрактного класса в качестве токена внедрения с предоставлением фактической реализации на более позднем этапе.
Абстрактный класс сохраняется и не подвергается tree-shaking, но он небольшой и не оказывает существенного влияния на размер приложения.

Следующий пример показывает, как это работает для `LibHeader`:

```ts {highlight: [[1],[5], [15]]}
abstract class LibHeaderToken {}

@Component({
  selector: 'lib-header',
  providers: [{provide: LibHeaderToken, useExisting: LibHeader}],
  …,
})
class LibHeader extends LibHeaderToken {}

@Component({
  selector: 'lib-card',
  …,
})
class LibCard {
  readonly header = contentChild(LibHeaderToken);
}
```

В этом примере реализация `LibCard` больше не ссылается на `LibHeader` ни в позиции типа, ни в позиции значения.
Это позволяет полностью удалить `LibHeader` через tree-shaking.
`LibHeaderToken` сохраняется, но это только объявление класса без конкретной реализации.
Он небольшой и не оказывает существенного влияния на размер приложения после компиляции.

Вместо этого сам `LibHeader` реализует абстрактный класс `LibHeaderToken`.
Вы можете безопасно использовать этот токен в качестве провайдера в определении компонента, позволяя Angular корректно внедрять конкретный тип.

Подводя итог, паттерн облегчённых токенов внедрения состоит из следующего:

1. Облегчённый токен внедрения, представленный как абстрактный класс.
2. Определение компонента, реализующего абстрактный класс.
3. Внедрение облегчённого паттерна с помощью `contentChild` или `contentChildren`.
4. Провайдер в реализации облегчённого токена внедрения, связывающий токен с реализацией.

### Использование облегчённого токена внедрения для определения API {#use-the-lightweight-injection-token-for-api-definition}

Компонент, внедряющий облегчённый токен внедрения, может нуждаться в вызове метода внедрённого класса.
Теперь токен является абстрактным классом. Поскольку injectable-компонент реализует этот класс, необходимо также объявить абстрактный метод в абстрактном классе облегчённого токена.
Реализация метода со всем его кодом находится в injectable-компоненте, который может быть удалён через tree-shaking.
Это позволяет родителю взаимодействовать с дочерним компонентом, если он присутствует, типобезопасным образом.

Например, `LibCard` теперь запрашивает `LibHeaderToken`, а не `LibHeader`.
Следующий пример показывает, как паттерн позволяет `LibCard` взаимодействовать с `LibHeader`, фактически не ссылаясь на `LibHeader`:

```ts {highlight: [[2],[7],[11],[19]]}
abstract class LibHeaderToken {
  abstract doSomething(): void;
}

@Component({
  selector: 'lib-header',
  providers: [{provide: LibHeaderToken, useExisting: LibHeader}],
})
class LibHeader extends LibHeaderToken {
  doSomething(): void {
    // Конкретная реализация `doSomething`
  }
}

@Component({
  selector: 'lib-card',
})
class LibCard implements AfterContentInit {
  readonly header = contentChild(LibHeaderToken);

  ngAfterContentInit(): void {
    if (this.header() !== undefined) {
      this.header()!.doSomething();
    }
  }
}
```

В этом примере родитель запрашивает токен для получения дочернего компонента и сохраняет ссылку на результирующий компонент, если он присутствует.
Перед вызовом метода дочернего компонента родительский компонент проверяет наличие дочернего компонента.
Если дочерний компонент был удалён через tree-shaking, во время выполнения нет ссылки на него и нет вызова его метода.

### Именование облегчённых токенов внедрения {#naming-your-lightweight-injection-token}

Облегчённые токены внедрения полезны только с компонентами.
[Руководство по стилю Angular](style-guide) предлагает называть компоненты без суффикса `Component`.
Пример `LibHeader` следует этому соглашению.

Следует поддерживать связь между компонентом и его токеном, при этом различая их.
Рекомендуемый стиль — использовать базовое имя компонента с суффиксом `Token` для именования облегчённых токенов внедрения: `LibHeaderToken`.
