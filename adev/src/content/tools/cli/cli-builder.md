# Angular CLI builders

Ряд команд Angular CLI выполняют сложные процессы над вашим кодом, такие как сборка, тестирование или запуск приложения.
Команды используют внутренний инструмент Architect для запуска _CLI builders_, которые вызывают другой инструмент (бандлер, тест-раннер, сервер) для выполнения необходимой задачи.
Пользовательские builders могут выполнять совершенно новую задачу или изменять, какой сторонний инструмент используется существующей командой.

Этот документ объясняет, как CLI builders интегрируются с файлом конфигурации рабочего пространства, и показывает, как создать собственный builder.

HELPFUL: Найдите код из примеров, используемых здесь, в этом [репозитории GitHub](https://github.com/mgechev/cli-builders-demo).

## CLI builders {#cli-builders}

Внутренний инструмент Architect делегирует работу функциям-обработчикам, называемым _builders_.
Функция-обработчик builder получает два аргумента:

| Аргумент  | Тип              |
| :-------- | :--------------- |
| `options` | `JSONObject`     |
| `context` | `BuilderContext` |

Разделение ответственности здесь такое же, как у [схематиков](tools/cli/schematics-authoring), которые используются для других команд CLI, изменяющих ваш код (например, `ng generate`).

- Объект `options` предоставляется из пользовательских параметров и конфигурации CLI, тогда как объект `context` предоставляется CLI Builder API автоматически.
- Помимо контекстной информации, объект `context` также предоставляет доступ к методу планирования `context.scheduleTarget()`.
  Планировщик выполняет функцию-обработчик builder с заданной конфигурацией цели.

Функция-обработчик builder может быть синхронной (возвращать значение), асинхронной (возвращать `Promise`) или наблюдать за событиями и возвращать несколько значений (возвращать `Observable`).
Возвращаемые значения всегда должны быть типа `BuilderOutput`.
Этот объект содержит булево поле `success` и необязательное поле `error`, которое может содержать сообщение об ошибке.

Angular предоставляет ряд builders, используемых CLI для команд `ng build` и `ng test`.
Конфигурации целей по умолчанию для этих и других встроенных CLI builders можно найти и настроить в разделе "architect" [файла конфигурации рабочего пространства](reference/configs/workspace-config), `angular.json`.
Кроме того, можно расширить и настроить Angular, создав собственные builders, которые можно запускать напрямую с помощью [команды CLI `ng run`](cli/run).

### Структура проекта builder {#builder-project-structure}

Builder находится в папке "project", которая по структуре похожа на рабочее пространство Angular: файлы глобальной конфигурации находятся на верхнем уровне, а более специфичная конфигурация — в папке с исходным кодом.
Например, папка `myBuilder` может содержать следующие файлы.

| Файлы                    | Назначение                                                                                                |
| :----------------------- | :-------------------------------------------------------------------------------------------------------- |
| `src/my-builder.ts`      | Основной исходный файл с определением builder.                                                            |
| `src/my-builder.spec.ts` | Исходный файл с тестами.                                                                                  |
| `src/schema.json`        | Определение входных параметров builder.                                                                   |
| `builders.json`          | Определение builders.                                                                                     |
| `package.json`           | Зависимости. См. [https://docs.npmjs.com/files/package.json](https://docs.npmjs.com/files/package.json). |
| `tsconfig.json`          | [Конфигурация TypeScript](https://www.typescriptlang.org/docs/handbook/tsconfig-json.html).              |

Builders можно публиковать в `npm`, см. [Публикация библиотеки](tools/libraries/creating-libraries).

### Отчёт о прогрессе и статусе {#progress-and-status-reporting}

CLI Builder API включает инструменты отчётности о прогрессе и статусе, которые могут предоставлять подсказки для определённых функций и интерфейсов.

Для отчёта о прогрессе используйте метод `context.reportProgress()`, который принимает текущее значение, необязательное итоговое значение и строку статуса в качестве аргументов.
Итоговое значение может быть любым числом. Например, если вы знаете, сколько файлов нужно обработать, итоговое значение может быть количеством файлов, а текущее — количеством уже обработанных.
Строка статуса не изменяется, если не передать новое строковое значение.

В нашем примере операция копирования либо завершается, либо ещё выполняется, поэтому нет необходимости в отчёте о прогрессе, но можно сообщать статус, чтобы родительский builder, вызвавший наш builder, знал, что происходит.
Используйте метод `context.reportStatus()` для формирования строки статуса произвольной длины.

HELPFUL: Нет гарантии, что длинная строка будет отображена полностью; она может быть обрезана для отображения в пользовательском интерфейсе.

Передайте пустую строку, чтобы убрать статус.

## Входные данные builder {#builder-input}

Вы можете вызвать builder косвенно через команду CLI, например `ng build`, или напрямую с помощью команды Angular CLI `ng run`.
В любом случае необходимо предоставить обязательные входные данные, но можно оставить другие параметры со значениями по умолчанию, которые предварительно настроены для конкретной _цели_, заданной [конфигурацией](tools/cli/environments) или переданной через командную строку.

### Валидация входных данных {#input-validation}

Входные данные builder определяются в JSON-схеме, связанной с этим builder.
Как и у схематиков, инструмент Architect собирает разрешённые входные значения в объект `options` и проверяет их типы по схеме перед передачей в функцию builder.

Для нашего примера builder параметр `options` должен быть `JsonObject` с двумя ключами:
`source` и `destination`, каждый из которых является строкой.

Для типовой проверки этих значений можно предоставить следующую схему.

```json {header: "schema.json"}
{
  "$schema": "http://json-schema.org/schema",
  "type": "object",
  "properties": {
    "source": {
      "type": "string"
    },
    "destination": {
      "type": "string"
    }
  }
}
```

HELPFUL: Это минимальный пример, но использование схемы для валидации может быть очень мощным инструментом.
Для получения дополнительной информации см. [сайт JSON schemas](http://json-schema.org).

Чтобы связать реализацию builder с его схемой и именем, необходимо создать файл _определения builder_, на который можно сослаться в `package.json`.

Создайте файл с именем `builders.json` следующего содержания:

```json {header: "builders.json"}
{
  "builders": {
    "copy": {
      "implementation": "./dist/my-builder.js",
      "schema": "./src/schema.json",
      "description": "Copies a file."
    }
  }
}
```

В файле `package.json` добавьте ключ `builders`, который указывает инструменту Architect, где найти файл определения builder.

```json {header: "package.json"}
{
  "name": "@example/copy-file",
  "version": "1.0.0",
  "description": "Builder for copying files",
  "builders": "builders.json",
  "dependencies": {
    "@angular-devkit/architect": "~0.1200.0",
    "@angular-devkit/core": "^12.0.0"
  }
}
```

Официальное имя нашего builder теперь `@example/copy-file:copy`.
Первая часть — имя пакета, вторая — имя builder, указанное в файле `builders.json`.

Доступ к этим значениям осуществляется через `options.source` и `options.destination`.

### Конфигурация цели {#target-configuration}

Builder должен иметь определённую цель, связывающую его с конкретной конфигурацией входных данных и проектом.

Цели определяются в [файле конфигурации CLI](reference/configs/workspace-config) `angular.json`.
Цель указывает используемый builder, конфигурацию параметров по умолчанию и именованные альтернативные конфигурации.
Architect в Angular CLI использует определение цели для разрешения входных параметров при конкретном запуске.

Файл `angular.json` содержит раздел для каждого проекта, и раздел "architect" каждого проекта настраивает цели для builders, используемых CLI командами 'build', 'test' и 'serve'.
По умолчанию, например, команда `ng build` запускает builder `@angular-devkit/build-angular:browser` для выполнения задачи сборки и передаёт значения параметров по умолчанию, указанные для цели `build` в `angular.json`.

```json {header: "angular.json"}
{
  "myApp": {
    "...": "...",
    "architect": {
      "build": {
        "builder": "@angular-devkit/build-angular:browser",
        "options": {
          "outputPath": "dist/myApp",
          "index": "src/index.html",
          "...": "..."
        },
        "configurations": {
          "production": {
            "fileReplacements": [
              {
                "replace": "src/environments/environment.ts",
                "with": "src/environments/environment.prod.ts"
              }
            ],
            "optimization": true,
            "outputHashing": "all",
            "...": "..."
          }
        }
      },
      "...": "..."
    }
  }
}
```

Команда передаёт builder набор параметров по умолчанию, указанных в разделе "options".
При передаче флага `--configuration=production` используются переопределяющие значения, указанные в конфигурации `production`.
Дополнительные переопределения параметров можно указать отдельно в командной строке.

#### Строки целей {#target-strings}

Обобщённая команда CLI `ng run` принимает в качестве первого аргумента строку цели следующего вида.

```shell

project:target[:configuration]

```

|               | Описание                                                                                                              |
| :------------ | :-------------------------------------------------------------------------------------------------------------------- |
| project       | Имя проекта Angular CLI, с которым связана цель.                                                                      |
| target        | Именованная конфигурация builder из раздела `architect` файла `angular.json`.                                         |
| configuration | (необязательно) Имя конкретного переопределения конфигурации для заданной цели, как определено в файле `angular.json`. |

Если ваш builder вызывает другой builder, ему может потребоваться прочитать переданную строку цели.
Разберите эту строку в объект с помощью вспомогательной функции `targetFromTargetString()` из `@angular-devkit/architect`.

## Планирование и запуск {#schedule-and-run}

Architect запускает builders асинхронно.
Для вызова builder планируется задача для выполнения после завершения всего разрешения конфигурации.

Функция builder не выполняется до тех пор, пока планировщик не вернёт управляющий объект `BuilderRun`.
CLI обычно планирует задачи, вызывая функцию `context.scheduleTarget()`, а затем разрешает входные параметры с помощью определения цели в файле `angular.json`.

Architect разрешает входные параметры для заданной цели, взяв объект параметров по умолчанию, затем перезаписывая значения из конфигурации, а затем дополнительно перезаписывая значения из объекта overrides, переданного в `context.scheduleTarget()`.
В Angular CLI объект overrides формируется из аргументов командной строки.

Architect проверяет полученные значения параметров по схеме builder.
Если входные данные корректны, Architect создаёт контекст и выполняет builder.

Для получения дополнительной информации см. [Конфигурация рабочего пространства](reference/configs/workspace-config).

HELPFUL: Можно также вызвать builder напрямую из другого builder или теста, вызвав `context.scheduleBuilder()`.
Объект `options` передаётся напрямую в метод, и значения параметров проверяются по схеме builder без дополнительных корректировок.

Только метод `context.scheduleTarget()` разрешает конфигурацию и переопределения через файл `angular.json`.

### Добавление цели {#adding-a-target}

Добавим новую цель, которая будет запускать наш builder для копирования файла.
Эта цель указывает builder скопировать файл `package.json`.

- Мы добавим новый раздел цели в объект `architect` нашего проекта
- Цель с именем `copy-package` использует наш builder, опубликованный в `@example/copy-file`.
- Объект options предоставляет значения по умолчанию для двух входных данных, которые вы определили.
  - `source` — существующий файл, который вы копируете.
  - `destination` — путь, в который нужно скопировать.

```json {header: "angular.json"}
{
  "projects": {
    "builder-test": {
      "architect": {
        "copy-package": {
          "builder": "@example/copy-file:copy",
          "options": {
            "source": "package.json",
            "destination": "package-copy.json"
          }
        }
        // Existing targets...
      }
    }
  }
}
```

### Запуск builder {#running-the-builder}

Для запуска нашего builder с конфигурацией по умолчанию новой цели используйте следующую команду CLI.

```shell

ng run builder-test:copy-package

```

Это скопирует файл `package.json` в `package-copy.json`.

Используйте аргументы командной строки для переопределения настроенных значений по умолчанию.
Например, для запуска с другим значением `destination` используйте следующую команду CLI.

```shell

ng run builder-test:copy-package --destination=package-other.json

```

Это скопирует файл в `package-other.json` вместо `package-copy.json`.
Поскольку вы не переопределили параметр _source_, копирование по-прежнему выполняется из файла `package.json` по умолчанию.

### Режим наблюдения {#watch-mode}

Большинство builders выполняются один раз и возвращают результат. Однако такое поведение не вполне совместимо с builder, который следит за изменениями (например, devserver).
Architect может поддерживать режим наблюдения, но есть несколько моментов, на которые следует обратить внимание.

- Для использования с режимом наблюдения функция-обработчик builder должна возвращать `Observable`.
  Architect подписывается на `Observable` до его завершения и может повторно использовать его, если builder снова запланирован с теми же аргументами.

- Builder всегда должен генерировать объект `BuilderOutput` после каждого выполнения.
  После выполнения он может перейти в режим наблюдения, ожидая запуска внешним событием.
  Если событие инициирует перезапуск, builder должен выполнить функцию `context.reportRunning()`, чтобы сообщить Architect, что он снова работает.
  Это предотвращает остановку builder системой Architect при планировании нового запуска.

Когда ваш builder вызывает `BuilderRun.stop()` для выхода из режима наблюдения, Architect отписывается от `Observable` builder и вызывает логику очистки builder.
Это поведение также позволяет останавливать и очищать длительные сборки.

В целом, если ваш builder следит за внешним событием, рекомендуется разделить его выполнение на три фазы.

| Фазы        | Описание                                                                                                                                                                                                                                                         |
| :---------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Выполнение  | Выполняемая задача, например вызов компилятора. Завершается, когда компилятор заканчивает работу и ваш builder генерирует объект `BuilderOutput`.                                                                                                                 |
| Наблюдение  | Между двумя запусками — наблюдение за потоком внешних событий. Например, слежение за файловой системой на предмет изменений. Завершается, когда компилятор перезапускается и вызывается `context.reportRunning()`.                                                |
| Завершение  | Либо задача полностью выполнена (например, компилятор должен был запуститься несколько раз), либо выполнение builder было остановлено (с помощью `BuilderRun.stop()`). Architect выполняет логику очистки и отписывается от `Observable` вашего builder. |

## Итоги {#summary}

CLI Builder API предоставляет возможность изменить поведение Angular CLI с помощью builders для выполнения пользовательской логики.

- Builders могут быть синхронными или асинхронными, выполняться один раз или следить за внешними событиями, а также планировать другие builders или цели.
- Builders имеют параметры по умолчанию, указанные в файле конфигурации `angular.json`, которые могут быть перезаписаны альтернативной конфигурацией для цели и дополнительно переопределены флагами командной строки.
- Команда Angular рекомендует использовать интеграционные тесты для тестирования Architect builders. Используйте модульные тесты для проверки логики, которую выполняет builder.
- Если ваш builder возвращает `Observable`, он должен выполнять очистку в логике teardown этого `Observable`.
