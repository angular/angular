# Проверка типов в шаблонах {#template-type-checking}

## Обзор проверки типов в шаблонах {#overview-of-template-type-checking}

Подобно тому, как TypeScript обнаруживает ошибки типов в коде, Angular проверяет выражения и привязки в шаблонах приложения и может сообщать об обнаруженных ошибках типов.
В настоящее время Angular поддерживает три режима, в зависимости от значений флагов `fullTemplateTypeCheck` и `strictTemplates` в [параметрах компилятора Angular](reference/configs/angular-compiler-options).

### Базовый режим {#basic-mode}

В самом базовом режиме проверки типов с флагом `fullTemplateTypeCheck`, установленным в `false`, Angular проверяет только выражения верхнего уровня в шаблоне.

При написании `<map [city]="user.address.city">` компилятор проверяет следующее:

- `user` является свойством класса компонента
- `user` — объект со свойством address
- `user.address` — объект со свойством city

Компилятор не проверяет, совместимо ли значение `user.address.city` с входным параметром city компонента `<map>`.

В этом режиме компилятор также имеет ряд существенных ограничений:

- В частности, он не проверяет встроенные представления, такие как `*ngIf`, `*ngFor` и другие встроенные представления `<ng-template>`.
- Он не определяет типы `#refs`, результаты каналов или тип `$event` в привязках событий.

Во многих случаях эти элементы имеют тип `any`, что может привести к тому, что последующие части выражения останутся непроверенными.

### Полный режим {#full-mode}

Если флаг `fullTemplateTypeCheck` установлен в `true`, Angular более агрессивно проверяет типы в шаблонах.
В частности:

- Встроенные представления \(например, внутри `*ngIf` или `*ngFor`\) проверяются
- Каналы имеют правильный возвращаемый тип
- Локальные ссылки на директивы и каналы имеют правильный тип \(за исключением обобщённых параметров, которые будут `any`\)

Следующие элементы по-прежнему имеют тип `any`.

- Локальные ссылки на DOM-элементы
- Объект `$event`
- Выражения безопасной навигации

IMPORTANT: Флаг `fullTemplateTypeCheck` устарел в Angular 13.
Вместо него следует использовать семейство параметров компилятора `strictTemplates`.

### Строгий режим {#strict-mode}

Angular сохраняет поведение флага `fullTemplateTypeCheck` и вводит третий «строгий режим».
Строгий режим — это расширенный набор возможностей полного режима, активируемый установкой флага `strictTemplates` в true.
Этот флаг заменяет флаг `fullTemplateTypeCheck`.

В дополнение к поведению полного режима Angular выполняет следующее:

- Проверяет, что привязки компонентов/директив совместимы с их `input()`-параметрами
- Учитывает флаг TypeScript `strictNullChecks` при проверке предыдущего режима
- Определяет правильный тип компонентов/директив, включая обобщённые типы
- Определяет контекстные типы шаблонов там, где это настроено \(например, обеспечивая корректную проверку типов `NgFor`\)
- Определяет правильный тип `$event` в привязках событий компонентов/директив, DOM и анимационных событий
- Определяет правильный тип локальных ссылок на DOM-элементы на основе имени тега \(например, тип, который `document.createElement` вернул бы для этого тега\)

## Проверка `*ngFor` {#checking-of-ngfor}

Три режима проверки типов по-разному обрабатывают встроенные представления.
Рассмотрим следующий пример.

```ts {header:"User interface"}
interface User {
  name: string;
  address: {
    city: string;
    state: string;
  };
}
```

```html
<div *ngFor="let user of users">
  <h2>{{config.title}}</h2>
  <span>City: {{user.address.city}}</span>
</div>
```

`<h2>` и `<span>` находятся во встроенном представлении `*ngFor`.
В базовом режиме Angular не проверяет ни один из них.
В полном режиме Angular проверяет, что `config` и `user` существуют, и предполагает тип `any`.
В строгом режиме Angular знает, что `user` в `<span>` имеет тип `User`, а `address` — объект со свойством `city` типа `string`.

## Устранение ошибок шаблона {#troubleshooting-template-errors}

В строгом режиме могут возникать ошибки шаблона, которых не было в предыдущих режимах.
Эти ошибки часто представляют реальные несоответствия типов в шаблонах, которые не были обнаружены предыдущим инструментарием.
В этом случае сообщение об ошибке должно указывать место в шаблоне, где возникла проблема.

Также возможны ложноположительные срабатывания, когда определения типов Angular-библиотеки неполные или неправильные, или когда типы не совсем соответствуют ожиданиям в следующих случаях.

- Когда типы библиотеки неправильные или неполные \(например, отсутствует `null | undefined`, если библиотека не была написана с учётом `strictNullChecks`\)
- Когда входные типы библиотеки слишком узкие, а библиотека не добавила соответствующих метаданных для Angular.
  Обычно это происходит с отключёнными или другими распространёнными булевыми входными параметрами, используемыми как атрибуты, например `<input disabled>`.

- При использовании `$event.target` для DOM-событий \(из-за возможности всплытия событий `$event.target` в типах DOM не имеет того типа, который можно ожидать\)

В случае подобных ложноположительных срабатываний есть несколько вариантов:

- Используйте функцию приведения типов `$any()` в определённых контекстах для отказа от проверки типов для части выражения
- Отключите строгие проверки полностью, установив `strictTemplates: false` в файле конфигурации TypeScript приложения `tsconfig.json`
- Отключите определённые операции проверки типов по отдельности, сохраняя строгость в других аспектах, установив _флаг строгости_ в `false`
- Если нужно использовать `strictTemplates` и `strictNullChecks` вместе, откажитесь от строгой проверки нулевых типов для входных привязок с помощью `strictNullInputTypes`

Если не указано иное, каждый следующий параметр устанавливается в значение для `strictTemplates` \(`true` при `strictTemplates: true` и наоборот\).

| Флаг строгости               | Эффект                                                                                                                                                                                                                                                                                                                                                                                         |
| :--------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `strictInputTypes`           | Проверяется ли совместимость выражения привязки с полем `@Input()`. Также влияет на вывод обобщённых типов директив.                                                                                                                                                                                                                                                                           |
| `strictInputAccessModifiers` | Учитываются ли модификаторы доступа `private`/`protected`/`readonly` при присваивании выражения привязки `@Input()` или `input()`. При отключении модификаторы доступа входного параметра игнорируются; проверяется только тип. По умолчанию `false`, даже при `strictTemplates: true`. Примечание: эта проверка применяется только к входным параметрам, но не к выходным. |
| `strictNullInputTypes`       | Учитывается ли `strictNullChecks` при проверке привязок `@Input()` \(согласно `strictInputTypes`\). Отключение полезно при использовании библиотеки, не написанной с учётом `strictNullChecks`.                                                                                                                                                                                                |
| `strictAttributeTypes`       | Проверять ли привязки `@Input()`, создаваемые с использованием текстовых атрибутов. Например, `<input matInput disabled="true">` \(установка свойства `disabled` в строку `'true'`\) против `<input matInput [disabled]="true">` \(установка свойства `disabled` в булево `true`\).                                                                                                            |
| `strictSafeNavigationTypes`  | Будет ли возвращаемый тип операций безопасной навигации корректно выведен \(например, `user?.name` будет корректно выведен на основе типа `user`\). При отключении `user?.name` будет типа `any`.                                                                                                                                                                                               |
| `strictDomLocalRefTypes`     | Будут ли локальные ссылки на DOM-элементы иметь правильный тип. При отключении `ref` будет типа `any` для `<input #ref>`.                                                                                                                                                                                                                                                                      |
| `strictOutputEventTypes`     | Будет ли `$event` иметь правильный тип для привязок событий к `@Output()` компонента/директивы или к анимационным событиям. При отключении будет `any`.                                                                                                                                                                                                                                       |
| `strictDomEventTypes`        | Будет ли `$event` иметь правильный тип для привязок событий к DOM-событиям. При отключении будет `any`.                                                                                                                                                                                                                                                                                       |
| `strictContextGenerics`      | Будут ли параметры типа обобщённых компонентов корректно выводиться \(включая обобщённые ограничения\). При отключении все параметры типа будут `any`.                                                                                                                                                                                                                                         |
| `strictLiteralTypes`         | Будет ли тип объектных и массивных литералов, объявленных в шаблоне, выводиться. При отключении тип таких литералов будет `any`. Этот флаг `true` при установке `fullTemplateTypeCheck` или `strictTemplates` в `true`.                                                                                                                                                                        |

Если проблемы сохраняются после работы с этими флагами, вернитесь к полному режиму, отключив `strictTemplates`.

Если это не помогает, крайним вариантом является полное отключение полного режима с помощью `fullTemplateTypeCheck: false`.

Ошибка проверки типов, которую невозможно устранить ни одним из рекомендованных методов, может быть следствием ошибки в самом инструменте проверки типов шаблонов.
Если ошибки требуют возврата к базовому режиму, это, скорее всего, такая ошибка.
В этом случае [создайте issue](https://github.com/angular/angular/issues), чтобы команда могла её исправить.

## Входные параметры и проверка типов {#inputs-and-type-checking}

Инструмент проверки типов шаблонов проверяет, совместим ли тип выражения привязки с типом соответствующего входного параметра директивы.
В качестве примера рассмотрим следующий компонент:

```angular-ts
export interface User {
  name: string;
}

@Component({
  selector: 'user-detail',
  template: '{{ user.name }}',
})
export class UserDetailComponent {
  user = input.required<User>();
}
```

Шаблон `AppComponent` использует этот компонент следующим образом:

```angular-ts
@Component({
  selector: 'app-root',
  template: '<user-detail [user]="selectedUser"></user-detail>',
})
export class AppComponent {
  selectedUser: User | null = null;
}
```

Здесь при проверке типов шаблона `AppComponent` привязка `[user]="selectedUser"` соответствует входному параметру `UserDetailComponent.user`.
Поэтому Angular присваивает свойство `selectedUser` параметру `UserDetailComponent.user`, что приведёт к ошибке при несовместимости их типов.
TypeScript проверяет присваивание в соответствии со своей системой типов, соблюдая такие флаги, как `strictNullChecks`, настроенные в приложении.

Избегайте ошибок типов во время выполнения, предоставляя более конкретные требования к типам в шаблоне инструменту проверки типов.
Делайте требования к типам входных параметров своих директив как можно более конкретными, предоставляя функции защиты шаблонов в определении директивы.
См. [Улучшение проверки типов шаблона для пользовательских директив](/guide/directives/structural-directives#improving-template-type-checking-for-custom-directives) в этом руководстве.

### Строгая проверка null {#strict-null-checks}

При включении `strictTemplates` и флага TypeScript `strictNullChecks` могут возникать ошибки проверки типов для определённых ситуаций, которые трудно избежать.
Например:

- Значение, допускающее null, привязанное к директиве из библиотеки, в которой не был включён `strictNullChecks`.

  Для библиотеки, скомпилированной без `strictNullChecks`, её файлы объявлений не будут указывать, может ли поле быть `null` или нет.
  Для ситуаций, когда библиотека корректно обрабатывает `null`, это проблематично, поскольку компилятор будет проверять значение, допускающее null, по файлам объявлений, в которых тип `null` отсутствует.
  В результате компилятор выдаёт ошибку проверки типов, поскольку соблюдает `strictNullChecks`.

- Использование канала `async` с Observable, который, как известно, генерирует значения синхронно.

  Канал `async` в настоящее время предполагает, что Observable, на который он подписывается, может быть асинхронным, то есть значение ещё может быть недоступно.
  В этом случае он всё равно должен что-то вернуть — `null`.
  Иными словами, возвращаемый тип канала `async` включает `null`, что может привести к ошибкам в ситуациях, когда Observable, как известно, синхронно генерирует ненулевое значение.

Есть два потенциальных обходных пути для указанных проблем:

- В шаблоне включите оператор утверждения ненулевого типа `!` в конце выражения, допускающего null:

```html
<user-detail [user]="user!"></user-detail>
```

В этом примере компилятор игнорирует несовместимость типов по нулевому значению, как и в коде TypeScript.
В случае канала `async` обратите внимание, что выражение нужно обернуть в скобки:

```html
<user-detail [user]="(user$ | async)!"></user-detail>
```

- Полностью отключить строгую проверку null в шаблонах Angular.

  При включённом `strictTemplates` всё равно можно отключить определённые аспекты проверки типов.
  Установка параметра `strictNullInputTypes` в `false` отключает строгую проверку null в шаблонах Angular.
  Этот флаг применяется ко всем компонентам приложения.

### Рекомендации для авторов библиотек {#advice-for-library-authors}

Как автор библиотеки, вы можете принять ряд мер для обеспечения оптимального опыта для пользователей.
Во-первых, включение `strictNullChecks` и включение `null` в тип входного параметра, если это уместно, сообщает потребителям, могут ли они передавать значение, допускающее null, или нет.
Кроме того, можно предоставлять подсказки типов, специфичные для инструмента проверки типов шаблонов.
См. [Улучшение проверки типов шаблона для пользовательских директив](/guide/directives/structural-directives#improving-template-type-checking-for-custom-directives) и [Приведение типов сеттера входного параметра](#input-setter-coercion).

## Приведение типов сеттера входного параметра {#input-setter-coercion}

Иногда желательно, чтобы свойство `input()` директивы или компонента изменяло привязанное к нему значение, обычно с помощью функции `transform` для входного параметра.
В качестве примера рассмотрим этот пользовательский компонент кнопки:

Рассмотрим следующую директиву:

```angular-ts
@Component({
  selector: 'submit-button',
  template: `
    <div class="wrapper">
      <button [disabled]="disabled">Submit</button>
    </div>
  `,
})
class SubmitButton {
  disabled = input.required({transform: booleanAttribute});
}
```

Здесь входной параметр `disabled` компонента передаётся в `<button>` в шаблоне.
Всё работает корректно, пока к входному параметру привязывается значение `boolean`.
Но предположим, что потребитель использует этот входной параметр в шаблоне как атрибут:

```html
<submit-button disabled></submit-button>
```

Это эквивалентно привязке:

```html
<submit-button [disabled]="''"></submit-button>
```

Во время выполнения входной параметр будет установлен в пустую строку, которая не является булевым значением.
Angular-библиотеки компонентов, решающие эту проблему, часто «приводят» значение к правильному типу в сеттере:

```ts

set disabled(value: boolean) {
  this._disabled = (value === '') || value;
}

```

Идеально было бы изменить тип `value` здесь с `boolean` на `boolean|''`, чтобы соответствовать набору значений, фактически принимаемых сеттером.
TypeScript до версии 4.3 требует, чтобы геттер и сеттер имели одинаковый тип, поэтому если геттер должен возвращать `boolean`, сеттер ограничен более узким типом.

Если у потребителя включена самая строгая проверка типов Angular для шаблонов, это создаёт проблему: пустая строка \(`''`\) фактически не может быть присвоена полю `disabled`, что создаёт ошибку типа при использовании атрибутного синтаксиса.

В качестве обходного пути для этой проблемы Angular поддерживает проверку более широкого, более разрешительного типа для `@Input()`, чем объявлен для самого поля входного параметра.
Включите это, добавив статическое свойство с префиксом `ngAcceptInputType_` к классу компонента:

```ts
class SubmitButton {
  private _disabled: boolean;

  @Input()
  get disabled(): boolean {
    return this._disabled;
  }

  set disabled(value: boolean) {
    this._disabled = value === '' || value;
  }

  static ngAcceptInputType_disabled: boolean | '';
}
```

Начиная с TypeScript 4.3 сеттер можно объявить принимающим `boolean|''` как тип, что делает поле приведения типов сеттера входного параметра устаревшим.
Поэтому поля приведения типов сеттеров входных параметров были помечены как устаревшие.

Это поле не нуждается в значении.
Его существование сообщает инструменту проверки типов Angular, что входной параметр `disabled` должен считаться принимающим привязки, соответствующие типу `boolean|''`.
Суффиксом должно быть имя _поля_ `@Input`.

Следует убедиться, что при наличии переопределения `ngAcceptInputType_` для данного входного параметра сеттер должен уметь обрабатывать любые значения переопределённого типа.

## Отключение проверки типов с помощью `$any()` {#disabling-type-checking-using-any}

Отключите проверку выражения привязки, обернув выражение в вызов псевдофункции приведения типов `$any()`.
Компилятор обрабатывает это как приведение к типу `any`, аналогично использованию `<any>` или `as any` в TypeScript.

В следующем примере приведение `person` к типу `any` подавляет ошибку `Property address does not exist`.

```angular-ts
@Component({
  selector: 'my-component',
  template: '{{$any(person).address.street}}',
})
class MyComponent {
  person?: Person;
}
```
