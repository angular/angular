# Создание библиотек {#creating-libraries}

На этой странице дан концептуальный обзор того, как создавать и публиковать новые библиотеки для расширения функциональности Angular.

Если вы обнаруживаете, что вам нужно решить одну и ту же проблему в нескольких приложениях (или хотите поделиться решением с другими разработчиками), у вас есть кандидат на создание библиотеки.
Простой пример — кнопка, направляющая пользователей на сайт вашей компании, которую нужно включить во все приложения, создаваемые компанией.

## Начало работы {#getting-started}

Используйте Angular CLI для генерации нового скелета библиотеки в новом рабочем пространстве с помощью следующих команд.

```shell

ng new my-workspace --no-create-application
cd my-workspace
ng generate library my-lib

```

<docs-callout title="Именование библиотеки">

Следует тщательно выбирать имя библиотеки, если вы планируете впоследствии опубликовать её в публичном реестре пакетов, например npm.
См. [Публикация библиотеки](tools/libraries/creating-libraries#publishing-your-library).

Избегайте использования имени с префиксом `ng-`, например `ng-library`.
Префикс `ng-` является зарезервированным ключевым словом Angular-фреймворка и его библиотек.
Предпочтительным является префикс `ngx-` как соглашение, указывающее, что библиотека подходит для использования с Angular.
Это также отличный ориентир для пользователей реестра, позволяющий различать библиотеки разных JavaScript-фреймворков.

</docs-callout>

Команда `ng generate` создаёт папку `projects/my-lib` в вашем рабочем пространстве, содержащую компонент.

HELPFUL: Подробнее о структуре проекта библиотеки см. в разделе [Файлы проекта библиотеки](reference/configs/file-structure#library-project-files) руководства [Структура файлов проекта](reference/configs/file-structure).

Используйте модель monorepo для работы с несколькими проектами в одном рабочем пространстве.
См. [Настройка рабочего пространства с несколькими проектами](reference/configs/file-structure#multiple-projects).

При генерации новой библиотеки файл конфигурации рабочего пространства `angular.json` обновляется с добавлением проекта типа `library`.

```json

"projects": {
  …
  "my-lib": {
    "root": "projects/my-lib",
    "sourceRoot": "projects/my-lib/src",
    "projectType": "library",
    "prefix": "lib",
    "architect": {
      "build": {
        "builder": "@angular-devkit/build-angular:ng-packagr",
        …

```

Собирайте, тестируйте и линтируйте проект с помощью команд CLI:

```shell

ng build my-lib --configuration development
ng test my-lib
ng lint my-lib

```

Обратите внимание, что настроенный builder для проекта отличается от builder'а по умолчанию для проектов приложений.
Этот builder, в числе прочего, гарантирует, что библиотека всегда собирается с [AOT-компилятором](tools/cli/aot-compiler).

Чтобы код библиотеки был повторно используемым, необходимо определить для неё публичный API.
Этот "пользовательский слой" определяет, что доступно потребителям вашей библиотеки.
Пользователь вашей библиотеки должен иметь возможность получать доступ к публичной функциональности (такой как провайдеры сервисов и общие утилитарные функции) через единственный путь импорта.

Публичный API вашей библиотеки поддерживается в файле `public-api.ts` в папке библиотеки.
Всё, что экспортируется из этого файла, становится публичным при импорте библиотеки в приложение.

Ваша библиотека должна содержать документацию (как правило, файл README) по установке и обслуживанию.

## Рефакторинг частей приложения в библиотеку {#refactoring-parts-of-an-application-into-a-library}

Чтобы сделать решение повторно используемым, необходимо адаптировать его так, чтобы оно не зависело от кода, специфичного для приложения.
Вот некоторые аспекты, которые следует учитывать при переносе функциональности приложения в библиотеку.

- Объявления, такие как компоненты и pipes, должны быть разработаны как stateless, то есть они не должны зависеть от внешних переменных или изменять их.
  Если вы всё же зависите от состояния, необходимо оценить каждый случай и решить, является ли оно состоянием приложения или состоянием, которым будет управлять библиотека.

- Все Observable, на которые компоненты подписываются внутри, должны быть очищены и уничтожены в течение жизненного цикла этих компонентов
- Компоненты должны предоставлять интерфейсы взаимодействия через inputs для предоставления контекста и outputs для передачи событий другим компонентам

- Проверьте все внутренние зависимости.
  - Для пользовательских классов или интерфейсов, используемых в компонентах или сервисах, проверьте, зависят ли они от дополнительных классов или интерфейсов, которые также нужно перенести
  - Аналогично, если код библиотеки зависит от сервиса, этот сервис тоже нужно перенести
  - Если код библиотеки или её шаблоны зависят от других библиотек (например, Angular Material), необходимо настроить библиотеку с этими зависимостями

- Рассмотрите, как вы предоставляете сервисы клиентским приложениям.
  - Сервисы должны объявлять собственных провайдеров, а не объявлять провайдеров в NgModule или компоненте.
    Объявление провайдера делает сервис _tree-shakable_.
    Эта практика позволяет компилятору исключить сервис из бандла, если он никогда не внедряется в приложение, импортирующее библиотеку.
    Подробнее см. в [Tree-shakable провайдеры](guide/di/lightweight-injection-tokens).

  - Если вы регистрируете глобальных провайдеров сервисов, предоставляйте функцию-провайдер `provideXYZ()`.
  - Если ваша библиотека предоставляет опциональные сервисы, которые могут использоваться не всеми клиентскими приложениями, поддерживайте правильный tree-shaking для этого случая, используя [паттерн проектирования с легковесным токеном](guide/di/lightweight-injection-tokens)

## Интеграция с CLI с помощью схематиков генерации кода {#integrating-with-the-cli-using-code-generation-schematics}

Библиотека, как правило, содержит _повторно используемый код_, определяющий компоненты, сервисы и другие артефакты Angular (pipes, директивы), которые вы импортируете в проект.
Библиотека упаковывается в npm-пакет для публикации и совместного использования.
Этот пакет также может включать схематики, предоставляющие инструкции по генерации или трансформации кода непосредственно в вашем проекте — так же, как CLI создаёт новый компонент с помощью `ng generate component`.
Схематик, упакованный с библиотекой, может, например, предоставить Angular CLI информацию, необходимую для генерации компонента, настраивающего и использующего определённую функцию или набор функций, определённых в этой библиотеке.
Один из примеров — [схематик навигации Angular Material](https://material.angular.dev/guide/schematics#navigation-schematic), который настраивает [BreakpointObserver](https://material.angular.dev/cdk/layout/overview#breakpointobserver) CDK и использует его с компонентами Material [MatSideNav](https://material.angular.dev/components/sidenav/overview) и [MatToolbar](https://material.angular.dev/components/toolbar/overview).

Создавайте и включайте следующие виды схематиков:

- Включите схематик установки, чтобы `ng add` мог добавить вашу библиотеку в проект
- Включите схематики генерации в библиотеку, чтобы `ng generate` мог создавать определённые вами артефакты (компоненты, сервисы, тесты) в проекте
- Включите схематик обновления, чтобы `ng update` мог обновлять зависимости вашей библиотеки и предоставлять миграции для критических изменений в новых релизах

Что включать в библиотеку зависит от вашей задачи.
Например, вы можете определить схематик для создания выпадающего списка, предварительно заполненного тестовыми данными, чтобы показать, как добавить его в приложение.
Если вы хотите выпадающий список, который каждый раз содержит разные передаваемые значения, ваша библиотека может определить схематик для его создания с заданной конфигурацией.
Затем разработчики смогут использовать `ng generate` для настройки экземпляра под своё приложение.

Предположим, вы хотите читать файл конфигурации и затем генерировать форму на основе этой конфигурации.
Если форма требует дополнительной настройки разработчиком, использующим библиотеку, лучше всего реализовать её как схематик.
Однако если форма всегда будет одинаковой и не потребует особой настройки, можно создать динамический компонент, принимающий конфигурацию и генерирующий форму.
В целом, чем сложнее настройка, тем полезнее подход с использованием схематика.

Дополнительную информацию см. в [Обзоре схематиков](tools/cli/schematics) и [Схематики для библиотек](tools/cli/schematics-for-libraries).

## Публикация библиотеки {#publishing-your-library}

Используйте Angular CLI и менеджер пакетов npm для сборки и публикации библиотеки в виде npm-пакета.

Angular CLI использует инструмент [ng-packagr](https://github.com/ng-packagr/ng-packagr/blob/master/README.md) для создания пакетов из скомпилированного кода, которые можно публиковать в npm.
См. [Сборка библиотек с Ivy](tools/libraries/creating-libraries#publishing-libraries) для получения информации о форматах дистрибуции, поддерживаемых `ng-packagr`, и рекомендаций по выбору правильного формата.

Для дистрибуции всегда собирайте библиотеки с использованием конфигурации `production`.
Это гарантирует, что генерируемый вывод использует соответствующие оптимизации и правильный формат пакета для npm.

```shell

ng build my-lib
cd dist/my-lib
npm publish

```

## Управление ресурсами в библиотеке {#managing-assets-in-a-library}

В Angular-библиотеке дистрибуция может включать дополнительные ресурсы, такие как файлы тем, Sass-миксины или документацию (например, журнал изменений).
Дополнительную информацию см. в [копировании ресурсов в библиотеку как часть сборки](https://github.com/ng-packagr/ng-packagr/blob/master/docs/copy-assets.md) и [встраивании ресурсов в стили компонентов](https://github.com/ng-packagr/ng-packagr/blob/master/docs/embed-assets-css.md).

IMPORTANT: При включении дополнительных ресурсов, таких как Sass-миксины или предкомпилированный CSS.
Необходимо вручную добавить их в условные ["exports"](tools/libraries/angular-package-format#exports) в `package.json` основной точки входа.

`ng-packagr` объединит рукописные `"exports"` с автоматически сгенерированными, что позволяет авторам библиотек настраивать дополнительные подпути экспорта или пользовательские условия.

```json

"exports": {
  ".": {
    "sass": "./_index.scss",
  },
  "./theming": {
    "sass": "./_theming.scss"
  },
  "./prebuilt-themes/indigo-pink.css": {
    "style": "./prebuilt-themes/indigo-pink.css"
  }
}

```

Выше приведён фрагмент из дистрибутива [@angular/material](https://unpkg.com/browse/@angular/material/package.json).

## Зависимости peer {#peer-dependencies}

Angular-библиотеки должны перечислять любые зависимости `@angular/*`, от которых зависит библиотека, в качестве peer dependencies.
Это гарантирует, что при запросе Angular всеми модулями они получат один и тот же модуль.
Если библиотека перечисляет `@angular/core` в `dependencies` вместо `peerDependencies`, она может получить другой модуль Angular, что приведёт к поломке приложения.

## Использование собственной библиотеки в приложениях {#using-your-own-library-in-applications}

Не обязательно публиковать библиотеку в менеджер пакетов npm, чтобы использовать её в том же рабочем пространстве, но её необходимо сначала собрать.

Для использования собственной библиотеки в приложении:

- Соберите библиотеку.
  Библиотеку нельзя использовать до её сборки.

```shell
  ng build my-lib
```

- В приложениях импортируйте из библиотеки по имени:

```ts
import {myExport} from 'my-lib';
```

### Сборка и пересборка библиотеки {#building-and-rebuilding-your-library}

Этап сборки важен, если вы не публиковали библиотеку как npm-пакет и не устанавливали пакет обратно в приложение из npm.
Например, если вы клонируете репозиторий git и выполняете `npm install`, редактор показывает импорты `my-lib` как отсутствующие, если вы ещё не собрали библиотеку.

HELPFUL: При импорте чего-либо из библиотеки в Angular-приложение Angular ищет сопоставление между именем библиотеки и местом на диске.
При установке пакета библиотеки сопоставление находится в папке `node_modules`.
При сборке собственной библиотеки она должна найти сопоставление в путях `tsconfig`.

Генерация библиотеки с помощью Angular CLI автоматически добавляет её путь в файл `tsconfig`.
Angular CLI использует пути `tsconfig`, чтобы сообщить системе сборки, где найти библиотеку.

Дополнительную информацию см. в [Обзоре сопоставления путей](https://www.typescriptlang.org/docs/handbook/module-resolution.html#path-mapping).

Вы можете пересобирать библиотеку при каждом внесении изменений, но этот дополнительный шаг занимает время.
Функциональность _инкрементальной сборки_ улучшает опыт разработки библиотек.
Каждый раз при изменении файла выполняется частичная сборка, генерирующая изменённые файлы.

Инкрементальные сборки можно запускать как фоновый процесс в среде разработки.
Чтобы воспользоваться этой функцией, добавьте флаг `--watch` к команде сборки:

```shell

ng build my-lib --watch

```

IMPORTANT: Команда CLI `build` использует другой builder и вызывает другой инструмент сборки для библиотек, чем для приложений.

- Система сборки для приложений, `@angular-devkit/build-angular`, основана на `webpack` и включена во все новые Angular CLI-проекты
- Система сборки для библиотек основана на `ng-packagr`.
  Она добавляется в зависимости только при добавлении библиотеки с помощью `ng generate library my-lib`.

Обе системы сборки поддерживают разные вещи, и даже там, где они поддерживают одно и то же, делают это по-разному.
Это означает, что TypeScript-исходник может давать разный JavaScript-код в собранной библиотеке и собранном приложении.

По этой причине приложение, зависящее от библиотеки, должно использовать только сопоставления путей TypeScript, указывающие на _собранную библиотеку_.
Сопоставления путей TypeScript _не_ должны указывать на исходные файлы `.ts` библиотеки.

### Линковка библиотек для локальной разработки {#linking-libraries-for-local-development}

В этом разделе объясняется, как использовать функцию локальной линковки менеджера пакетов
(например, [`npm link`](https://docs.npmjs.com/cli/v11/commands/npm-link) или [`pnpm link`](https://pnpm.io/cli/link)) для тестирования standalone Angular-библиотеки с внешним приложением во время
локальной разработки, не полагаясь на структуру monorepo или публикацию в реестр NPM.

NOTE: Если ваша библиотека и приложение находятся в одном Angular-рабочем пространстве (monorepo), стандартный рабочий процесс monorepo автоматически обрабатывает линковку и, как правило, более эффективен. Этот подход локальной линковки лучше всего подходит, когда:

- Вы разрабатываете standalone-библиотеку и нуждаетесь в тестировании изменений с внешним потребляющим приложением.
- Вы тестируете изменения библиотеки в потребляющем приложении за пределами monorepo.

#### Настройка потребляющего приложения {#configuring-the-consuming-application}

Для использования слинкованных библиотек необходимо настроить файл `angular.json` приложения со следующими параметрами:

```json
{
  "projects": {
    "your-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "preserveSymlinks": true
          },
          "configurations": {
            "development": {
              "sourceMap": {
                "scripts": true,
                "styles": true,
                "vendor": true
              }
            }
          }
        },
        "serve": {
          "builder": "@angular-devkit/build-angular:dev-server",
          "options": {
            "prebundle": {
              "exclude": ["my-lib"]
            }
          }
        }
      }
    }
  }
}
```

**Пояснение параметров конфигурации:**

- `preserveSymlinks: true`: Указывает системе сборки следовать символическим ссылкам, созданным командой линковки менеджера пакетов, вместо того чтобы разрешать их к исходному расположению. Это необходимо для избежания множественных копий зависимых node-пакетов.
- `sourceMap.vendor`: Включение vendor source maps (особенно `vendor: true`) для более удобной отладки кода слинкованной библиотеки.
- `prebundle.exclude`: По умолчанию Angular CLI может предварительно бандлировать все node-зависимости. Исключение вашей библиотеки гарантирует, что слинкованный исходный код правильно отслеживается и пересобирается при изменениях.

## Публикация библиотек {#publishing-libraries}

Есть два формата дистрибуции для публикации библиотеки:

| Форматы дистрибуции          | Подробности                                                                                                                                                                                                                                                                                                                                      |
| :---------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Partial-Ivy (рекомендуется)   | Содержит переносимый код, который может потребляться Ivy-приложениями, собранными с любой версией Angular начиная с v12.                                                                                                                                                                                                                         |
| Full-Ivy                      | Содержит приватные инструкции Angular Ivy, которые не гарантированно работают в разных версиях Angular. Этот формат требует, чтобы библиотека и приложение были собраны с _точно одинаковой_ версией Angular. Этот формат полезен в средах, где весь код библиотек и приложений собирается непосредственно из исходников.                         |

Для публикации в npm используйте формат partial-Ivy, так как он стабилен между патч-версиями Angular.

Избегайте компиляции библиотек с полным кодом Ivy при публикации в npm, поскольку генерируемые инструкции Ivy не являются частью публичного API Angular и могут меняться между патч-версиями.

## Обеспечение совместимости версий библиотеки {#ensuring-library-version-compatibility}

Версия Angular, используемая для сборки приложения, всегда должна быть равна или выше версий Angular, используемых для сборки любых зависимых библиотек.
Например, если у вас есть библиотека, использующая Angular версии 13, приложение, зависящее от этой библиотеки, должно использовать Angular версии 13 или выше.
Angular не поддерживает использование более ранней версии для приложения.

Если вы намерены публиковать библиотеку в npm, компилируйте с кодом partial-Ivy, установив `"compilationMode": "partial"` в `tsconfig.prod.json`.
Этот частичный формат стабилен между разными версиями Angular, поэтому его безопасно публиковать в npm.
Код в этом формате обрабатывается во время сборки приложения с использованием той же версии Angular-компилятора, что гарантирует использование приложением и всеми его библиотеками единственной версии Angular.

Избегайте компиляции библиотек с полным кодом Ivy при публикации в npm, поскольку генерируемые инструкции Ivy не являются частью публичного API Angular и могут меняться между патч-версиями.

Если вы никогда не публиковали пакет в npm, необходимо создать учётную запись пользователя.
Подробнее см. в [Публикация npm-пакетов](https://docs.npmjs.com/getting-started/publishing-npm-packages).

## Использование кода partial-Ivy вне Angular CLI {#consuming-partial-ivy-code-outside-the-angular-cli}

Приложение устанавливает множество Angular-библиотек из npm в свою директорию `node_modules`.
Однако код в этих библиотеках не может быть напрямую бандлирован вместе с собранным приложением, поскольку он не полностью скомпилирован.
Для завершения компиляции используйте Angular linker.

Для приложений, не использующих Angular CLI, linker доступен как плагин [Babel](https://babeljs.io).
Плагин импортируется из `@angular/compiler-cli/linker/babel`.

Плагин Angular linker Babel поддерживает кэширование сборки, что означает, что библиотеки нужно обрабатывать linker'ом только один раз, независимо от других npm-операций.

Пример интеграции плагина в пользовательскую сборку [webpack](https://webpack.js.org) путём регистрации linker как плагина [Babel](https://babeljs.io) с использованием [babel-loader](https://webpack.js.org/loaders/babel-loader/#options).
