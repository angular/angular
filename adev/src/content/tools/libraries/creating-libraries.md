# Создание библиотек {#creating-libraries}

На этой странице представлен концептуальный обзор того, как создавать и публиковать новые библиотеки для расширения функциональности Angular.

Если вы обнаруживаете, что одну и ту же проблему нужно решать в нескольких приложениях \(или хотите поделиться решением с другими разработчиками\), у вас есть кандидат для библиотеки.
Простым примером может быть кнопка, перенаправляющая пользователей на сайт компании, которую можно включать во все приложения компании.

## Начало работы {#getting-started}

Используйте Angular CLI для создания нового скелета библиотеки в новом рабочем пространстве с помощью следующих команд.

```shell

ng new my-workspace --no-create-application
cd my-workspace
ng generate library my-lib

```

<docs-callout title="Именование библиотеки">

При выборе имени библиотеки следует быть очень осторожными, если планируется публикация в публичном реестре пакетов, например npm.
См. [Публикация библиотеки](tools/libraries/creating-libraries#publishing-your-library).

Избегайте имён с префиксом `ng-`, например `ng-library`.
Префикс `ng-` является зарезервированным ключевым словом для Angular-фреймворка и его библиотек.
Предпочтительно использовать префикс `ngx-` как соглашение, указывающее на то, что библиотека предназначена для использования с Angular.
Это также отличный сигнал для потребителей реестра для различения библиотек разных JavaScript-фреймворков.

</docs-callout>

Команда `ng generate` создаёт папку `projects/my-lib` в рабочем пространстве, содержащую компонент.

HELPFUL: Подробнее о структуре проекта библиотеки см. в разделе [Файлы проекта библиотеки](reference/configs/file-structure#library-project-files) руководства по [Структуре файлов проекта](reference/configs/file-structure).

Используйте монорепо-модель для работы с несколькими проектами в одном рабочем пространстве.
См. [Настройка рабочего пространства с несколькими проектами](reference/configs/file-structure#multiple-projects).

При генерации новой библиотеки файл конфигурации рабочего пространства `angular.json` обновляется с добавлением проекта типа `library`.

```json

"projects": {
  …
  "my-lib": {
    "root": "projects/my-lib",
    "sourceRoot": "projects/my-lib/src",
    "projectType": "library",
    "prefix": "lib",
    "architect": {
      "build": {
        "builder": "@angular-devkit/build-angular:ng-packagr",
        …

```

Сборка, тестирование и линтинг проекта с помощью команд CLI:

```shell

ng build my-lib --configuration development
ng test my-lib
ng lint my-lib

```

Обратите внимание, что настроенный Builder для проекта отличается от Builder по умолчанию для проектов приложений.
Этот Builder, помимо прочего, обеспечивает, что библиотека всегда собирается с помощью [AOT-компилятора](tools/cli/aot-compiler).

Чтобы код библиотеки можно было переиспользовать, необходимо определить для него публичный API.
Этот «пользовательский слой» определяет, что доступно потребителям библиотеки.
Пользователь библиотеки должен иметь доступ к публичным функциям \(таким как провайдеры сервисов и общие утилитные функции\) через единый путь импорта.

Публичный API библиотеки поддерживается в файле `public-api.ts` в папке библиотеки.
Всё, что экспортируется из этого файла, становится публичным при импорте библиотеки в приложение.

Библиотека должна предоставлять документацию \(как правило, README-файл\) для установки и обслуживания.

## Рефакторинг частей приложения в библиотеку {#refactoring-parts-of-an-application-into-a-library}

Для обеспечения возможности повторного использования решения необходимо адаптировать его так, чтобы оно не зависело от специфичного для приложения кода.
При миграции функциональности приложения в библиотеку учитывайте следующее.

- Объявления, такие как компоненты и каналы, должны быть спроектированы как не имеющие состояния, то есть не зависящие от внешних переменных и не изменяющие их.
  При зависимости от состояния необходимо оценивать каждый случай и решать, является ли это состоянием приложения или состоянием, управляемым библиотекой.

- Любые Observable, на которые подписываются компоненты внутренне, должны очищаться и уничтожаться в течение жизненного цикла этих компонентов
- Компоненты должны представлять взаимодействие через входные параметры для предоставления контекста и выходные параметры для передачи событий другим компонентам

- Проверьте все внутренние зависимости.
  - Для пользовательских классов или интерфейсов, используемых в компонентах или сервисах, проверьте, зависят ли они от дополнительных классов или интерфейсов, которые также нужно мигрировать
  - Аналогично, если код библиотеки зависит от сервиса, этот сервис тоже нужно мигрировать
  - Если код библиотеки или её шаблоны зависят от других библиотек \(например, Angular Material\), необходимо настроить библиотеку с этими зависимостями

- Рассмотрите, как предоставлять сервисы клиентским приложениям.
  - Сервисы должны объявлять собственные провайдеры, а не объявлять провайдеры в NgModule или компоненте.
    Объявление провайдера делает сервис _tree-shakable_.
    Эта практика позволяет компилятору исключить сервис из бандла, если он никогда не внедряется в приложение, импортирующее библиотеку.
    Подробнее см. в разделе [Tree-shakable провайдеры](guide/di/lightweight-injection-tokens).

  - При регистрации глобальных провайдеров сервисов предоставляйте функцию-провайдер `provideXYZ()`.
  - Если библиотека предоставляет опциональные сервисы, которые могут не использоваться всеми клиентскими приложениями, поддерживайте правильный tree-shaking для этого случая, используя [паттерн лёгкого токена](guide/di/lightweight-injection-tokens)

## Интеграция с CLI с помощью схематиков генерации кода {#integrating-with-the-cli-using-code-generation-schematics}

Библиотека обычно включает _переиспользуемый код_, определяющий компоненты, сервисы и другие Angular-артефакты \(каналы, директивы\), которые вы импортируете в проект.
Библиотека упаковывается в npm-пакет для публикации и совместного использования.
Этот пакет также может включать схематики, предоставляющие инструкции по генерации или преобразованию кода непосредственно в проекте, аналогично тому, как CLI создаёт новый компонент с помощью `ng generate component`.
Схематик, упакованный с библиотекой, может, например, предоставить Angular CLI информацию, необходимую для генерации компонента, настраивающего и использующего определённую функцию или набор функций из этой библиотеки.
Одним из примеров является [схематик навигации Angular Material](https://material.angular.dev/guide/schematics#navigation-schematic), настраивающий [BreakpointObserver](https://material.angular.dev/cdk/layout/overview#breakpointobserver) CDK и использующий его с компонентами Material [MatSideNav](https://material.angular.dev/components/sidenav/overview) и [MatToolbar](https://material.angular.dev/components/toolbar/overview).

Создавайте и включайте следующие виды схематиков:

- Включите схематик установки, чтобы `ng add` мог добавить библиотеку в проект
- Включите схематики генерации в библиотеку, чтобы `ng generate` мог создавать определённые артефакты \(компоненты, сервисы, тесты\) в проекте
- Включите схематик обновления, чтобы `ng update` мог обновлять зависимости библиотеки и предоставлять миграции для критических изменений в новых выпусках

Содержимое библиотеки зависит от задачи.
Например, можно определить схематик для создания выпадающего списка, предварительно заполненного шаблонными данными, чтобы показать, как добавить его в приложение.
Если нужен выпадающий список с разными значениями при каждом использовании, библиотека может определить схематик для его создания с заданной конфигурацией.
Разработчики смогут использовать `ng generate` для настройки экземпляра под своё приложение.

Предположим, нужно прочитать файл конфигурации, а затем сгенерировать форму на основе этой конфигурации.
Если форма требует дополнительной настройки разработчиком, использующим библиотеку, лучшим вариантом может быть схематик.
Если же форма всегда одинакова и не требует особой настройки, можно создать динамический компонент, принимающий конфигурацию и генерирующий форму.
В общем, чем сложнее настройка, тем полезнее схематик.

Подробнее см. в разделах [Обзор схематиков](tools/cli/schematics) и [Схематики для библиотек](tools/cli/schematics-for-libraries).

## Публикация библиотеки {#publishing-your-library}

Используйте Angular CLI и менеджер пакетов npm для сборки и публикации библиотеки как npm-пакета.

Angular CLI использует инструмент [ng-packagr](https://github.com/ng-packagr/ng-packagr/blob/master/README.md) для создания пакетов из скомпилированного кода, которые можно публиковать в npm.
Информацию о форматах дистрибутива, поддерживаемых `ng-packagr`, и руководство по выбору правильного формата для библиотеки см. в разделе [Сборка библиотек с Ivy](tools/libraries/creating-libraries#publishing-libraries).

Всегда собирайте библиотеки для дистрибутива с использованием конфигурации `production`.
Это гарантирует использование соответствующих оптимизаций и правильного формата пакета для npm.

```shell

ng build my-lib
cd dist/my-lib
npm publish

```

## Управление ресурсами в библиотеке {#managing-assets-in-a-library}

В Angular-библиотеке распространяемый пакет может включать дополнительные ресурсы, такие как файлы тем, Sass-миксины или документация \(например, журнал изменений\).
Подробнее о [копировании ресурсов в библиотеку как часть сборки](https://github.com/ng-packagr/ng-packagr/blob/master/docs/copy-assets.md) и [встраивании ресурсов в стили компонентов](https://github.com/ng-packagr/ng-packagr/blob/master/docs/embed-assets-css.md).

IMPORTANT: При включении дополнительных ресурсов, таких как Sass-миксины или предварительно скомпилированный CSS,
их необходимо вручную добавить в условные ["exports"](tools/libraries/angular-package-format#exports) в `package.json` основной точки входа.

`ng-packagr` объединит рукописные `"exports"` с автоматически сгенерированными, позволяя авторам библиотек настраивать дополнительные подпути экспорта или пользовательские условия.

```json

"exports": {
  ".": {
    "sass": "./_index.scss",
  },
  "./theming": {
    "sass": "./_theming.scss"
  },
  "./prebuilt-themes/indigo-pink.css": {
    "style": "./prebuilt-themes/indigo-pink.css"
  }
}

```

Приведённое выше — это фрагмент из распространяемого пакета [@angular/material](https://unpkg.com/browse/@angular/material/package.json).

## Одноранговые зависимости {#peer-dependencies}

Angular-библиотеки должны перечислять все зависимости `@angular/*`, от которых зависит библиотека, как одноранговые зависимости.
Это гарантирует, что при запросе Angular модулями все они получают один и тот же модуль.
Если библиотека перечисляет `@angular/core` в `dependencies` вместо `peerDependencies`, она может получить другой Angular-модуль, что нарушит работу приложения.

## Использование собственной библиотеки в приложениях {#using-your-own-library-in-applications}

Для использования библиотеки в приложении в том же рабочем пространстве не обязательно публиковать её в npm, но сначала нужно выполнить её сборку.

Для использования собственной библиотеки в приложении:

- Соберите библиотеку.
  Нельзя использовать библиотеку до её сборки.

```shell
  ng build my-lib
```

- В приложении импортируйте из библиотеки по имени:

```ts
import {myExport} from 'my-lib';
```

### Сборка и пересборка библиотеки {#building-and-rebuilding-your-library}

Шаг сборки важен, если библиотека не была опубликована как npm-пакет и затем установлена обратно в приложение из npm.
Например, при клонировании репозитория git и запуске `npm install`, редактор покажет импорты `my-lib` как отсутствующие, если библиотека ещё не собрана.

HELPFUL: При импорте чего-либо из библиотеки в Angular-приложении Angular ищет маппинг между именем библиотеки и расположением на диске.
При установке пакета библиотеки маппинг находится в папке `node_modules`.
При сборке собственной библиотеки маппинг должен находиться в путях `tsconfig`.

Генерация библиотеки с помощью Angular CLI автоматически добавляет её путь в файл `tsconfig`.
Angular CLI использует пути `tsconfig`, чтобы сообщить системе сборки, где найти библиотеку.

Подробнее см. в разделе [Обзор маппинга путей](https://www.typescriptlang.org/docs/handbook/module-resolution.html#path-mapping).

Библиотеку можно пересобирать при каждом изменении, но этот дополнительный шаг занимает время.
Функциональность _инкрементальных сборок_ улучшает опыт разработки библиотеки.
При каждом изменении файла выполняется частичная сборка, генерирующая изменённые файлы.

Инкрементальные сборки можно запускать как фоновый процесс в среде разработки.
Для использования этой функции добавьте флаг `--watch` к команде сборки:

```shell

ng build my-lib --watch

```

IMPORTANT: Команда `build` CLI использует другой Builder и вызывает другой инструмент сборки для библиотек по сравнению с приложениями.

- Система сборки для приложений, `@angular-devkit/build-angular`, основана на `webpack` и включена во все новые проекты Angular CLI
- Система сборки для библиотек основана на `ng-packagr`.
  Она добавляется в зависимости только при добавлении библиотеки с помощью `ng generate library my-lib`.

Две системы сборки поддерживают разные возможности, и даже там, где они поддерживают одно и то же, делают это по-разному.
Это означает, что TypeScript-исходный код может давать разный JavaScript-код в собранной библиотеке по сравнению со собранным приложением.

По этой причине приложение, зависящее от библиотеки, должно использовать только маппинги путей TypeScript, указывающие на _собранную библиотеку_.
Маппинги путей TypeScript _не должны_ указывать на исходные `.ts` файлы библиотеки.

### Связывание библиотек для локальной разработки {#linking-libraries-for-local-development}

В этом разделе объясняется, как использовать функцию локального связывания менеджера пакетов
(например, [`npm link`](https://docs.npmjs.com/cli/v11/commands/npm-link) или [`pnpm link`](https://pnpm.io/cli/link)) для тестирования отдельной Angular-библиотеки с внешним приложением
в процессе локальной разработки, без использования монорепо-структуры рабочего пространства или публикации в реестре NPM.

NOTE: Если библиотека и приложение находятся в одном рабочем пространстве Angular (монорепо), стандартный монорепо-рабочий процесс автоматически обрабатывает связывание и, как правило, более эффективен. Подход с локальным связыванием лучше всего подходит, когда:

- Разрабатывается отдельная библиотека и нужно тестировать изменения с внешним потребляющим приложением.
- Тестируются изменения библиотеки в потребляющем приложении за пределами монорепо-рабочего пространства.

#### Настройка потребляющего приложения {#configuring-the-consuming-application}

Для использования связанных библиотек необходимо настроить файл `angular.json` приложения со следующими параметрами:

```json
{
  "projects": {
    "your-app": {
      "architect": {
        "build": {
          "builder": "@angular/build:application",
          "options": {
            "preserveSymlinks": true
          },
          "configurations": {
            "development": {
              "sourceMap": {
                "scripts": true,
                "styles": true,
                "vendor": true
              }
            }
          }
        },
        "serve": {
          "builder": "@angular-devkit/build-angular:dev-server",
          "options": {
            "prebundle": {
              "exclude": ["my-lib"]
            }
          }
        }
      }
    }
  }
}
```

**Пояснение параметров конфигурации:**

- `preserveSymlinks: true`: Указывает системе сборки следовать символическим ссылкам, созданным командой связывания менеджера пакетов, вместо разрешения к исходному расположению символической ссылки. Это необходимо для предотвращения множественных копий зависимых node-пакетов.
- `sourceMap.vendor`: Включение source map вендора \(особенно `vendor: true`\) для упрощения отладки кода связанной библиотеки.
- `prebundle.exclude`: По умолчанию Angular CLI может предварительно объединять все node-зависимости. Исключение библиотеки гарантирует правильное отслеживание связанного исходного кода и его пересборку при изменениях.

## Публикация библиотек {#publishing-libraries}

Существует два формата дистрибутива при публикации библиотеки:

| Форматы дистрибутива        | Описание                                                                                                                                                                                                                                                                                                                                  |
| :-------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Partial-Ivy \(рекомендуется\) | Содержит переносимый код, который могут потреблять Ivy-приложения, собранные с любой версией Angular от v12 и выше.                                                                                                                                                                                                                     |
| Full-Ivy                    | Содержит приватные инструкции Angular Ivy, не гарантирующие работу в разных версиях Angular. Этот формат требует, чтобы библиотека и приложение были собраны с _точно одной и той же_ версией Angular. Этот формат полезен в средах, где все библиотеки и код приложения собираются непосредственно из источника. |

Для публикации в npm используйте формат partial-Ivy, так как он стабилен между патч-версиями Angular.

Избегайте компиляции библиотек с полным Ivy-кодом при публикации в npm, поскольку генерируемые Ivy-инструкции не являются частью публичного API Angular и могут меняться между патч-версиями.

## Обеспечение совместимости версий библиотеки {#ensuring-library-version-compatibility}

Версия Angular, используемая для сборки приложения, всегда должна быть той же или большей, чем версия Angular, используемая для сборки всех зависимых библиотек.
Например, если библиотека использует Angular версии 13, приложение, зависящее от неё, должно использовать Angular версии 13 или выше.
Angular не поддерживает использование более ранней версии для приложения.

Если планируется публикация библиотеки в npm, компилируйте с частичным Ivy-кодом, установив `"compilationMode": "partial"` в `tsconfig.prod.json`.
Этот частичный формат стабилен между разными версиями Angular, поэтому его безопасно публиковать в npm.
Код в этом формате обрабатывается во время сборки приложения с использованием той же версии компилятора Angular, гарантируя, что приложение и все его библиотеки используют единую версию Angular.

Избегайте компиляции библиотек с полным Ivy-кодом при публикации в npm, поскольку генерируемые Ivy-инструкции не являются частью публичного API Angular и могут меняться между патч-версиями.

Если вы никогда не публиковали пакет в npm, необходимо создать учётную запись пользователя.
Подробнее см. в разделе [Публикация npm-пакетов](https://docs.npmjs.com/getting-started/publishing-npm-packages).

## Использование кода частичного Ivy вне Angular CLI {#consuming-partial-ivy-code-outside-the-angular-cli}

Приложение устанавливает множество Angular-библиотек из npm в директорию `node_modules`.
Однако код в этих библиотеках не может быть напрямую объединён в бандл вместе с собранным приложением, поскольку он не полностью скомпилирован.
Для завершения компиляции используйте Angular linker.

Для приложений, не использующих Angular CLI, linker доступен как [Babel](https://babeljs.io)-плагин.
Плагин импортируется из `@angular/compiler-cli/linker/babel`.

Angular linker Babel-плагин поддерживает кэширование сборки, то есть библиотеки нужно обрабатывать linker-ом только один раз независимо от других npm-операций.

Пример интеграции плагина в пользовательскую сборку [webpack](https://webpack.js.org) путём регистрации linker как [Babel](https://babeljs.io)-плагина с использованием [babel-loader](https://webpack.js.org/loaders/babel-loader/#options).
