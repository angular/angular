# Формат пакетов Angular {#angular-package-format}

В этом документе описывается формат пакетов Angular \(APF\).
APF — это специфичная для Angular спецификация структуры и формата npm-пакетов, используемая всеми первичными пакетами Angular \(`@angular/core`, `@angular/material` и т. д.\) и большинством сторонних Angular-библиотек.

APF позволяет пакету беспрепятственно работать в большинстве распространённых сценариев использования Angular.
Пакеты, использующие APF, совместимы с инструментами, предлагаемыми командой Angular, а также с более широкой экосистемой JavaScript.
Сторонним разработчикам библиотек рекомендуется следовать тому же формату npm-пакетов.

HELPFUL: APF версионируется вместе с остальным Angular, и каждая мажорная версия улучшает формат пакетов.
Версии спецификации до v13 можно найти в этом [google doc](https://docs.google.com/document/d/1CZC2rcpxffTDfRDs6p1cfbmKNLA6x5O-NtkJglDaBVs/preview).

## Зачем задавать формат пакетов? {#why-specify-a-package-format}

В современном JavaScript-ландшафте разработчики используют пакеты самыми разными способами с различными инструментальными цепочками \(webpack, Rollup, esbuild и т. д.\).
Эти инструменты могут понимать и требовать различных входных данных — одни инструменты могут обрабатывать последнюю версию языка ES, тогда как другим выгоднее напрямую работать со старой версией ES.

Формат дистрибутива Angular поддерживает все широко используемые инструменты и рабочие процессы разработки, делая акцент на оптимизациях, которые приводят либо к меньшему размеру бандла приложения, либо к более быстрому циклу итерации разработки \(времени сборки\).

Разработчики могут полагаться на Angular CLI и [ng-packagr](https://github.com/ng-packagr/ng-packagr) \(инструмент сборки, используемый Angular CLI\) для создания пакетов в формате Angular.
Подробнее см. в руководстве по [созданию библиотек](tools/libraries/creating-libraries).

Следующий пример показывает упрощённую версию структуры файлов пакета `@angular/core` с пояснением для каждого файла в пакете.

```markdown

├── README.md
├── package.json
├── fesm2022
│ ├── core.mjs
│ ├── core.mjs.map
```

| Файлы                                                                                                                                                     | Назначение                                                                                                                                                                                                                        |
| :-------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `README.md`                                                                                                                                               | README пакета, используемый веб-интерфейсом npmjs.                                                                                                                                                                               |
| `package.json`                                                                                                                                            | Основной `package.json`, описывающий сам пакет, а также все доступные точки входа и форматы кода. ## `package.json` {#packagejson}

Основной `package.json` содержит важные метаданные пакета, включая следующее:

- [Объявляет](#esm-declaration) пакет в формате EcmaScript Module \(ESM\)
- Содержит поле `"exports"`, определяющее доступные форматы исходного кода для всех точек входа
- Содержит [ключи](#legacy-resolution-keys), определяющие доступные форматы исходного кода основной точки входа `@angular/core`, для инструментов, не понимающих `"exports"`.
  Эти ключи считаются устаревшими и могут быть удалены по мере распространения поддержки `"exports"` в экосистеме.

- Объявляет, содержит ли пакет [побочные эффекты](#side-effects)

### Объявление ESM {#esm-declaration}

Верхнеуровневый `package.json` содержит ключ:

```js
{
  "type": "module"
}
```

Это сообщает резолверам, что код в пакете использует EcmaScript Modules, а не CommonJS модули.

### `"exports"` {#exports}

Поле `"exports"` имеет следующую структуру:

```js
"exports": {
  "./schematics/*": {
    "default": "./schematics/*.js"
  },
  "./package.json": {
    "default": "./package.json"
  },
  ".": {
    "types": "./types/core.d.ts",
    "default": "./fesm2022/core.mjs"
  },
  "./testing": {
    "types": "./types/testing.d.ts",
    "default": "./fesm2022/testing.mjs"
  }
}
```

Основной интерес представляют ключи `"."` и `"./testing"`, определяющие доступные форматы кода для основной точки входа `@angular/core` и вторичной точки входа `@angular/core/testing` соответственно.
Для каждой точки входа доступны следующие форматы:

| Форматы                    | Описание                                                                       |
| :------------------------- | :----------------------------------------------------------------------------- |
| Типизации \(файлы `.d.ts`\) | Файлы `.d.ts` используются TypeScript при зависимости от данного пакета.      |
| `default`                  | Код ES2022, сглаженный в единый источник.                                      |

Инструменты, понимающие эти ключи, могут предпочтительно выбирать желаемый формат кода из `"exports"`.

Библиотеки могут захотеть предоставлять дополнительные статические файлы, не охваченные экспортами JavaScript-точек входа, например Sass-миксины или предварительно скомпилированный CSS.

Подробнее см. в разделе [Управление ресурсами в библиотеке](tools/libraries/creating-libraries#managing-assets-in-a-library).

### Устаревшие ключи разрешения {#legacy-resolution-keys}

В дополнение к `"exports"`, верхнеуровневый `package.json` также определяет устаревшие ключи разрешения модулей для резолверов, не поддерживающих `"exports"`.
Для `@angular/core` это:

```js
{
  "module": "./fesm2022/core.mjs",
  "typings": "./types/core.d.ts",
}
```

Как показано в предыдущем фрагменте кода, резолвер модулей может использовать эти ключи для загрузки конкретного формата кода.

### Побочные эффекты {#side-effects}

Последняя функция `package.json` — объявить, имеет ли пакет [побочные эффекты](#sideeffects-flag).

```js
{
  "sideEffects": false
}
```

Большинство Angular-пакетов не должны зависеть от побочных эффектов верхнего уровня, и поэтому должны включать это объявление.

## Точки входа и разделение кода {#entrypoints-and-code-splitting}

Пакеты в формате Angular Package содержат одну основную точку входа и ноль или более вторичных точек входа \(например, `@angular/common/http`\).
Точки входа выполняют несколько функций.

1. Они определяют спецификаторы модулей, из которых пользователи импортируют код \(например, `@angular/core` и `@angular/core/testing`\).

   Пользователи обычно воспринимают эти точки входа как отдельные группы символов с разными назначениями или возможностями.

   Конкретные точки входа могут использоваться только для особых целей, например для тестирования.
   Такие API можно выделить из основной точки входа, чтобы снизить вероятность их случайного или неправильного использования.

1. Они определяют степень детализации, с которой код может загружаться лениво.

   Многие современные инструменты сборки способны «разделять код» \(ленивая загрузка\) только на уровне ES Module.
   Формат Angular Package использует преимущественно один «сглаженный» ES Module на точку входа. Это означает, что большинство инструментов сборки не может разбивать код с одной точкой входа на несколько выходных частей.

Общее правило для пакетов APF — использовать точки входа для наименьших наборов логически связанного кода.
Например, пакет Angular Material публикует каждый логический компонент или набор компонентов как отдельную точку входа — одну для Button, одну для Tabs и т. д.
Это позволяет каждому компоненту Material при желании загружаться отдельно лениво.

Не все библиотеки требуют такой детализации.
Большинство библиотек с единой логической целью должны публиковаться как единая точка входа.
Например, `@angular/core` использует единую точку входа для среды выполнения, поскольку Angular runtime обычно используется как единое целое.

### Разрешение вторичных точек входа {#resolution-of-secondary-entry-points}

Вторичные точки входа могут разрешаться через поле `"exports"` файла `package.json` пакета.

## Частичная компиляция {#partial-compilation}

Библиотеки в формате Angular Package должны публиковаться в режиме «частичной компиляции».
Это режим компиляции для `ngc`, создающий скомпилированный Angular-код, не привязанный к конкретной версии Angular runtime, в отличие от полной компиляции, используемой для приложений, где версии компилятора и runtime Angular должны точно совпадать.

Для частичной компиляции Angular-кода используйте флаг `compilationMode` в свойстве `angularCompilerOptions` файла `tsconfig.json`:

```js
{
  …
  "angularCompilerOptions": {
    "compilationMode": "partial",
  }
}
```

Частично скомпилированный код библиотеки затем преобразуется в полностью скомпилированный код в процессе сборки приложения Angular CLI.

Если в конвейере сборки не используется Angular CLI, обратитесь к руководству [Использование кода частичного Ivy вне Angular CLI](tools/libraries/creating-libraries#consuming-partial-ivy-code-outside-the-angular-cli).

## Оптимизации {#optimizations}

### Сглаживание ES-модулей {#flattening-of-es-modules}

Формат Angular Package определяет, что код публикуется в «сглаженном» формате ES Module.
Это значительно сокращает время сборки Angular-приложений, а также время загрузки и разбора финального бандла приложения.
Ознакомьтесь с отличной статьёй [«Стоимость небольших модулей»](https://nolanlawson.com/2016/08/15/the-cost-of-small-modules) Нолана Лоусона.

Компилятор Angular может генерировать индексные файлы ES Module. Такие инструменты, как Rollup, могут использовать эти файлы для генерации сглаженных модулей в формате _Flattened ES Module_ (FESM).

FESM — это формат файлов, созданный путём сглаживания всех ES Module, доступных из точки входа, в единый ES Module.
Он формируется путём следования всем импортам из пакета и копирования кода в единый файл с сохранением всех публичных ES-экспортов и удалением всех приватных импортов.
Однако в некоторых случаях FESM может зависеть от общих фрагментов, разделяемых между несколькими точками входа.

Аббревиатура FESM, произносимая как _фи-сом_, может следовать за числом, например FESM2020.
Число обозначает уровень языка JavaScript внутри модуля.
Соответственно, файл FESM2022 будет ESM+ES2022 и содержать операторы import/export и исходный код ES2022.

Для генерации сглаженного индексного файла ES Module используйте следующие параметры конфигурации в файле `tsconfig.json`:

```js
{
  "compilerOptions": {
    …
    "module": "esnext",
    "target": "es2022",
    …
  },
  "angularCompilerOptions": {
    …
    "flatModuleOutFile": "my-ui-lib.js",
    "flatModuleId": "my-ui-lib"
  }
}
```

После того как индексный файл \(например, `my-ui-lib.js`\) будет сгенерирован ngc, такие сборщики и оптимизаторы, как Rollup, могут использоваться для создания сглаженного ESM-файла.

### Флаг "sideEffects" {#sideeffects-flag}

По умолчанию EcmaScript Modules имеют побочные эффекты: импорт из модуля гарантирует выполнение любого кода на верхнем уровне этого модуля.
Это часто нежелательно, поскольку большинство кода с побочными эффектами в типичных модулях не является действительно побочно-эффектным, а лишь влияет на конкретные символы.
Если эти символы не импортируются и не используются, желательно удалить их в процессе оптимизации, известном как tree-shaking, а код с побочными эффектами может препятствовать этому.

Такие инструменты сборки, как webpack, поддерживают флаг, позволяющий пакетам объявлять, что они не зависят от кода с побочными эффектами на верхнем уровне модулей, предоставляя инструментам большую свободу в tree-shaking кода из пакета.
Конечным результатом этих оптимизаций должен быть меньший размер бандла и лучшее распределение кода по частям бандла после разделения кода.
Эта оптимизация может нарушить работу кода, если он содержит нелокальные побочные эффекты — однако это редкость в Angular-приложениях и обычно является признаком плохого проектирования.
Рекомендуется всем пакетам заявлять статус без побочных эффектов, установив свойство `sideEffects` в `false`, а разработчикам следовать [руководству по стилю Angular](/style-guide), которое естественным образом приводит к коду без нелокальных побочных эффектов.

Дополнительная информация: [документация webpack о побочных эффектах](https://github.com/webpack/webpack/tree/master/examples/side-effects)

### Уровень языка ES2022 {#es2022-language-level}

Уровень языка ES2022 теперь является уровнем языка по умолчанию, используемым Angular CLI и другими инструментами.
Angular CLI понижает уровень бандла до уровня языка, поддерживаемого всеми целевыми браузерами, во время сборки приложения.

### Объединение d.ts / сглаживание определений типов {#dts-bundling-type-definition-flattening}

Начиная с APF v8, рекомендуется объединять определения TypeScript.
Объединение определений типов может значительно ускорить компиляцию для пользователей, особенно при наличии большого количества отдельных исходных `.ts` файлов в библиотеке.

Angular использует [`rollup-plugin-dts`](https://github.com/Swatinem/rollup-plugin-dts) для сглаживания файлов `.d.ts` (с помощью `rollup`, аналогично созданию FESM-файлов).

Использование rollup для объединения `.d.ts` выгодно, поскольку поддерживает разделение кода между точками входа.
Например, при наличии нескольких точек входа, использующих один общий тип, будет создан общий файл `.d.ts` вместе с большими сглаженными файлами `.d.ts`.
Это желательно и позволяет избежать дублирования типов.

### Tslib {#tslib}

Начиная с APF v10, рекомендуется добавлять tslib как прямую зависимость основной точки входа.
Это связано с тем, что версия tslib привязана к версии TypeScript, используемой для компиляции библиотеки.

## Определение терминов {#definition-of-terms}

Следующие термины используются в этом документе намеренно.
В этом разделе приведены определения всех из них для большей ясности.

### Пакет {#package}

Наименьший набор файлов, публикуемых в NPM и устанавливаемых вместе, например `@angular/core`.
Этот пакет включает манифест package.json, скомпилированный исходный код, файлы определений TypeScript, source map, метаданные и т. д.
Пакет устанавливается с помощью `npm install @angular/core`.

### Символ {#symbol}

Класс, функция, константа или переменная, содержащаяся в модуле и опционально делаемая видимой для внешнего мира через экспорт модуля.

### Модуль {#module}

Сокращение от ECMAScript Modules.
Файл, содержащий операторы импорта и экспорта символов.
Это идентично определению модулей в спецификации ECMAScript.

### ESM {#esm}

Сокращение от ECMAScript Modules \(см. выше\).

### FESM {#fesm}

Сокращение от Flattened ES Modules — формат файлов, созданный путём сглаживания всех ES Modules, доступных из точки входа, в единый ES Module.
Обратите внимание, что FESM обычно является единым файлом, но может зависеть от общего фрагмента, разделяемого с другими FESM.

### ID модуля {#module-id}

Идентификатор модуля, используемый в операторах импорта \(например, `@angular/core`\).
Идентификатор часто напрямую соответствует пути в файловой системе, но это не всегда так из-за различных стратегий разрешения модулей.

### Спецификатор модуля {#module-specifier}

Идентификатор модуля \(см. выше\).

### Стратегия разрешения модуля {#module-resolution-strategy}

Алгоритм, используемый для преобразования ID модулей в пути в файловой системе.
Node.js имеет хорошо определённую и широко используемую стратегию, TypeScript поддерживает несколько стратегий разрешения модулей, [Closure Compiler](https://developers.google.com/closure/compiler) имеет свою стратегию.

### Формат модуля {#module-format}

Спецификация синтаксиса модуля, охватывающая как минимум синтаксис импорта и экспорта из файла.
Распространённые форматы модулей: CommonJS \(CJS, обычно используемый для Node.js-приложений\) или ECMAScript Modules \(ESM\).
Формат модуля указывает только на упаковку отдельных модулей, но не на языковые возможности JavaScript, используемые для составления содержимого модуля.
Поэтому команда Angular часто использует спецификатор уровня языка в качестве суффикса к формату модуля \(например, ESM+ES2022 указывает, что модуль в формате ESM содержит код ES2022\).

### Бандл {#bundle}

Артефакт в виде единого JS-файла, созданный инструментом сборки \(например, [webpack](https://webpack.js.org) или [Rollup](https://rollupjs.org)\), содержащий символы из одного или нескольких модулей.
Бандлы — это браузерно-специфичный обходной путь, снижающий нагрузку на сеть, которая возникла бы, если бы браузеры начали загружать сотни, а то и десятки тысяч файлов.
Node.js обычно не использует бандлы.
Распространённые форматы бандлов: UMD и System.register.

### Уровень языка {#language-level}

Язык кода \(ES2022\).
Не зависит от формата модуля.

### Точка входа {#entry-point}

Модуль, предназначенный для импорта пользователем.
Он идентифицируется уникальным ID модуля и экспортирует публичный API, на который ссылается этот ID модуля.
Примером является `@angular/core` или `@angular/core/testing`.
Обе точки входа существуют в пакете `@angular/core`, но экспортируют разные символы.
Пакет может иметь множество точек входа.

### Глубокий импорт {#deep-import}

Процесс получения символов из модулей, не являющихся точками входа.
Идентификаторы этих модулей обычно считаются приватными API, которые могут меняться на протяжении жизни проекта или в процессе создания бандла для данного пакета.

### Импорт верхнего уровня {#top-level-import}

Импорт из точки входа.
Доступные импорты верхнего уровня определяют публичный API и представлены в модулях "@angular/name", таких как `@angular/core` или `@angular/common`.

### Tree-shaking {#tree-shaking}

Процесс идентификации и удаления кода, не используемого приложением — также известный как устранение мёртвого кода.
Это глобальная оптимизация, выполняемая на уровне приложения с помощью таких инструментов, как [Rollup](https://rollupjs.org), [Closure Compiler](https://developers.google.com/closure/compiler) или [Terser](https://github.com/terser/terser).

### AOT-компилятор {#aot-compiler}

Компилятор Ahead of Time для Angular.

### Сглаженные определения типов {#flattened-type-definitions}

Объединённые определения TypeScript, созданные с помощью таких инструментов, как [API Extractor](https://api-extractor.com) или [rollup-plugin-dts](https://github.com/Swatinem/rollup-plugin-dts).
