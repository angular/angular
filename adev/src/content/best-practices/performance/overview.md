# Производительность

Angular включает множество оптимизаций из коробки, но по мере роста приложений может потребоваться тонкая настройка как скорости загрузки, так и отзывчивости во время использования. Эти руководства охватывают инструменты и методы, которые Angular предоставляет для создания быстрых приложений.

## Производительность загрузки {#loading-performance}

Производительность загрузки определяет, насколько быстро приложение становится видимым и интерактивным. Медленная загрузка напрямую влияет на [Core Web Vitals](https://web.dev/vitals/), такие как Largest Contentful Paint (LCP) и Time to First Byte (TTFB).

| Метод                                                                                                   | Что делает                                                                                                                                                                                                                                    | Когда использовать                                                                                                         |
| :------------------------------------------------------------------------------------------------------ | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------- |
| [Ленивые маршруты](best-practices/performance/lazy-loaded-routes#lazily-loaded-components-and-routes) | Откладывает загрузку компонентов маршрутов до навигации, уменьшая размер начального бандла                                                                                                                                                   | Приложения с несколькими маршрутами, не все из которых нужны при начальной загрузке                                        |
| [Отложенная загрузка с `@defer`](best-practices/performance/defer)                                      | Разделяет компоненты на отдельные бандлы, загружаемые по требованию                                                                                                                                                                          | Компоненты, невидимые при начальном рендере, тяжёлые сторонние библиотеки, контент ниже линии просмотра                   |
| [Оптимизация изображений](best-practices/performance/image-optimization)                                | Приоритизирует LCP-изображения, лениво загружает остальные, генерирует адаптивные атрибуты `srcset`                                                                                                                                           | Любое приложение, отображающее изображения                                                                                 |
| [Серверный рендеринг](best-practices/performance/ssr)                                                   | Рендерит страницы на сервере для более быстрой первой отрисовки и лучшего SEO, с [гидратацией](guide/hydration) для восстановления интерактивности и [инкрементальной гидратацией](guide/incremental-hydration) для отложенной гидратации секций | Контент-ориентированные приложения, страницы, требующие индексирования поисковыми системами                                |

## Производительность во время выполнения {#runtime-performance}

Производительность во время выполнения определяет, насколько отзывчивым кажется приложение после загрузки. Система обнаружения изменений Angular поддерживает синхронизацию DOM с данными, и оптимизация того, как и когда она запускается, является основным рычагом для улучшения производительности во время выполнения.

| Метод                                                           | Что делает                                                                                                       | Когда использовать                                                                                          |
| :-------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------- |
| [Обнаружение изменений без Zone](guide/zoneless)                | Устраняет накладные расходы ZoneJS и запускает обнаружение изменений только при изменении сигналов или событий  | Новые приложения (по умолчанию в Angular v21+) или существующие приложения, готовые к миграции              |
| [Медленные вычисления](best-practices/slow-computations)        | Выявляет и оптимизирует затратные выражения шаблонов и хуки жизненного цикла                                     | Профилирование выявляет конкретные компоненты, вызывающие замедление циклов обнаружения изменений           |
| [Пропуск поддеревьев компонентов](best-practices/skipping-subtrees) | Использует стратегию `OnPush` для пропуска неизменённых деревьев компонентов                                | Приложения, требующие более тонкого контроля над обнаружением изменений                                     |
| [Загрязнение зон](best-practices/zone-pollution)                | Предотвращает излишнее обнаружение изменений, вызванное сторонними библиотеками или таймерами                    | Приложения на основе Zone, где профилирование выявляет чрезмерные циклы обнаружения изменений               |

## Измерение производительности {#measuring-performance}

Определить, что именно нужно оптимизировать, не менее важно, чем знать, как это сделать. Angular интегрируется с инструментами разработчика браузера для поиска узких мест.

| Инструмент                                                                      | Что делает                                                                                                                                                                                         |
| :------------------------------------------------------------------------------ | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| [Профилирование в Chrome DevTools](best-practices/profiling-with-chrome-devtools) | Записывает данные о производительности Angular вместе с данными профилирования браузера, с цветными flame-диаграммами, показывающими рендеринг компонентов, циклы обнаружения изменений и хуки жизненного цикла |
| [Angular DevTools](tools/devtools)                                               | Расширение браузера, предоставляющее инспектор дерева компонентов и профайлер для визуализации циклов обнаружения изменений                                                                       |

## С чего начать оптимизацию {#what-to-optimize-first}

Если вы не уверены, с чего начать, сначала профилируйте приложение с помощью [трека Angular в Chrome DevTools](best-practices/profiling-with-chrome-devtools), чтобы выявить конкретные узкие места.

В качестве общей отправной точки:

- **Медленная начальная загрузка** — используйте [`@defer`](best-practices/performance/defer) для выделения крупных компонентов из основного бандла, [`NgOptimizedImage`](best-practices/performance/image-optimization) для приоритизации изображений выше линии просмотра и [серверный рендеринг](best-practices/performance/ssr) для более быстрой доставки контента.
- **Медленные взаимодействия после загрузки** — проверьте, включено ли [обнаружение изменений без Zone](guide/zoneless), поищите [медленные вычисления](best-practices/slow-computations) в шаблонах или хуках жизненного цикла и рассмотрите [`OnPush`](best-practices/skipping-subtrees) для сокращения лишних проверок изменений.
