# Производительность

Angular включает множество оптимизаций «из коробки», но по мере роста приложений может понадобиться тонкая настройка и скорости загрузки, и отзывчивости во время работы. Эти руководства описывают инструменты и приёмы Angular для создания быстрых приложений.

## Производительность загрузки {#loading-performance}

Производительность загрузки определяет, как быстро приложение становится видимым и интерактивным. Медленная загрузка напрямую влияет на [Core Web Vitals](https://web.dev/vitals/), такие как Largest Contentful Paint (LCP) и Time to First Byte (TTFB).

| Приём                                                                                               | Что делает                                                                                                                                                                                                                    | Когда использовать                                                                                |
| :------------------------------------------------------------------------------------------------------ | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :-------------------------------------------------------------------------------------------- |
| [Лениво загружаемые маршруты](best-practices/performance/lazy-loaded-routes#lazily-loaded-components-and-routes) | Откладывает загрузку компонентов маршрутов до навигации, уменьшая начальный размер бандла                                                                                                                                              | Приложения с несколькими маршрутами, не все из которых нужны при первой загрузке                    |
| [Отложенная загрузка с `@defer`](best-practices/performance/defer)                                      | Выносит компоненты в отдельные бандлы, загружаемые по требованию                                                                                                                                                                     | Компоненты, не видимые при первом рендере, тяжёлые сторонние библиотеки, контент ниже сгиба |
| [Ленивая загрузка сервисов с `injectAsync`](guide/di/lazy-loading-services)                              | Выносит редко используемые сервисы в отдельные чанки и загружает их по требованию                                                                                                                                                       | Сервисы на базе больших библиотек или редко используемые функции                              |
| [Оптимизация изображений](best-practices/performance/image-optimization)                                     | Приоритизирует LCP-изображения, лениво загружает остальные, генерирует адаптивные атрибуты `srcset`                                                                                                                                             | Любое приложение, отображающее изображения                                                          |
| [Рендеринг на стороне сервера](best-practices/performance/ssr)                                                 | Рендерит страницы на сервере для более быстрой первой отрисовки и лучшего SEO, с [гидратацией](guide/hydration) для восстановления интерактивности и [инкрементальной гидратацией](guide/incremental-hydration) для отложенной гидратации секций | Контентные приложения, страницы, которым нужна индексация поисковиками                            |

## Производительность во время выполнения {#runtime-performance}

Производительность во время выполнения определяет, насколько отзывчиво приложение после загрузки. Система обнаружения изменений Angular синхронизирует DOM с данными; оптимизация того, как и когда она запускается, — главный рычаг улучшения runtime-производительности.

| Приём                                                       | Что делает                                                                                        | Когда использовать                                                                        |
| :-------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------ |
| [Zoneless обнаружение изменений](guide/zoneless)                     | Убирает накладные расходы ZoneJS и запускает обнаружение изменений только когда сигналы или события указывают на изменение | Новые приложения (по умолчанию в Angular v21+), или существующие, готовые к миграции |
| [Медленные вычисления](best-practices/slow-computations)           | Выявляет и оптимизирует дорогие выражения в шаблонах и хуки жизненного цикла                         | Профилирование показывает конкретные компоненты, замедляющие циклы обнаружения изменений            |
| [Пропуск поддеревьев компонентов](best-practices/skipping-subtrees) | Использует `OnPush` обнаружение изменений, чтобы пропускать неизменённые деревья компонентов                                    | Приложениям нужен более тонкий контроль над обнаружением изменений                            |
| [Загрязнение зон](best-practices/zone-pollution)                 | Предотвращает лишнее обнаружение изменений из-за сторонних библиотек или таймеров                     | Приложения на зонах, где профилирование показывает избыточные циклы обнаружения изменений     |

## Измерение производительности {#measuring-performance}

Понять, что оптимизировать, не менее важно, чем знать, как это делать. Angular интегрируется с инструментами разработчика браузера, чтобы находить узкие места.

| Инструмент                                                                       | Что делает                                                                                                                                                                     |
| :------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| [Профилирование в Chrome DevTools](best-practices/profiling-with-chrome-devtools) | Записывает Angular-специфичные данные производительности вместе с профилированием браузера; цветовые flame chart показывают рендеринг компонентов, циклы обнаружения изменений и хуки жизненного цикла |
| [Angular DevTools](tools/devtools)                                         | Расширение браузера с инспектором дерева компонентов и профайлером для визуализации циклов обнаружения изменений                                                              |

## С чего начать оптимизацию {#what-to-optimize-first}

Если неясно, с чего начать, сначала профильируйте приложение с помощью [Angular track в Chrome DevTools](best-practices/profiling-with-chrome-devtools), чтобы найти конкретные узкие места.

Общая отправная точка:

- **Медленная начальная загрузка** — используйте [`@defer`](best-practices/performance/defer), чтобы вынести крупные компоненты из основного бандла, [`NgOptimizedImage`](best-practices/performance/image-optimization) для приоритизации изображений above-the-fold и [рендеринг на стороне сервера](best-practices/performance/ssr) для более быстрой доставки контента.
- **Медленные взаимодействия после загрузки** — проверьте, включено ли [zoneless обнаружение изменений](guide/zoneless), ищите [медленные вычисления](best-practices/slow-computations) в шаблонах или хуках жизненного цикла и рассмотрите [`OnPush`](best-practices/skipping-subtrees) для сокращения лишнего обнаружения изменений.
