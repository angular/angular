# Доступность в Angular {#accessibility-in-angular}

Веб используется самыми разными людьми, в том числе теми, кто имеет нарушения зрения или моторики.
Существует множество вспомогательных технологий, которые значительно облегчают этим группам пользователей взаимодействие с веб-приложениями.
Кроме того, разработка более доступного приложения в целом улучшает пользовательский опыт для всех.

Для глубокого введения в проблематику и методы проектирования доступных приложений см. курс Google web.dev [Learn Accessibility](https://web.dev/learn/accessibility/).

На этой странице описаны лучшие практики разработки Angular-приложений, удобных для всех пользователей, в том числе для тех, кто использует вспомогательные технологии.

## Атрибуты доступности {#accessibility-attributes}

Создание доступных веб-интерфейсов нередко требует задания атрибутов [Accessible Rich Internet Applications \(ARIA\)](https://web.dev/learn/accessibility/aria-html/) для придания семантического смысла там, где его может не хватать.
Используйте синтаксис [привязки атрибутов](guide/templates/binding#binding-dynamic-properties-and-attributes) в шаблоне для управления значениями атрибутов, связанных с доступностью.

### ARIA-атрибуты и свойства {#aria-attributes-and-properties}

При привязке к ARIA-атрибутам в Angular их можно использовать напрямую, как любой другой HTML-атрибут.

```html
<button [aria-label]="myActionLabel">…</button>
```

Статические ARIA-атрибуты работают как обычные HTML-атрибуты.

```html
<button aria-label="Save document">…</button>
```

HELPFUL: По соглашению HTML-атрибуты используют имена в нижнем регистре \(`tabindex`\), а свойства — в camelCase \(`tabIndex`\).

Некоторые паттерны ARIA открывают DOM-API или входные данные директив, принимающие структурированные значения (например, коллекции ссылок на `Element`). Для таких случаев используйте стандартные привязки свойств, чтобы связь оставалась синхронизированной.

```angular-ts
@Component({
  template: `
    <h2 #dialogTitle>Attention</h2>
    <p #dialogDescription>Please review your answers before continuing.</p>

    <section role="dialog" [ariaLabelledByElements]="[dialogTitle, dialogDescription]">
      <ng-content />
    </section>
  `,
})
export class ReviewDialog {}
```

Здесь `[ariaLabelledByElements]` принимает массив элементов, поэтому привязка свойства поддерживает актуальность ссылок на элементы при изменении данных шаблона.

См. [руководство по привязке](guide/templates/binding#aria-attributes) для краткого описания синтаксиса ARIA-атрибутов.

## UI-компоненты Angular {#angular-ui-components}

Библиотека [Angular Material](https://material.angular.dev), поддерживаемая командой Angular, — это набор переиспользуемых UI-компонентов, ориентированных на полную доступность.
[Component Development Kit (CDK)](https://material.angular.dev/cdk/categories) включает пакет `a11y`, предоставляющий инструменты для поддержки различных аспектов доступности.
Например:

- `LiveAnnouncer` используется для объявления сообщений пользователям программ чтения с экрана с помощью региона `aria-live`.
  Подробнее об [aria-live regions](https://www.w3.org/WAI/PF/aria-1.1/states_and_properties#aria-live) см. в документации W3C.

- Директива `cdkTrapFocus` удерживает фокус клавиши Tab внутри элемента.
  Используйте её для обеспечения доступности таких компонентов, как модальные диалоги, где фокус должен оставаться ограниченным.

Полные сведения об этих и других инструментах см. в [обзоре доступности Angular CDK](https://material.angular.dev/cdk/a11y/overview).

### Расширение нативных элементов {#augmenting-native-elements}

Нативные HTML-элементы охватывают несколько стандартных паттернов взаимодействия, важных для доступности.
При создании Angular-компонентов следует по возможности повторно использовать эти нативные элементы напрямую, а не воссоздавать хорошо поддерживаемые поведения.

Например, вместо создания пользовательского элемента для новой разновидности кнопки создайте компонент, использующий селектор атрибута с нативным элементом `<button>`.
Чаще всего это применяется к `<button>` и `<a>`, но подходит и для многих других типов элементов.

Примеры этого паттерна можно найти в Angular Material:
[`MatButton`](https://github.com/angular/components/blob/main/src/material/button/button.ts#L33C3-L36C5), [`MatTabNav`](https://github.com/angular/components/blob/main/src/material/tabs/tab-nav-bar/tab-nav-bar.ts#L62) и [`MatTable`](https://github.com/angular/components/blob/main/src/material/table/table.ts#L40).

### Использование контейнеров для нативных элементов {#using-containers-for-native-elements}

Иногда использование подходящего нативного элемента требует элемента-контейнера.
Например, нативный элемент `<input>` не может иметь дочерних элементов, поэтому любые пользовательские компоненты ввода текста должны оборачивать `<input>` в дополнительные элементы.
Если просто включить `<input>` в шаблон компонента, пользователи компонента не смогут задавать произвольные свойства и атрибуты элементу `<input>`.
Вместо этого следует создать компонент-контейнер, использующий проекцию содержимого для включения нативного элемента управления в API компонента.

Пример такого паттерна — [`MatFormField`](https://material.angular.dev/components/form-field/overview).

## Маршрутизация {#routing}

### Управление фокусом после навигации {#focus-management-after-navigation}

Отслеживание и управление [фокусом](https://web.dev/learn/accessibility/focus/) в пользовательском интерфейсе — важное соображение при проектировании доступности.
При использовании маршрутизации Angular следует решить, куда переходит фокус страницы после навигации.

Чтобы не полагаться исключительно на визуальные подсказки, необходимо убедиться, что код маршрутизации обновляет фокус после навигации по страницам.
Используйте событие `NavigationEnd` из сервиса `Router`, чтобы узнать, когда обновлять фокус.

Следующий пример показывает, как найти и сфокусировать заголовок основного содержимого в DOM после навигации.

```ts
router.events.pipe(filter((e) => e instanceof NavigationEnd)).subscribe(() => {
  const mainHeader = document.querySelector('#main-content-header');
  if (mainHeader) {
    mainHeader.focus();
  }
});
```

В реальном приложении элемент, получающий фокус, зависит от конкретной структуры и компоновки приложения.
Сфокусированный элемент должен позволять пользователям сразу перейти к основному контенту, который только что стал виден.
Следует избегать ситуаций, когда фокус возвращается к элементу `body` после смены маршрута.

### Идентификация активных ссылок {#active-links-identification}

CSS-классы, применяемые к активным элементам `RouterLink`, такие как `RouterLinkActive`, дают визуальную подсказку для идентификации активной ссылки.
К сожалению, визуальная подсказка не помогает незрячим или слабовидящим пользователям.
Применение атрибута `aria-current` к элементу помогает идентифицировать активную ссылку.
Подробнее см. [Mozilla Developer Network \(MDN\) aria-current](https://developer.mozilla.org/docs/Web/Accessibility/ARIA/Attributes/aria-current)).

Директива `RouterLinkActive` предоставляет входной параметр `ariaCurrentWhenActive`, который задаёт `aria-current` указанным значением, когда ссылка становится активной.

Следующий пример показывает, как применить класс `active-page` к активным ссылкам и одновременно установить их атрибут `aria-current` в `"page"`:

```angular-html
<nav>
  <a routerLink="home" routerLinkActive="active-page" ariaCurrentWhenActive="page"> Home </a>
  <a routerLink="about" routerLinkActive="active-page" ariaCurrentWhenActive="page"> About </a>
  <a routerLink="shop" routerLinkActive="active-page" ariaCurrentWhenActive="page"> Shop </a>
</nav>
```

## Отложенная загрузка {#deferred-loading}

При использовании блоков `@defer` в Angular для ленивой загрузки контента следует учитывать аспекты доступности для пользователей с вспомогательными технологиями.
Программы чтения с экрана могут не объявлять автоматически об изменениях контента при загрузке отложенных компонентов, в результате чего пользователи могут не знать о новом контенте.

Чтобы изменения в отложенном контенте правильно объявлялись, оберните блоки `@defer` в элементы с соответствующими ARIA-регионами live.
Подробные инструкции и примеры см. в [разделе о доступности в руководстве по defer](/guide/templates/defer#keep-accessibility-in-mind).
