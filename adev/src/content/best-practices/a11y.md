# Доступность в Angular {#accessibility-in-angular}

Веб используется самыми разными людьми, включая тех, кто имеет нарушения зрения или двигательных функций.
Существует множество вспомогательных технологий, которые значительно упрощают взаимодействие этих групп людей с веб-приложениями.
Кроме того, проектирование приложения с учётом доступности, как правило, улучшает пользовательский опыт для всех пользователей.

Для подробного знакомства с проблемами и методами проектирования доступных приложений см. курс Google [Learn Accessibility](https://web.dev/learn/accessibility/) на web.dev.

На этой странице рассматриваются лучшие практики проектирования Angular-приложений, удобных для всех пользователей, включая тех, кто полагается на вспомогательные технологии.

## Атрибуты доступности {#accessibility-attributes}

Создание доступного веб-опыта часто включает установку [атрибутов Accessible Rich Internet Applications (ARIA)](https://web.dev/learn/accessibility/aria-html/) для обеспечения семантического значения там, где оно может отсутствовать.
Используйте синтаксис шаблона [привязки атрибутов](guide/templates/binding#binding-dynamic-properties-and-attributes) для управления значениями атрибутов, связанных с доступностью.

### Атрибуты и свойства ARIA {#aria-attributes-and-properties}

При привязке к ARIA-атрибутам в Angular их можно использовать напрямую, как и любой другой HTML-атрибут.

```html
<button [aria-label]="myActionLabel">…</button>
```

Статические ARIA-атрибуты работают как обычные HTML-атрибуты.

```html
<button aria-label="Save document">…</button>
```

HELPFUL: По соглашению HTML-атрибуты используют имена в нижнем регистре (`tabindex`), а свойства — имена в camelCase (`tabIndex`).

Некоторые паттерны ARIA предоставляют DOM API или входные параметры (inputs) директив, принимающие структурированные значения (например, коллекции ссылок на `Element`). В таких случаях используйте стандартные привязки свойств, чтобы соответствующая связь оставалась синхронизированной.

```angular-ts
@Component({
  template: `
    <h2 #dialogTitle>Attention</h2>
    <p #dialogDescription>Please review your answers before continuing.</p>

    <section role="dialog" [ariaLabelledByElements]="[dialogTitle, dialogDescription]">
      <ng-content />
    </section>
  `,
})
export class ReviewDialog {}
```

Здесь `[ariaLabelledByElements]` принимает массив элементов, поэтому привязка свойства поддерживает актуальность ссылок на элементы при каждом изменении данных шаблона.

Краткое описание синтаксиса ARIA-атрибутов см. в [руководстве по привязкам](guide/templates/binding#aria-attributes).

## UI-компоненты Angular {#angular-ui-components}

Библиотека [Angular Material](https://material.angular.dev), поддерживаемая командой Angular, представляет собой набор переиспользуемых UI-компонентов, целью которых является полная доступность.
[Component Development Kit (CDK)](https://material.angular.dev/cdk/categories) включает пакет `a11y`, предоставляющий инструменты для поддержки различных аспектов доступности.
Например:

- `LiveAnnouncer` используется для объявления сообщений для пользователей программ чтения с экрана с помощью региона `aria-live`.
  Дополнительную информацию о [регионах aria-live](https://www.w3.org/WAI/PF/aria-1.1/states_and_properties#aria-live) см. в документации W3C.

- Директива `cdkTrapFocus` удерживает фокус клавиши Tab внутри элемента.
  Используйте её для создания доступного опыта для компонентов, таких как модальные диалоги, где фокус должен быть ограничен.

Полные сведения об этих и других инструментах см. в [обзоре доступности Angular CDK](https://material.angular.dev/cdk/a11y/overview).

### Расширение нативных элементов {#augmenting-native-elements}

Нативные HTML-элементы реализуют несколько стандартных паттернов взаимодействия, важных для доступности.
При создании Angular-компонентов следует по возможности напрямую использовать эти нативные элементы, а не переопределять хорошо поддерживаемое поведение.

Например, вместо создания пользовательского элемента для нового вида кнопки создайте компонент, использующий селектор атрибута с нативным элементом `<button>`.
Чаще всего это применимо к `<button>` и `<a>`, но может использоваться и со многими другими типами элементов.

Примеры этого паттерна можно найти в Angular Material:
[`MatButton`](https://github.com/angular/components/blob/main/src/material/button/button.ts#L33C3-L36C5), [`MatTabNav`](https://github.com/angular/components/blob/main/src/material/tabs/tab-nav-bar/tab-nav-bar.ts#L62) и [`MatTable`](https://github.com/angular/components/blob/main/src/material/table/table.ts#L40).

### Использование контейнеров для нативных элементов {#using-containers-for-native-elements}

Иногда использование подходящего нативного элемента требует элемента-контейнера.
Например, нативный элемент `<input>` не может иметь дочерних элементов, поэтому любые пользовательские компоненты ввода текста должны оборачивать `<input>` дополнительными элементами.
Просто включив `<input>` в шаблон вашего компонента, пользователи компонента не смогут устанавливать произвольные свойства и атрибуты элемента `<input>`.
Вместо этого создайте компонент-контейнер, использующий проекцию содержимого для включения нативного элемента управления в API компонента.

Пример этого паттерна можно увидеть в [`MatFormField`](https://material.angular.dev/components/form-field/overview).

## Маршрутизация {#routing}

### Управление фокусом после навигации {#focus-management-after-navigation}

Отслеживание и управление [фокусом](https://web.dev/learn/accessibility/focus/) в пользовательском интерфейсе является важным аспектом проектирования доступных приложений.
При использовании маршрутизации Angular следует определить, куда переходит фокус страницы при навигации.

Чтобы не полагаться исключительно на визуальные подсказки, необходимо убедиться, что код маршрутизации обновляет фокус после навигации по страницам.
Используйте событие `NavigationEnd` из сервиса `Router`, чтобы знать, когда обновлять фокус.

Следующий пример показывает, как найти и сфокусироваться на заголовке основного содержимого в DOM после навигации.

```ts
router.events.pipe(filter((e) => e instanceof NavigationEnd)).subscribe(() => {
  const mainHeader = document.querySelector('#main-content-header');
  if (mainHeader) {
    mainHeader.focus();
  }
});
```

В реальном приложении элемент, получающий фокус, зависит от конкретной структуры и компоновки приложения.
Сфокусированный элемент должен сразу позволять пользователям перейти к основному содержимому, только что ставшему видимым.
Следует избегать ситуаций, когда фокус возвращается на элемент `body` после смены маршрута.

### Идентификация активных ссылок {#active-links-identification}

CSS-классы, применяемые к активным элементам `RouterLink`, например `RouterLinkActive`, обеспечивают визуальную подсказку для идентификации активной ссылки.
К сожалению, визуальная подсказка не помогает слабовидящим или незрячим пользователям.
Применение атрибута `aria-current` к элементу может помочь идентифицировать активную ссылку.
Дополнительную информацию см. в [Mozilla Developer Network (MDN) aria-current](https://developer.mozilla.org/docs/Web/Accessibility/ARIA/Attributes/aria-current)).

Директива `RouterLinkActive` предоставляет входной параметр `ariaCurrentWhenActive`, который устанавливает `aria-current` в указанное значение, когда ссылка становится активной.

Следующий пример показывает, как применять класс `active-page` к активным ссылкам, а также устанавливать их атрибут `aria-current` в значение `"page"`, когда они активны:

```angular-html
<nav>
  <a routerLink="home" routerLinkActive="active-page" ariaCurrentWhenActive="page"> Home </a>
  <a routerLink="about" routerLinkActive="active-page" ariaCurrentWhenActive="page"> About </a>
  <a routerLink="shop" routerLinkActive="active-page" ariaCurrentWhenActive="page"> Shop </a>
</nav>
```

## Отложенная загрузка {#deferred-loading}

При использовании блоков `@defer` в Angular для ленивой загрузки содержимого следует учитывать последствия для доступности у пользователей со вспомогательными технологиями.
Программы чтения с экрана могут не объявлять автоматически об изменениях содержимого при загрузке отложенных компонентов, что может оставить пользователей в неведении о новом содержимом.

Чтобы изменения отложенного содержимого были должным образом объявлены, оберните блоки `@defer` в элементы с соответствующими живыми регионами ARIA.
Подробное руководство и примеры см. в [разделе о доступности в руководстве по defer](guide/templates/defer#keep-accessibility-in-mind).
