# Необработанные ошибки в Angular {#unhandled-errors-in-angular}

В процессе работы Angular-приложения часть кода может выбрасывать ошибки. Если их не обрабатывать, это может приводить к непредсказуемому поведению и неотзывчивому пользовательскому интерфейсу. В этом руководстве рассматривается, как Angular справляется с ошибками, не перехваченными явно кодом приложения. Рекомендации по написанию собственной логики обработки ошибок в приложении см. в лучших практиках обработки ошибок в JavaScript и Angular.

Фундаментальный принцип стратегии обработки ошибок в Angular состоит в том, что ошибки должны, по возможности, показываться разработчикам в месте их возникновения. Такой подход гарантирует, что код, инициировавший операцию, имеет необходимый контекст для понимания ошибки, её корректной обработки и выбора подходящего состояния приложения. Делая ошибки видимыми в месте их возникновения, разработчики могут реализовать обработку ошибок, специфичную для неудавшейся операции, с доступом к соответствующей информации для восстановления или предоставления информативной обратной связи конечному пользователю. Это также помогает избежать запаха «слишком общей ошибки», когда ошибки сообщаются без достаточного контекста для понимания их причины.

Например, рассмотрим компонент, получающий данные пользователя из API. Код, выполняющий вызов API, должен включать обработку ошибок (например, с помощью блока `try...catch` или оператора `catchError` в RxJS) для управления возможными проблемами сети или ошибками, возвращаемыми API. Это позволяет компоненту отображать понятное пользователю сообщение об ошибке или повторить запрос, вместо того чтобы позволить ошибке распространиться необработанной.

## Необработанные ошибки передаются в `ErrorHandler` {#unhandled-errors-are-reported-to-the-errorhandler}

Angular передаёт необработанные ошибки в корневой `ErrorHandler` приложения. При предоставлении пользовательского `ErrorHandler` его следует указать в `ApplicationConfig` при вызове `bootstrapApplication`.

При разработке Angular-приложения часто пишется код, который вызывается _фреймворком_ автоматически. Например, Angular отвечает за вызов конструктора компонента и методов жизненного цикла, когда этот компонент появляется в шаблоне. Когда фреймворк выполняет ваш код, нет разумного места, куда можно было бы добавить блок `try` для корректной обработки ошибок. В таких ситуациях Angular перехватывает ошибки и отправляет их в `ErrorHandler`.

Angular _не_ перехватывает ошибки внутри API, вызываемых непосредственно вашим кодом. Например, если у вас есть сервис с методом, который выбрасывает ошибку, и вы вызываете этот метод в компоненте, Angular не перехватит её автоматически. Обрабатывать её — ваша ответственность, с помощью таких механизмов, как `try...catch`.

Angular перехватывает _асинхронные_ ошибки из пользовательских промисов или наблюдаемых только когда:

- Существует явный контракт для ожидания Angular результата асинхронной операции и его использования, и
- Когда ошибки не представлены в возвращаемом значении или состоянии.

Например, `AsyncPipe` и `PendingTasks.run` передают ошибки в `ErrorHandler`, тогда как `resource` представляет ошибку в свойствах `status` и `error`.

Ошибки, которые Angular передаёт в `ErrorHandler`, — это _неожиданные_ ошибки. Такие ошибки могут быть невосстановимыми или указывать на то, что состояние приложения повреждено. Приложения должны обеспечивать обработку ошибок с помощью блоков `try` или соответствующих операторов обработки ошибок (например, `catchError` в RxJS) в месте возникновения ошибки всегда, когда это возможно, а не полагаться на `ErrorHandler`, который наиболее уместен и чаще всего используется исключительно как механизм для сообщения о потенциально фатальных ошибках в инфраструктуру отслеживания и журналирования ошибок.

```ts
export class GlobalErrorHandler implements ErrorHandler {
  private readonly analyticsService = inject(AnalyticsService);
  private readonly router = inject(Router);

  handleError(error: any) {
    const url = this.router.url;
    const errorMessage = error?.message ?? 'unknown';

    this.analyticsService.trackEvent({
      eventName: 'exception',
      description: `Screen: ${url} | ${errorMessage}`,
    });

    console.error(GlobalErrorHandler.name, {error});
  }
}
```

### `TestBed` по умолчанию повторно выбрасывает ошибки {#testbed-rethrows-errors-by-default}

Во многих случаях `ErrorHandler` может только журналировать ошибки, позволяя приложению продолжать работу. Однако в тестах почти всегда требуется показывать эти ошибки. `TestBed` Angular повторно выбрасывает неожиданные ошибки, чтобы ошибки, перехваченные фреймворком, не могли быть случайно пропущены или проигнорированы. В редких случаях тест может специально проверять, что ошибки не вызывают неотзывчивость или сбой приложения. В таких ситуациях можно [настроить `TestBed` так, чтобы он _не_ повторно выбрасывал ошибки приложения](api/core/testing/TestModuleMetadata#rethrowApplicationErrors) с помощью `TestBed.configureTestingModule({rethrowApplicationErrors: false})`.

## Глобальные обработчики ошибок {#global-error-listeners}

Ошибки, не перехваченные ни кодом приложения, ни экземпляром приложения фреймворка, могут достичь глобальной области видимости. Ошибки в глобальной области видимости могут иметь непредвиденные последствия, если их не учитывать. В небраузерных средах они могут вызвать сбой процесса. В браузере эти ошибки могут оставаться незамеченными, и посетители сайта могут видеть ошибки в консоли браузера. Angular предоставляет глобальные обработчики для обеих сред.

### Клиентская отрисовка {#client-side-rendering}

Добавление [`provideBrowserGlobalErrorListeners()`](/api/core/provideBrowserGlobalErrorListeners) в [ApplicationConfig](guide/di/defining-dependency-providers#application-bootstrap) добавляет обработчики `'error'` и `'unhandledrejection'` к окну браузера и передаёт эти ошибки в `ErrorHandler`. Angular CLI генерирует новые приложения с этим провайдером по умолчанию. Команда Angular рекомендует обрабатывать эти глобальные ошибки в большинстве приложений — либо с помощью встроенных обработчиков фреймворка, либо с собственными пользовательскими обработчиками. Если вы предоставляете пользовательские обработчики, можно удалить `provideBrowserGlobalErrorListeners`.

### Серверная и гибридная отрисовка {#server-side-and-hybrid-rendering}

При использовании [Angular с SSR](guide/ssr) Angular автоматически добавляет обработчики `'unhandledRejection'` и `'uncaughtException'` к серверному процессу. Эти обработчики предотвращают сбой сервера и вместо этого журналируют перехваченные ошибки в консоль.

IMPORTANT: Если приложение использует Zone.js, добавляется только обработчик `'unhandledRejection'`. При наличии Zone.js ошибки внутри Zone приложения уже передаются в `ErrorHandler` приложения и не достигают серверного процесса.
